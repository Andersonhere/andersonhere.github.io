## 5.1 相关实际问题

## 概述

前面的章节深度分析了网络包的接收，也拆分了网络包的发送，总之收发流程算是闭环了。不过还有一种特殊情况没有讨论，那就是接收和发送都在本机进行。

而且实践中这种本机网络IO出现的场景还不少，而且还有越来越多的趋势：
- LNMP技术栈中的nginx和php-fpm进程就是通过本机来通信的
- 最近流行的微服务中sidecar模式也是本机网络IO

所以，如果能深度理解这个问题，在实践中将非常有意义。

---

## 问题1：127.0.0.1 本机网络IO需要经过网卡吗？

### 问题描述

在跨机网络IO中，数据包肯定都是要经过网卡发送出去的。那么，在本机网络IO的情况下，收发数据需要经过网卡吗？

### 实验假设

如果把网卡拔了，127.0.0.1上数据收发能否正常工作？

### 答案

**不需要经过网卡！**

本机网络IO使用的是虚拟的loopback设备（lo），不需要物理网卡参与。

```bash
## 查看lo设备
$ ip link show lo
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00

## 即使禁用物理网卡，本机通信仍然正常
$ ifconfig eth0 down
$ ping 127.0.0.1  # 仍然可以ping通
```

---

## 问题2：数据包在内核中是什么走向？

### 问题描述

假如本机网络IO和跨机IO收发流程不一样，那么是在哪几个环节上不同呢？

### 跨机IO流程

```
用户进程
    │ send()
    ▼
协议栈处理
    │
    ▼
网卡驱动 → 物理网卡 ──网络──► 物理网卡 → 网卡驱动
    │                              │
    ▼                              ▼
硬中断                          协议栈处理
    │                              │
    ▼                              ▼
软中断                          用户进程
```

### 本机IO流程

```
用户进程
    │ send()
    ▼
协议栈处理
    │
    ▼
loopback驱动（不经过物理网卡）
    │
    │ netif_rx() 直接触发软中断
    ▼
软中断
    │
    ▼
协议栈处理
    │
    ▼
用户进程 recv()
```

### 差异点

| 环节 | 跨机IO | 本机IO |
|------|--------|--------|
| 路由查找 | main路由表 | local路由表 |
| 网卡设备 | 物理网卡 | lo虚拟设备 |
| 网卡驱动 | 真实驱动 | loopback_xmit |
| DMA传输 | 需要 | 不需要 |
| 硬中断 | 需要 | 不需要 |
| 网络传输 | 需要 | 不需要 |

---

## 问题3：127.0.0.1 vs 本机IP，哪个更快？

### 问题描述

实际上，使用本机IO通信的时候也有两种方法：
- 一种方法是用 127.0.0.1
- 一种方法是使用本机IP，例如 192.168.x.x 这种

那么这两种方法在性能上会有什么差异吗？哪种方法性能更好呢？

### 测试方法

```bash
## 使用127.0.0.1
$ ab -n 100000 -c 100 http://127.0.0.1:80/

## 使用本机IP
$ ab -n 100000 -c 100 http://192.168.1.100:80/
```

### 答案

**性能几乎没有差异！**

### 原因分析

两者都会：
1. 在local路由表中找到路由项
2. 使用lo设备进行数据传输
3. 不经过物理网卡

```bash
## 查看local路由表
$ ip route list table local

## 127.0.0.1 路由到lo设备
local 127.0.0.1 dev lo proto kernel scope host src 127.0.0.1

## 本机IP也路由到lo设备（对于本机访问）
local 192.168.1.100 dev eth0 proto kernel scope host src 192.168.1.100
```

### 内核代码证明

```c
// file: net/ipv4/route.c
struct rtable *ip_route_output_key(struct net *net, struct flowi4 *fl4)
{
    if (fib_lookup(net, fl4, &res)) {
        if (res.type == RTN_LOCAL) {
            // 无论是127.0.0.1还是本机IP，都使用loopback设备
            dev_out = net->loopback_dev;
            rth = mkroute_output(&res, fl4, orig_oif, dev_out, flags);
            return rth;
        }
    }
}
```

### 结论

| 访问方式 | 路由表 | 网络设备 | 性能 |
|---------|--------|----------|------|
| 127.0.0.1 | local | lo | 相同 |
| 本机IP | local | lo | 相同 |

**使用127.0.0.1和本机IP，在内核处理上几乎完全一致，性能没有差异。**

---

## 小结

### 三个问题的答案

1. **127.0.0.1需要经过网卡吗？**
   - 不需要，使用虚拟的lo设备

2. **数据包在内核中的走向差异？**
   - 差异在路由和驱动层面
   - 本机IO跳过了网卡硬件和DMA传输

3. **127.0.0.1 vs 本机IP？**
   - 性能几乎没有差异
   - 都使用local路由表和lo设备

### 实际应用意义

理解这些问题有助于：
- 正确配置本地服务通信
- 性能调优时的决策
- 问题排查时的思路

```c
SYSCALL_DEFINE6(sendto, int, fd, ...)
{
    // 1. 查找socket
    sock = sockfd_lookup_light(fd, &err, &fput_needed);

    // 2. 构造msghdr
    struct msghdr msg;
    ...

    // 3. 发送数据
    sock_sendmsg(sock, &msg, len);

// 传输层
// file: net/ipv4/tcp.c
int tcp_sendmsg(...)
{
    // 申请skb，拷贝用户数据
    skb = sk_stream_alloc_skb(...);
    skb_add_data_nocache(sk, skb, from, copy);

    // 发送
    tcp_push(sk, flags);
}

// 网络层
// file: net/ipv4/ip_output.c
int ip_queue_xmit(struct sk_buff *skb, struct flowi *fl)
{

    // 发送
    ip_local_out(skb);

// 邻居子系统
// file: include/net/neighbour.h
int neigh_resolve_output(...)
{
    // 封装MAC头

}

// file: net/core/dev.c
int dev_queue_xmit(struct sk_buff *skb)
{

}

// 驱动程序
// file: drivers/net/ethernet/intel/igb/igb_main.c
static int igb_xmit_frame(...)
{
    // DMA映射，写入RingBuffer

}
```

## 5.2.3 跨机网络通信汇总

### 完整流程

一次跨机网络通信的完整过程：

```
客户端主机                                    服务端主机
┌──────────────────┐                      ┌──────────────────┐
│  用户进程         │                      │  用户进程         │
│  send()          │                      │  recv()          │
├──────────────────┤                      ├──────────────────┤
│  系统调用         │                      │  进程调度         │
├──────────────────┤                      ├──────────────────┤
│  协议栈           │                      │  协议栈           │
│  - TCP层         │                      │  - TCP层         │
│  - IP层          │                      │  - IP层          │
├──────────────────┤                      ├──────────────────┤
│  网络设备子系统    │                      │  网络设备子系统    │
│  邻居子系统       │                      │  邻居子系统       │
├──────────────────┤                      ├──────────────────┤
│  驱动程序         │                      │  驱动程序         │
│  软中断          │                      │  软中断          │
├──────────────────┤                      ├──────────────────┤
│  网卡硬件         │ ──── 网络传输 ────► │  网卡硬件         │
│  RingBuffer      │                      │  RingBuffer      │
└──────────────────┘                      └──────────────────┘
```

### 时间消耗估算

| 阶段 | 时间 | 说明 |
|------|------|------|
| 系统调用 | ~1μs | 用户态→内核态切换 |
| 协议处理 | ~10μs | TCP/IP协议处理 |
| 网卡发送 | ~10μs | DMA传输 |
| 网络传输 | ~100μs-1ms | 取决于距离和带宽 |
| 网卡接收 | ~10μs | 硬中断+软中断 |
| 协议处理 | ~10μs | TCP/IP协议处理 |
| 唤醒进程 | ~1μs | 内核态→用户态切换 |

**总计**：约 **0.2ms - 2ms**（同机房）

## 小结

### 跨机通信的关键特点

1. **经过物理网卡**：DMA传输、硬中断、软中断
2. **有网络延迟**：取决于网络距离和质量
3. **可能丢包重传**：网络不可靠
4. **MTU限制**：通常1500字节

### 与本机通信的对比

这些特点正是本机通信所跳过的，因此本机通信会更快更可靠。

## 5.3 本机发送过程

## 概述

本机网络IO的过程中，流程会有一些差别。有差异的地方总共有两处，分别是**路由**和**驱动程序**。

## 5.3.1 网络层路由

发送数据进入协议栈到达网络层的时候，网络层入口函数是 `ip_queue_xmit`。在网络层里会进行路由选择，路由选择完毕，再设置一些IP头，进行一些netfilter的过滤，将包交给邻居子系统。

### 路由查找过程

```c
// file: net/ipv4/ip_output.c
int ip_queue_xmit(struct sk_buff *skb, struct flowi *fl)
{
    struct rtable *rt;

    // 检查socket中是否有缓存的路由表
    rt = (struct rtable *)__sk_dst_check(sk, 0);
    if (rt == NULL) {
        // 没有缓存则展开查找
        rt = ip_route_output_ports(...);
        sk_setup_caps(sk, &rt->dst);
    }

    // 为skb设置路由表
    skb_dst_set_noref(skb, &rt->dst);

    // 设置IP头
    iph = ip_hdr(skb);
    iph->protocol = sk->sk_protocol;
    iph->ttl = ip_select_ttl(inet, &rt->dst);

    // 发送
    ip_local_out(skb);
}
```

### 路由表查找

```c
// file: include/net/ip_fib.h
static inline int fib_lookup(struct net *net, const struct flowi4 *flp,
                             struct fib_result *res)
{
    struct fib_table *table;

    // 先查 local 路由表
    table = fib_get_table(net, RT_TABLE_LOCAL);
    if (!fib_table_lookup(table, flp, res, FIB_LOOKUP_NOREF))
        return 0;

    // 再查 main 路由表
    table = fib_get_table(net, RT_TABLE_MAIN);
    if (!fib_table_lookup(table, flp, res, FIB_LOOKUP_NOREF))
        return 0;

    return -ENETUNREACH;
}
```

### 查看本地路由表

```bash
$ ip route list table local

broadcast 10.0.0.0 dev eth0 proto kernel scope link src 10.0.0.6
local 10.0.0.6 dev eth0 proto kernel scope host src 10.0.0.6
broadcast 127.0.0.0 dev lo proto kernel scope link src 127.0.0.1
local 127.0.0.0/8 dev lo proto kernel scope host src 127.0.0.1
local 127.0.0.1 dev lo proto kernel scope host src 127.0.0.1
```

### 本机路由的特殊处理

```c
// file: net/ipv4/route.c
struct rtable *ip_route_output_key(struct net *net, struct flowi4 *fl4)
{
    if (fib_lookup(net, fl4, &res)) {
        if (res.type == RTN_LOCAL) {
            // 本机IP，使用loopback设备
            dev_out = net->loopback_dev;
            rth = mkroute_output(&res, fl4, orig_oif, dev_out, flags);
            return rth;
        }
    }
}
```

### 路由流程图

```
ip_queue_xmit
    │
    │  ip_route_output_ports()
    ▼
ip_route_output_key
    │
    │  fib_lookup()
    ▼
┌─────────────────────────────────────────┐
│  先查 local 路由表                        │
│  ┌─────────────────────────────────────┐│
│  │ local 127.0.0.1 dev lo              ││
│  │ local 192.168.x.x dev lo（本机访问） ││
│  └─────────────────────────────────────┘│
│                                         │
│  在local表中找到 → 使用loopback_dev      │
└─────────────────────────────────────────┘
    │
    │  res.type == RTN_LOCAL
    ▼
dev_out = net->loopback_dev
```

## 5.3.2 本机IP路由

### 问题：使用本机IP和127.0.0.1有差异吗？

**答案：几乎没有差异！**

### 原因分析

无论是访问127.0.0.1还是访问本机IP（如192.168.x.x），都会：

1. 在local路由表中找到路由项
2. 设备都被设置为loopback_dev
3. 不经过物理网卡

```bash
## 查看local路由表
$ ip route list table local

## 127.0.0.1 路由到lo设备
local 127.0.0.1 dev lo proto kernel scope host src 127.0.0.1

## 本机IP在本机访问时也路由到lo设备
local 192.168.1.100 dev eth0 proto kernel scope host src 192.168.1.100
```

### 内核判断逻辑

```c
// file: net/ipv4/route.c
if (res.type == RTN_LOCAL) {
    // 只要类型是RTN_LOCAL，就使用loopback设备
    dev_out = net->loopback_dev;
}
```

### 性能测试

```bash

## 测试本机IP

## 结论：性能几乎相同
```

## 5.3.3 网络设备子系统

### loopback设备

本机IO使用的是**虚拟的loopback设备**：

```bash
$ ip link show lo
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
    inet 127.0.0.1/8 scope host lo
```

### loopback设备特点

| 特点 | 说明 |
|------|------|
| **没有物理硬件** | 纯软件实现 |
| **没有中断开销** | 不需要硬中断 |
| **没有DMA传输** | 数据直接在内存传递 |
| **MTU很大** | 默认65535，减少分片 |
| **速度极快** | 比物理网卡快很多 |

### 与物理网卡的对比

```
物理网卡发送：
用户数据 → skb → DMA映射 → RingBuffer → 网卡硬件 → 网络传输

loopback发送：
用户数据 → skb → 直接放入接收队列
```

## 5.3.4 "驱动"程序

loopback设备的"驱动"非常简单，它不会真正发送数据，而是直接将数据放回接收队列。

### loopback_xmit函数

```c
// file: drivers/net/loopback.c
static netdev_tx_t loopback_xmit(struct sk_buff *skb,
                                  struct net_device *dev)
{
    struct pcpu_lstats *lb_stats;
    int len;

    // 保存长度
    len = skb->len;

    // 设置协议类型
    skb->protocol = eth_type_trans(skb, dev);

    // 关键：直接将数据放回接收队列！
    netif_rx(skb);

    // 统计
    lb_stats = this_cpu_ptr(dev->lstats);
    lb_stats->bytes += len;
    lb_stats->packets++;

    return NETDEV_TX_OK;
}
```

### netif_rx的作用

```c
// file: net/core/dev.c
int netif_rx(struct sk_buff *skb)
{
    // 将skb放入接收队列
    // 触发软中断 NET_RX_SOFTIRQ
    return netif_rx_internal(skb);
}
```

### 关键点

`netif_rx(skb)` 直接触发软中断，将数据放入接收流程，完全跳过了：
- 物理网卡
- DMA传输
- 硬中断处理
- RingBuffer

### 流程对比

```
跨机发送：
驱动 → DMA → 网卡硬件 → 网络传输 → 网卡 → 硬中断 → 软中断

本机发送：
驱动 → netif_rx() → 软中断（直接跳过网卡和网络）
```

### netif_rx

```c
// file: net/core/dev.c
int netif_rx(struct sk_buff *skb)
{
    int ret;

    // 可能的RPS处理（多核负载均衡）
    if (static_key_false(&rps_needed)) {
        struct rps_dev_flow voidflow, *rflow = &voidflow;
        int cpu;

        preempt_disable();
        rcu_read_lock();

        cpu = get_rps_cpu(skb->dev, skb, &rflow);

        if (cpu < 0)
            cpu = smp_processor_id();

        ret = enqueue_to_backlog(skb, cpu, &rflow->last_qtail);

        rcu_read_unlock();
        preempt_enable();
    } else {
        unsigned int qtail;
        ret = enqueue_to_backlog(skb, get_cpu(), &qtail);
        put_cpu();
    }

    return ret;
}

static int enqueue_to_backlog(struct sk_buff *skb, int cpu,
                               unsigned int *qtail)
{
    struct softnet_data *sd;
    unsigned long flags;

    sd = &per_cpu(softnet_data, cpu);

    local_irq_save(flags);

    // 将skb放入接收队列
    __skb_queue_tail(&sd->input_pkt_queue, skb);

    // 触发软中断
    if (!__test_and_set_bit(NAPI_STATE_SCHED, &sd->backlog.state)) {
        ____napi_schedule(sd, &sd->backlog);
    }

    local_irq_restore(flags);

    return NET_RX_SUCCESS;
}
```

### 软中断处理

```c
// file: net/core/dev.c
static void net_rx_action(struct softirq_action *h)
{
    struct softnet_data *sd = &__get_cpu_var(softnet_data);
    unsigned long time_limit = jiffies + 2;
    int budget = netdev_budget;
    void *have;

    local_irq_disable();

    while (!list_empty(&sd->poll_list)) {
        struct napi_struct *n;
        int work, weight;

        n = list_first_entry(&sd->poll_list, struct napi_struct, poll_list);

        weight = n->weight;

        if (test_bit(NAPI_STATE_SCHED, &n->state)) {
            // 对于loopback，这里的poll是process_backlog
            work = n->poll(n, weight);
        }

        budget -= work;
    }

    local_irq_enable();
}
```

### process_backlog

```c
// file: net/core/dev.c
static int process_backlog(struct napi_struct *napi, int quota)
{
    struct softnet_data *sd = container_of(napi, struct softnet_data, backlog);
    int work = 0;

    while (work < quota) {
        struct sk_buff *skb;
        unsigned int qlen;

        while ((skb = __skb_dequeue(&sd->input_pkt_queue))) {
            // 处理每个skb
            __netif_receive_skb(skb);
            work++;
        }
    }

    return work;
}
```

## 5.1 相关实际问题

## 5.3 本机发送过程

本机网络IO的过程中，流程会有一些差别。有差异的地方总共有两处，分别是**路由**和**驱动程序**。

### 5.3.1 网络层路由

发送数据进入协议栈到达网络层的时候，网络层入口函数是 `ip_queue_xmit`。

```c
// file: net/ipv4/ip_output.c
int ip_queue_xmit(struct sk_buff *skb, struct flowi *fl)
{
    // 检查socket中是否有缓存的路由表
    rt = (struct rtable *)__sk_dst_check(sk, 0);
    if (rt == NULL) {
        // 没有缓存则展开查找
        // 查找路由项，并缓存到socket中
        rt = ip_route_output_ports(...);
        sk_setup_caps(sk, &rt->dst);
    }

    // 为skb设置路由表
    skb_dst_set_noref(skb, &rt->dst);

    // 设置IP头
    iph = ip_hdr(skb);
    iph->protocol = sk->sk_protocol;
    iph->ttl = ip_select_ttl(inet, &rt->dst);

    // 发送
    ip_local_out(skb);
}
```

#### 路由查找过程

```c
// file: include/net/ip_fib.h
static inline int fib_lookup(struct net *net, const struct flowi4 *flp,
                             struct fib_result *res)
{
    struct fib_table *table;

    // 先查 local 路由表
    table = fib_get_table(net, RT_TABLE_LOCAL);
    if (!fib_table_lookup(table, flp, res, FIB_LOOKUP_NOREF))
        return 0;

    // 再查 main 路由表
    table = fib_get_table(net, RT_TABLE_MAIN);
    if (!fib_table_lookup(table, flp, res, FIB_LOOKUP_NOREF))
        return 0;

    return -ENETUNREACH;
}
```

#### local路由表

```bash
## 查看本地路由表
$ ip route list table local

local 10.143.x.y dev eth0 proto kernel scope host src 10.143.x.y
local 127.0.0.1 dev lo proto kernel scope host src 127.0.0.1
```

对于本机的网络请求，设备将全部使用 `net->loopback_dev`，也就是 **lo虚拟网卡**。

```c
// file: net/ipv4/route.c
struct rtable *ip_route_output_key(struct net *net, struct flowi4 *fl4)
{
    if (fib_lookup(net, fl4, &res)) {
        if (res.type == RTN_LOCAL) {
            dev_out = net->loopback_dev;  // 使用 lo 设备
            rth = mkroute_output(&res, fl4, orig_oif, dev_out, flags);
            return rth;
        }
    }
}
```

### 5.3.2 本机IP路由

**问题：使用本机IP（如192.168.x.x）和127.0.0.1有差异吗？**

答案：**几乎没有差异！**

原因：
- 都会匹配到 local 路由表
- 都使用 `lo` (loopback) 设备
- 都不走真实网卡

```bash
## 查看本地路由表
$ ip route list table local

## 本机IP也会路由到lo设备
local 192.168.1.100 dev eth0 proto kernel scope host src 192.168.1.100
local 127.0.0.1 dev lo proto kernel scope host src 127.0.0.1
```

### 5.3.3 网络设备子系统

本机IO使用的是 **虚拟的loopback设备**：

```
┌─────────────────────────────────────────────────────────┐
│                   lo (loopback) 设备                     │
│  ┌─────────────────────────────────────────────────┐   │
│  │  特点：                                           │   │
│  │  - 没有物理网卡的中断开销                          │   │
│  │  - 没有真实的DMA传输                              │   │
│  │  - 数据直接在内存中传递                           │   │
│  │  - MTU可以很大（默认65535）                       │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 5.3.4 "驱动"程序

loopback设备的"驱动"非常简单，它不会真正发送数据，而是直接将数据放回接收队列：

```c
// file: drivers/net/loopback.c
static netdev_tx_t loopback_xmit(struct sk_buff *skb,
                                  struct net_device *dev)
{
    struct pcpu_lstats *lb_stats;
    int len;

    // 保存长度
    len = skb->len;

    // 设置协议类型
    skb->protocol = eth_type_trans(skb, dev);

    // 直接将数据放回接收队列！
    netif_rx(skb);

    // 统计
    lb_stats = this_cpu_ptr(dev->lstats);
    lb_stats->bytes += len;
    lb_stats->packets++;

    return NETDEV_TX_OK;
}
```

#### 关键点

`netif_rx(skb)` 直接触发软中断，将数据放入接收流程：

```c
// file: net/core/dev.c
int netif_rx(struct sk_buff *skb)
{
    // 将skb放入接收队列
    // 触发软中断 NET_RX_SOFTIRQ
    return netif_rx_internal(skb);
}
```

## 5.4 本机接收过程

由于loopback设备的特殊性，本机网络IO的接收过程非常高效：

```
用户进程 A (发送)
    │
    │  send()
    ▼
┌──────────────────────────────────────┐
│              协议栈                    │
│  ┌────────────────────────────────┐  │
│  │  TCP → IP → 路由查找            │  │

│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
    │
    │  loopback_xmit()
    ▼
┌──────────────────────────────────────┐
│          loopback "驱动"              │
│  ┌────────────────────────────────┐  │
│  │  netif_rx(skb)                 │  │
│  │  直接触发软中断                  │  │
│  │  不经过网卡！                    │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
    │
    │  软中断 NET_RX_SOFTIRQ
    ▼
┌──────────────────────────────────────┐
│              协议栈                    │
│  ┌────────────────────────────────┐  │
│  │  IP → TCP → 放入接收队列        │  │
```

| 步骤 | 跨机IO | 本机IO |
|------|--------|--------|
| 发送系统调用 | ✓ | ✓ |
| 协议处理 | ✓ | ✓ |
| 路由查找 | main路由表 | local路由表 |

| 物理网卡 | ✓ 经过 | ✗ 不经过 |
| DMA传输 | ✓ | ✗ |

| 软中断 | NET_RX | NET_RX |
| 接收协议处理 | ✓ | ✓ |

## 5.5 本章总结

### 本机IO性能优势

| 优势 | 说明 |
|------|------|
| **不经过网卡** | 无DMA开销，无网卡中断 |
| **无网络延迟** | 无网络传输时间 |
| **无丢包重传** | 可靠性极高 |
| **MTU更大** | lo设备MTU可达65535，减少分片 |

### 127.0.0.1 vs 本机IP

**性能几乎没有差异！**

两者都：
- 使用local路由表
- 使用lo设备
- 不经过物理网卡

### 性能对比

```
跨机IO：  ~1ms（同机房）
本机IO：  ~0.01ms

本机IO比跨机IO快约 100 倍
```

### 应用场景

- **LNMP架构**：nginx与php-fpm通过本机通信
- **微服务Sidecar**：服务进程与sidecar代理通信
- **本地数据库**：应用与本地数据库通信
- **本地缓存**：应用与本地缓存服务通信

### 最佳实践

1. 服务部署在同一主机时，使用127.0.0.1或本机IP都可以
2. 本机通信不需要优化网络参数
3. 可以适当增大lo设备的MTU
