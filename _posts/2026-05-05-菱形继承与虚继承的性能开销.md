## 引言

写一个经典的菱形继承：

```cpp
struct Base { int val = 42; };
struct Left  : virtual Base {};
struct Right : virtual Base {};
struct Bottom : Left, Right {};
```

现在问一个问题：`sizeof(Bottom)` 是多少？

如果没有 `virtual`，`Left` 和 `Right` 各带一份 `Base`，`Bottom` 里有两个 `val`——数据冗余，访问二义性，老生常谈。加上 `virtual` 之后，`Bottom` 里只剩一份 `Base`，问题解决了。

但 `sizeof` 变大了。

在 64 位 MSVC 上，不加 `virtual` 的 `Bottom` 是 8 字节（两个 `int`）；加了 `virtual` 之后，变成了 24 字节。多出来的不是数据——是编译器塞进去的指针和表。而这些指针和表，意味着每一次你访问虚基类的成员、每一次你调用虚基类的虚函数，都要多走几步弯路。

这篇讲的是：**编译器为了实现虚继承，在底层给你的每次访问和调用加了多少"过路费"。**

---

## 一、先看账单——虚继承到底往对象里塞了什么

普通继承的内存布局很直白。`Left` 继承 `Base`，`Left` 对象的开头就是 `Base` 子对象，后面跟 `Left` 自己的成员。编译器在编译期就知道 `Base` 在 `Left` 内的偏移量——是 0。访问 `val`？直接 `this + 0`，一条指令，零额外开销。

虚继承打破了这个假设。

当 `Left` 虚继承 `Base` 时，`Base` 子对象不再固定在 `Left` 的开头。它的位置取决于最终的派生类——如果 `Left` 是 `Bottom` 的一部分，`Base` 可能被放在整个对象的末尾；如果 `Left` 自己就是最底层对象，`Base` 可能就在紧挨着 `Left` 的后面。同一个类 `Left`，在不同的继承体系里，它到 `Base` 的距离是不一样的。

编译器不能在编译期把这个偏移量写死。怎么办？

**加一个指针，运行时去查。**

这就是 **vbptr（Virtual Base Pointer）**。

### MSVC 的对象布局

MSVC 的做法是在 `Left` 对象里塞一个 vbptr，指向一张 vbtable（虚基类偏移量表）。vbtable 里存的是从 vbptr 的位置到 `Base` 子对象的偏移量。

用 MSVC 的 `/d1reportSingleClassLayout` 看 `Bottom` 的布局：

```
class Bottom size(24):
+---
| +--- (base class Left)
| | {vbptr}                    // Left 的虚基类指针
| +---
| +--- (base class Right)
| | {vbptr}                    // Right 的虚基类指针
| +---
+---
+--- (virtual base Base)
| val                          // 唯一一份 Base 数据
+---
```

两个 vbptr，各 8 字节（64位下），加上 `val` 的 4 字节，再加上对齐填充——`sizeof(Bottom)` 就膨胀到了 24 字节。对比普通继承的 8 字节，内存开销翻了 3 倍。

但内存膨胀只是表面。**真正的代价在运行时。**

## 二、vbptr 间接寻址——每次访问虚基类成员都要"查表"

普通继承下访问 `val`，编译器生成的代码是：

```asm
mov eax, [rcx]          ; this->val，直接偏移，一条指令
```

虚继承下呢？假设你通过 `Left*` 指针访问 `val`：

```asm
mov rax, [rcx]          ; 第1步：读 vbptr（rcx 指向 Left 子对象）
movsxd rdx, [rax + 4]   ; 第2步：从 vbtable 读偏移量（entry 1）
mov eax, [rcx + rdx]    ; 第3步：rcx + 偏移量 = Base 子对象地址，读 val
```

**三条指令，两次内存读取，才拿到一个 `int`。**

这不是理论推演——这是编译器实际生成的代码。用 Compiler Explorer 看 MSVC 的输出，你会清楚地看到这三步。

### 为什么这很贵？

不只是多了两条指令的问题。关键在于**数据依赖链**：

- 第二条指令依赖第一条的结果（需要 vbptr 的值才能查 vbtable）
- 第三条依赖第二条的结果（需要偏移量才能算出 `Base` 的地址）

这三条指令无法并行执行——CPU 的乱序执行引擎在这里帮不上忙，因为每一步都要等上一步的结果从缓存里回来。

如果 vbptr 或 vbtable 不在 L1 缓存里——比如你在遍历一个包含大量虚继承对象的数组——这个代价会进一步放大。每次 cache miss 意味着几十甚至上百个周期的等待。

### GCC/Clang 的实现

GCC 遵循 Itanium C++ ABI，用的是 VTT（Virtual Table Table）机制，思路略有不同但本质一样——都是通过间接寻址在运行时定位虚基类。Itanium ABI 把虚基类偏移量编码进 vtable 的负索引区域（vbase offset），而不是用独立的 vbtable。

但间接寻址的开销是一样的：都需要先读指针、再读偏移、最后算地址。

## 三、thunk——编译器偷偷塞进来的"this 指针校正器"

访问数据成员要查表，那调用虚函数呢？**更贵。**

假设 `Base` 有一个虚函数 `foo()`，`Bottom` 重写了它：

```cpp
struct Base {
    int val = 42;
    virtual void foo() {}
};

struct Left  : virtual Base {};
struct Right : virtual Base {};

struct Bottom : Left, Right {
    void foo() override { val = 100; }
};
```

现在通过 `Left*` 指针调用 `foo()`：

```cpp
Left* p = new Bottom();
p->foo();   // 发生了什么？
```

普通的虚函数调用已经有一层间接寻址了：读 vfptr → 查 vtable → 跳转到函数。虚继承在这之上又加了一层：**this 指针调整**。

### 为什么要调整？

因为 `p` 指向的是 `Bottom` 对象中 `Left` 子对象的起始地址，但 `Bottom::foo()` 期望的 `this` 是 `Bottom` 对象的起始地址。两者之间有偏移。如果不调整，`foo()` 里访问 `val` 的时候，`this + offset` 算出来的地址是错的——轻则读到垃圾数据，重则直接段错误。

编译器的解决方案是生成一个 **thunk**——一小段汇编代码，专门负责在跳转到真正的函数之前调整 `this` 指针。

### thunk 在汇编层面长什么样？

```asm
; --- thunk for Bottom::foo(), 从 Left 子对象入口 ---
sub rcx, 8          ; 把 this 从 Left 子对象调整回 Bottom 起始地址
jmp Bottom::foo     ; 跳转到真正的实现
```

就两条指令。但问题是，这两条指令构成了一次**额外的间接跳转**。

现代 CPU 的分支预测器对间接跳转（indirect jump / indirect call）比较敏感。直接调用一个函数地址，分支预测器几乎不会预测错误；但通过 vtable 间接调用再加上 thunk 的跳转，预测器需要追踪的分支链变长了。在紧密循环里大量调用虚继承体系下的虚函数时，分支预测失败的概率会上升。

### 虚继承下的复杂 thunk

虚继承下的 thunk 比普通多继承的 thunk 更复杂的地方在于：**偏移量可能不是编译期常量。**

普通多继承里，`Left` 子对象到 `Bottom` 起始地址的偏移量在编译期就确定了（比如固定是 8 字节），所以 thunk 里的 `sub rcx, 8` 是个硬编码的常数。但在虚继承里，虚基类的偏移量可能需要从 vbtable 动态查找——这时候 thunk 就不再是简单的 `sub + jmp`：

```asm
; --- 虚继承下的复杂 thunk ---
mov rax, [rcx]          ; 读 vbptr
movsxd rdx, [rax + 4]   ; 从 vbtable 读虚基类偏移
lea rcx, [rcx + rdx]    ; 调整 this 指向虚基类子对象
jmp Target::func        ; 跳转
```

**四条指令，两次内存读取，加上一次地址计算——然后才跳到真正的函数体。** 这个代价在每次虚函数调用时都要付。

还有一个容易忽略的点：thunk 是一段独立的代码，它有自己的地址。这意味着它会占用指令缓存（I-cache）的空间。如果你的继承体系很复杂，编译器可能生成大量的 thunk，这些小代码片段散落在 `.text` 段里，在热路径上可能造成 I-cache 的压力。

## 四、构造函数的额外负担——谁来初始化虚基类？

普通继承下，构造函数的调用链很清晰：`Bottom` 的构造函数调 `Left` 的构造函数，`Left` 调 `Base` 的构造函数——层层向上，每个类负责初始化自己的基类子对象。

虚继承打破了这条链。

因为虚基类在整个对象里只有一份，所以它的初始化不能交给中间层——否则 `Left` 初始化一次 `Base`，`Right` 又初始化一次，同一块内存被写了两遍。C++ 标准规定：

**虚基类的初始化由最底层派生类（most derived class）负责。**

这意味着什么？编译器要在构造函数里插入额外的逻辑来判断"我是不是最底层派生类"：

- 如果是（比如你直接 `new Bottom()`），就初始化虚基类
- 如果不是（比如 `Bottom` 的构造函数内部调用 `Left` 的构造函数时），就跳过虚基类初始化

### 实现方式

在 MSVC 的实现里，这通常表现为构造函数的一个隐含参数——一个标志位，告诉构造函数"你需不需要初始化虚基类"。

在 Itanium ABI（GCC/Clang）里，编译器为每个有虚基类的类生成**两个版本的构造函数**：

- **完整构造函数**（complete constructor）：负责初始化虚基类
- **基类构造函数**（base constructor）：跳过虚基类初始化

无论哪种实现，代价都是：更多的代码分支、更大的代码体积、以及在深继承层次下更长的构造链路。如果你的继承体系有 4-5 层虚继承，编译器生成的构造函数代码量可能是普通继承的 2-3 倍——这不是你手写的代码多了，是编译器替你生成的代码多了。

## 五、账单汇总——三层"过路费"

| 开销层级 | 具体代价 | 触发场景 |
|---------|---------|---------|
| vbptr 间接寻址 | 每次访问虚基类成员多 2 次内存读取 | 读写虚基类的数据成员 |
| thunk this 调整 | 每次虚函数调用多 2-4 条指令 + 间接跳转 | 通过基类指针调虚函数 |
| 构造函数分裂 | 双版本构造函数 / 标志位判断 | 每次构造对象 |
| 对象体积膨胀 | 每个虚继承路径增加 8 字节 vbptr | 永久性内存开销 |

## 六、什么时候该承受这个代价？

这些代价在大多数应用场景里不会成为瓶颈——如果你的虚继承对象不在热路径上，几条额外指令完全可以忽略。

但如果你在写：

- 游戏引擎的实体系统
- 高频交易的订单处理
- 任何需要在紧密循环里操作大量对象的代码

虚继承的这些间接寻址开销就值得认真对待了。I-cache 的 thunk 污染、数据访问的 cache miss 放大、分支预测的额外压力——在每秒执行数百万次调用的场景下，这些"小"开销会累积成可测量的性能差异。

另外一个容易忽略的隐性代价：**虚继承会抑制编译器的内联优化。**

编译器要内联一个虚函数调用，需要在编译期确定调用目标。thunk 引入的间接跳转让这个推断变得更难。对于性能敏感的代码，这可能意味着本来能被优化掉的函数调用开销被"锁死"了。

---

## 七、替代方案

### 1. 组合（Composition）

可以完全避免继承层次带来的间接寻址。

### 2. CRTP（奇异递归模板模式）

在编译期完成多态分派，零运行时开销。

### 3. 接口 + 组合

如果你的设计确实需要运行时多态但又想避免虚继承，把共享基类拆成接口（纯虚类）+ 组合持有数据，通常是更干净的架构选择。

---

## 总结

**虚继承用一个问题（间接寻址开销）换掉了另一个问题（数据冗余），这不是免费的交换——代价藏在 vbptr 的每次查表、thunk 的每次跳转、和构造函数的每次分裂里。**

知道代价在哪，才能判断什么时候值得付。
