## 7.1 相关实际问题

## 概述

在应用程序里，我们使用多少内存都是自己能掌握和控制的。但是总观Linux整台服务器，除了应用程序以外，内核也会申请和管理大量的内存。本章将深入分析内核内存管理以及TCP连接的内存开销。

---

## 问题1：内核是如何管理内存的？

### 问题背景

内核作为整个Linux服务器的基石，它的内存管理方案的优劣将直接影响整台服务器的稳定性。那么内核是如何高效管理和使用内存的呢？

### 初步思考

应用程序申请内存时：
- 使用 `malloc()`、`new()` 等函数
- 最终通过 `brk()`、`mmap()` 系统调用向内核申请

内核自己申请内存时：
- 不能使用用户态的函数
- 需要自己实现一套高效的内存管理机制

### 内核内存管理特点

```
┌─────────────────────────────────────────────────────────────┐
│                    内核内存管理特点                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 不能使用用户态内存分配函数                               │
│     - 内核运行在内核态，无法调用 malloc/free                 │
│     - 需要自己实现内存分配器                                 │
│                                                             │
│  2. 需要高效处理各种大小的内存请求                           │
│     - 小到几个字节（如 struct inode）                        │
│     - 大到几兆字节（如网络缓冲区）                           │
│                                                             │
│  3. 需要减少内存碎片                                        │
│     - 内核对象频繁创建销毁                                   │
│     - 碎片过多影响性能                                       │
│                                                             │
│  4. 需要考虑NUMA架构                                        │
│     - 多CPU多内存控制器                                      │
│     - 本地内存访问更快                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 问题3：一条ESTABLISH状态的空连接需要消耗多少内存？

## 问题4：3万多个TIME_WAIT，内存开销会不会很大？

### 问题背景

这是由第1章中提到的另外一个线上问题引发的思考，3万多个TIME_WAIT会占用多大的内存呢？会不会因为TIME_WAIT过多消耗过多内存而挤占应用程序的可用内存？

### TIME_WAIT状态分析

TIME_WAIT状态的连接：
- 只保留 `struct tcp_sock` 对象
- 不需要 `struct file` 和 `struct socket_alloc`
- 内存开销约 **2KB**

### 计算

```
30000 × 2KB = 60MB
```

**结论**：3万个TIME_WAIT连接约消耗60MB内存，对现代服务器来说影响不大。

## 7.2 Linux内核如何管理内存

## 概述

内核针对自己的应用场景，使用了一种叫作SLAB/SLUB的内存管理机制。这种管理机制通过四个步骤把物理内存条管理起来，供内核申请和分配内核对象。

---

## 7.2.1 node划分

在现代的服务器上，内存和CPU都是所谓的NUMA（Non-Uniform Memory Access）架构。

### NUMA架构示意图

```
│     │  │ Core 0    │  │           │  │ Core 8    │  │       │
│     │  │ Core 1    │  │           │  │ Core 9    │  │       │
│     │  │ ...       │  │           │  │ ...       │  │       │
```

### NUMA的特点

1. **每个CPU有自己的本地内存**
   - CPU访问本地内存更快
   - 访问远端内存需要经过QPI总线

2. **CPU和内存组成node**
   - 每一个CPU以及和它直连的内存条组成了一个node（节点）

## 7.2.2 zone划分

每个node又会划分成若干的zone（区域），zone表示内存中的一块范围。

### zone类型

| zone类型 | 地址范围 | 说明 |
|---------|---------|------|
| **ZONE_DMA** | 最低地址 | 供IO设备DMA访问 |
| **ZONE_DMA32** | 低地址 | 用于支持32位地址总线的DMA设备（64位系统） |
| **ZONE_NORMAL** | 正常地址 | 普通内存区域（64位系统主要区域） |

> **注意**：ZONE_HIGHMEM是32位机时代的产物，现在64位系统基本不用了。

## 7.2.3 基于伙伴系统管理空闲页面

每个zone下面都有大量的页面，Linux使用伙伴系统对这些页面进行高效的管理。

### 伙伴系统数据结构

```c
// file: include/linux/mmzone.h
#define MAX_ORDER 11

struct zone {
    free_area free_area[MAX_ORDER];
};

struct free_area {
    struct list_head free_list[MIGRATE_TYPES];
    unsigned long nr_free;
};
```

### 伙伴系统原理

free_area是一个包含11个元素的数组：

| 索引 | 内存块大小 | 页面数 |
|-----|-----------|--------|
| 0 | 4KB | 1 |
| 1 | 8KB | 2 |
| 2 | 16KB | 4 |
| 3 | 32KB | 8 |
| 4 | 64KB | 16 |
| 5 | 128KB | 32 |
| 6 | 256KB | 64 |
| 7 | 512KB | 128 |
| 8 | 1MB | 256 |
| 9 | 2MB | 512 |
| 10 | 4MB | 1024 |

```
│  │  [0] │  [1] │  [2] │  [3] │ ...  │      [10]        │  │
│  │ 4KB  │ 8KB  │ 16KB │ 32KB │      │      4MB         │  │
```

### 内存分配示例

**申请8KB内存的过程**：

```
步骤1: 到free_area[1]（8KB链表）查询
       └─→ 无可用

步骤2: 到free_area[2]（16KB链表）查询
       └─→ 找到了

步骤3: 拆分成两个8KB的"小伙伴"
       └─→ 使用其中一个

步骤4: 将另一个小伙伴放到8KB链表中
```

## 7.2.4 slab分配器

目前介绍的内存分配都是以页面（4KB）为单位的。对于各个内核运行中实际使用的对象来说，多大的对象都有。如果都直接分配一个4KB的页面来存储的话太铺张了。

### 为什么需要slab

```
问题：内核对象大小不一
- 有的对象有1KB多
- 有的对象只有几百字节
- 有的对象只有几十字节

如果都分配4KB页面，浪费严重！
```

### slab的特点

1. **一个slab内只分配特定大小、甚至是特定的对象**
2. **当一个对象释放内存后，另一个同类对象可以直接使用这块内存**
3. **极大地降低了碎片发生的概率**

### slab数据结构

```c
// file: include/linux/slab_def.h
struct kmem_cache {
    const char *name;
    struct kmem_cache_node **node;
    unsigned int object_size;
    unsigned int size;
    ...
};

// file: mm/slab.h
struct kmem_cache_node {
    struct list_head slabs_partial;  // 半满
    struct list_head slabs_full;     // 满
    struct list_head slabs_free;     // 空
};
```

### slab向伙伴系统申请内存

```c
// file: mm/slab.c
static void *kmem_getpages(struct kmem_cache *cachep, gfp_t flags, int nodeid)
{
    struct page *page;

    flags |= cachep->allocflags;

    if (cachep->flags & SLAB_RECLAIM_ACCOUNT)
        flags |= GFP_RECLAIMABLE;

    // 调用伙伴系统分配页面
    page = alloc_pages_exact_node(nodeid, flags, cachep->gfporder);

    return page_address(page);
}
```

### slab碎片问题

```bash
## cat /proc/slabinfo | grep TCP
TCP    288    384    1984    16    8
```

分析：
- 每个slab占用8个页面：8 × 4096B = 32768B
- 每个对象大小：1984B
- 每个slab存放16个对象：1984 × 16 = 31744B
- 剩余浪费：32768 - 31744 = 1024B（约1KB）

碎片率：1024 / 32768 ≈ 3%，已经非常低了。

## 7.2.5 小结

### 内核内存管理四步骤

```
┌────────────────────────────────────────────────────────┐
│                 内核内存管理步骤                          │
├────────────────────────────────────────────────────────┤
│                                                        │
│  第一步：把所有的内存条和CPU划分成node                    │
│         └── NUMA架构，每个CPU有自己的本地内存             │
│                                                        │
│  第二步：把每一个node划分成zone                          │
│         └── DMA、DMA32、Normal                          │
│                                                        │
│  第三步：每个zone下都用伙伴系统管理空闲页面               │
│         └── 4KB、8KB、16KB...4MB                        │
│                                                        │
│  第四步：内核提供slab分配器为自己专用                    │
│         └── 特定大小对象的缓存                           │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 前三步 vs 第四步

| 步骤 | 服务对象 | 说明 |
|------|---------|------|

## 7.3 TCP连接相关内核对象

## 概述

TCP连接当然也会使用内存，每申请一个内核对象就都需要到相应的slab缓存里申请一块内存。本节看看TCP连接中都使用了哪些内核对象。

## 7.3.1 socket函数直接创建

socket函数会进入sock_create内核函数：

```c
// file: net/socket.c
int __sock_create(struct net *net, int family, int type, int protocol,
                  struct socket **res, int kern)
{
    struct socket *sock;
    int err;

    // 申请struct socket内核对象
    sock = sock_alloc();

    // 调用协议族的创建函数创建sock
    err = pf->create(net, sock, protocol, kern);

    *res = sock;
    return 0;
}
```

### socket_alloc对象申请

在sock_alloc函数中，申请了一个struct socket_alloc内核对象：

```c
// file: include/net/sock.h
struct socket_alloc {
    struct socket socket;
    struct inode vfs_inode;
};
```

socket_alloc内核对象将socket和inode信息关联了起来。

### sock_inode_cache

sock_inode_cache是专门用来存储struct socket_alloc的slab缓存：

```c
// file: net/socket.c
static int __init inodecache(void)
{
    sock_inode_cachep = kmem_cache_create("sock_inode_cache",
                                           sizeof(struct socket_alloc),
                                           (SLAB_HWCACHE_ALIGN |
                                            SLAB_RECLAIM_ACCOUNT |
                                            SLAB_MEM_SPREAD),
                                           init_once);
    return 0;
}
```

### 申请过程

```c
// file: net/socket.c
static struct inode *sock_alloc_inode(struct super_block *sb)
{
    struct socket_alloc *ei;
    struct socket_wq *wq;

    // 从sock_inode_cache slab缓存中申请
    ei = kmem_cache_alloc(sock_inode_cachep, GFP_KERNEL);
    if (!ei)
        return NULL;

    // 申请等待队列
    wq = kmalloc(sizeof(*wq), GFP_KERNEL);
    if (!wq) {
        kmem_cache_free(sock_inode_cachep, ei);
        return NULL;
    }

    init_waitqueue_head(&wq->wait);
    ei->socket.wq = wq;

    return &ei->vfs_inode;
}
```

### inet_create函数

```c
// file: net/ipv4/af_inet.c
static int inet_create(struct net *net, struct socket *sock, int protocol,
                       int kern)
{
    struct sock *sk;
    struct inet_sock *inet;
    struct proto *answer;

    // 查找协议
    answer = inet_protos[protocol];

    // 设置socket操作
    sock->ops = answer->ops;

    // 申请sock对象
    sk = sk_alloc(net, PF_INET, GFP_KERNEL, answer);
    if (!sk)
        goto out;

    // 初始化sock
    sock_init_data(sock, sk);

    return 0;
}
```

## 7.3.2 服务端socket创建

服务端通过accept接收连接时，socket内核对象的创建方式有所不同。

### accept系统调用

```c
// file: net/socket.c
SYSCALL_DEFINE4(accept4, int, fd, struct sockaddr __user *, upeer_sockaddr,
                int __user *, upeer_addrlen, int, flags)
{
    struct socket *sock, *newsock;
    struct file *newfile;

    // 查找监听socket
    sock = sockfd_lookup_light(fd, &err, &fput_needed);

    // 申请新的socket
    newsock = sock_alloc();

    // 申请file
    newfile = sock_alloc_file(newsock, flags, sock->sk->sk_prot_creator->name);

    // 接受连接
    err = sock->ops->accept(sock, newsock, sock->file->f_flags);

    // 安装文件描述符
    fd_install(newfd, newfile);

    return newfd;
}
```

### inet_accept函数

```c
// file: net/ipv4/af_inet.c
int inet_accept(struct socket *sock, struct socket *newsock, int flags)
{
    struct sock *sk1 = sock->sk;
    struct sock *sk2;

    // 从全连接队列获取已建立的连接
    sk2 = sk1->sk_prot->accept(sk1, flags, &err);

    // 将新的sock关联到新socket
    sock_graft(sk2, newsock);
    newsock->state = SS_CONNECTED;

    return 0;
}
```

### 从全连接队列获取

```c
// file: net/ipv4/inet_connection_sock.c
struct sock *inet_csk_accept(struct sock *sk, int flags, int *err)
{
    struct inet_connection_sock *icsk = inet_csk(sk);
    struct request_sock_queue *queue = &icsk->icsk_accept_queue;
    struct request_sock *req;

    // 等待全连接队列非空
    if (reqsk_queue_empty(queue)) {
        long timeo = sock_rcvtimeo(sk, flags & O_NONBLOCK);
        error = inet_csk_wait_for_connect(sk, timeo);
        if (error)
            goto out_err;
    }

    // 从全连接队列取出request_sock
    req = reqsk_queue_remove(queue);
    newsk = req->sk;

    // 释放request_sock
    reqsk_put(req);

    return newsk;
}
```

5. **缓冲区开销可能远大于连接本身**

## 7.5.5 关键结论

### 核心结论

1. **内核内存管理采用四层架构**
   - NUMA Node → Zone → 伙伴系统 → Slab
   - Slab分配器专为内核对象优化

2. **一条TCP空连接消耗约3.4KB内存**
   - tcp_sock: ~2KB
   - socket_alloc: ~1KB
   - file + dentry: ~0.4KB

3. **TIME_WAIT状态约2KB**
   - 只保留tcp_sock
   - 3万个TIME_WAIT约60MB

4. **100万连接约需3.4GB内存**
   - 实际瓶颈是内存而非端口
   - 缓冲区开销可能更大

5. **内存是连接数的主要限制因素**

## 7.1 相关实际问题

### 问题1：内核是如何管理内存的？

内核作为整个Linux服务器的基石，它的内存管理方案的优劣将直接影响整台服务器的稳定性。那么内核是如何高效管理和使用内存的呢？

### 问题2：如何查看内核使用的内存信息？

这是由第1章中提到的另外一个线上问题引发的思考，3万多个TIME_WAIT会占用多大的内存呢？会不会因为TIME_WAIT过多消耗过多内存而挤占应用程序的可用内存？

## 7.2 Linux内核如何管理内存

内核针对自己的应用场景，使用了一种叫作SLAB/SLUB的内存管理机制。这种管理机制通过四个步骤把物理内存条管理起来，供内核申请和分配内核对象。

```
│  │  ┌─────────┐                       ┌─────────┐              │   │
│  │  │ CPU     │                       │ CPU     │              │   │
│  │  │ core... │                       │ core... │              │   │
│  │  │ 内存    │                       │ 内存    │              │   │
│  │  └─────────┘                       └─────────┘              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  第二步：每个node被划分成多个zone，每个zone下包含很多个页面              │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │    node0                              node1                  │   │

│  │  │DMA │DMA32│Normal │              │    Normal        │     │   │

│  └─────────────────────────────────────────────────────────────┘   │

```

## 7.2.4 slab分配器

目前介绍的内存分配都是以页面（4KB）为单位的。对于各个内核运行中实际使用的对象来说，多大的对象都有。如果都直接分配一个4KB的页面来存储的话太铺张了。

### slab特点

- 一个slab内只分配特定大小、甚至是特定的对象
- 当一个对象释放内存后，另一个同类对象可以直接使用这块内存
- 极大地降低了碎片发生的概率

### slab数据结构

```c
// file: include/linux/slab_def.h
struct kmem_cache {
    struct kmem_cache_node **node;
    ...
};

// file: mm/slab.h
struct kmem_cache_node {
    struct list_head slabs_partial;  // 半满
    struct list_head slabs_full;     // 满
    struct list_head slabs_free;     // 空
};
```

## 7.2.5 小结

### 内核内存管理四步骤

```
┌────────────────────────────────────────────┐
│           内核内存管理                       │
├────────────────────────────────────────────┤
│  第一步：把所有的内存条和CPU划分成node        │
│  第二步：把每一个node划分成zone              │
│  第三步：每个zone下都用伙伴系统管理空闲页面   │
│  第四步：内核提供slab分配器为自己专用        │
└────────────────────────────────────────────┘
```

前三步是基础模块，为应用程序分配内存时的请求调页组件也能够用到。但第四步，就是内核的小灶了。内核根据自己的使用场景，量身打造的一套自用的高效内存分配管理机制。

## 7.3 TCP连接相关内核对象

TCP连接当然也会使用内存，每申请一个内核对象就都需要到相应的slab缓存里申请一块内存。

### 7.3.1 socket函数直接创建

socket函数会进入sock_create内核函数：

```c
// file: net/socket.c
int __sock_create(struct net *net, int family, int type, int protocol,
                  struct socket **res, int kern)
{
    // 申请struct socket内核对象
    sock = sock_alloc();

    // 调用协议族的创建函数创建sock
    err = pf->create(net, sock, protocol, kern);
}
```

### socket_alloc对象申请

sock_inode_cache是专门用来存储struct socket_alloc的slab缓存：

```c

static int __init inodecache(void)
{
    sock_inode_cachep = kmem_cache_create("sock_inode_cache",
                                           sizeof(struct socket_alloc),
                                           (SLAB_HWCACHE_ALIGN |
                                            SLAB_RECLAIM_ACCOUNT |

                                           init_once);
```

### TCP对象申请

```c
// file: net/ipv4/tcp_ipv4.c
struct proto tcp_prot = {
    .name      = "TCP",
    .close     = tcp_close,
    .connect   = tcp_v4_connect,
    ...
    .obj_size  = sizeof(struct tcp_sock),
};

// file: net/core/sock.c
int proto_register(struct proto *prot, int alloc_slab)
{
    if (alloc_slab) {
        prot->slab = kmem_cache_create(prot->name, prot->obj_size, 0,
                                       SLAB_HWCACHE_ALIGN | prot->slab_flags,
                                       NULL);
    }
}
```
