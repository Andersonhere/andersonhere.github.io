> **核心问题**：多核 CPU 如何调度？

## 本章大纲

### 10.1 背景：多处理器架构

**单处理器 vs 多处理器**：

```
单处理器:
┌─────────┐
│   CPU   │ ←→ 缓存 ←→ 内存
└─────────┘

多处理器:
┌─────┐   ┌─────┐
│CPU 0│   │CPU 1│
└──┬──┘   └──┬──┘
   ↓          ↓
┌─────┐   ┌─────┐
│缓存0│   │缓存1│
└──┬──┘   └──┬──┘
   └────┬────┘
        ↓
     ┌─────┐
     │内存 │
     └─────┘
```

**缓存一致性问题**：
- 不同 CPU 的缓存可能包含同一内存位置的不同值
- 硬件通过缓存一致性协议（如 MESI）解决

### 10.2 别忘了同步

**问题**：多 CPU 同时访问共享数据结构

```c
// 需要锁保护共享队列
lock(&queue_lock);
enqueue(run_queue, process);
unlock(&queue_lock);
```

### 10.3 最后一个问题：缓存亲和度

**缓存亲和度 (Cache Affinity)**：
- 进程在某个 CPU 上运行时，缓存中积累了该进程的数据
- 迁移到其他 CPU 会导致缓存失效
- 最好让进程保持在同一 CPU 上

### 10.4 单队列调度 (SQMS)

**方案**：所有 CPU 共享一个调度队列

```
           ┌──────────────────────┐
           │    共享调度队列       │
           │  P1 P2 P3 P4 P5 P6   │
           └──────────────────────┘
                    ↓    ↓
              ┌────┐  ┌────┐
              │CPU0│  │CPU1│
              └────┘  └────┘
```

**优点**：
- 实现简单
- 天然负载均衡

**缺点**：
- 锁竞争严重
- 缓存亲和度差

### 10.5 多队列调度 (MQMS)

**方案**：每个 CPU 一个调度队列

```
┌──────────────┐   ┌──────────────┐
│ CPU0 队列    │   │ CPU1 队列    │
│ P1 P3 P5    │   │ P2 P4 P6    │
└──────────────┘   └──────────────┘
       ↓                  ↓
   ┌────┐            ┌────┐
   │CPU0│            │CPU1│
   └────┘            └────┘
```

**优点**：
- 无锁竞争
- 缓存亲和度好

**缺点**：
- 负载不均衡

**解决方案 - 任务迁移**：
- 定期检查队列负载
- 从繁忙队列"窃取"任务
- 工作窃取 (Work Stealing) 算法

### 10.6 Linux 多处理器调度

**历史演变**：

| 调度器 | 版本 | 特点 |
|--------|------|------|
| O(1) | 2.6.0 | 多队列，O(1) 复杂度 |
| CFS | 2.6.23 | 完全公平调度，红黑树 |
| BFS | 衍生版 | 单队列，低延迟 |

**CFS 特点**：
- 虚拟运行时间 (vruntime)
- 每个调度组独立红黑树
- 支持调度域和负载均衡

### 10.7 小结

| 方案 | 负载均衡 | 缓存亲和度 | 锁开销 |
|------|----------|------------|--------|
| SQMS | 好 | 差 | 高 |
| MQMS | 需迁移 | 好 | 低 |
