## 操作系统（Operating System）学习记录

本文梳理操作系统的三大主题：**虚拟化（CPU / 内存）**、**并发**、**持久化**，面向**面试口述与追问**：每个小节尽量给出「定义 → 机制/硬件 → 典型问题 → 可举例」四层结构。

> [!IMPORTANT]
>
> - 操作系统通过哪些**机制**和**策略**来实现虚拟化？
> - 操作系统如何**高效**实现虚拟化？（时间/空间、软硬协同）
> - 需要哪些**硬件**支持？（特权级、MMU、TLB、原子指令、中断/时钟等）

**虚拟化**：把物理资源变成**多个**、**隔离**、对上层**透明**的虚拟资源；典型包括 CPU 虚拟化（进程/调度）、内存虚拟化（地址空间 / 页表）。

下面两段经典示例（来自 OSTEP 风格演示）帮助建立直觉，正文从「一、CPU 虚拟化」起按大纲展开。

### 示例：虚拟化 CPU

```cpp
#include "common.h"
#include <assert.h>
#include <stdio.h>
#include <stdlib.h>
#include <sys/time.h>
int main(int argc, char *argv[])
{
    if (argc != 2) {
        fprintf(stderr, "usage: cpu <string>\n");
        exit(1);
    }
    char *str = argv[1];
    while (1) {
        Spin(1);
        printf("%s\n", str);
    }
    return 0;
}
```

```bash
# 编译：gcc -o cpu cpu.c -Wall
# 同时跑多个进程，观察「交替打印」的假象
./cpu A & ./cpu B & ./cpu C & ./cpu D &
```

**现象**：单核或多核上多个进程看起来在「同时」跑。  
**本质**：OS 在硬件支持下通过**时间片 / 调度**让多个任务**轮流**占用 CPU，提供「无限 vCPU」的**假象**——即 **CPU 虚拟化**。

### 示例：虚拟化内存

```c
#include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
#include "common.h"

int main(int argc, char *argv[])
{
    int *p = malloc(sizeof(int));
    assert(p != NULL);
    printf("(%d) memory address of p: %08x\n",
           getpid(), (unsigned)p);
    *p = 0;
    while (1) {
        Spin(1);
        *p = *p + 1;
        printf("(%d) p: %d\n", getpid(), *p);
    }
    return 0;
}
```

```text
(2134) memory address of p: 00200000
(2134) p: 1
(2134) p: 2
...
```

**现象**：每个进程有自己的指针 `p`，打印的「地址」形式相似，但彼此**隔离**（不同进程各自一份用户空间映射）。  
**本质**：**虚拟地址**经 **MMU + 页表** 映射到物理页；OS 与硬件共同完成 **内存虚拟化**。

## 一、CPU 虚拟化（以进程为主线）

### 1.1 进程：数据结构、创建与状态

- **进程是什么**：资源分配与调度的基本单位（拥有 PCB、地址空间、打开文件表等）；**线程**是 CPU 调度单位（共享地址空间，私有栈与寄存器现场）。
- **PCB / task_struct（Linux）**：PID、状态、优先级、寄存器现场、内存描述符、文件描述符表、调度实体等。
- **状态机（典型）**：新建 → 就绪 → 运行 → 阻塞 → 终止；可追问：**就绪与运行的区别**（是否在 CPU 上）、**阻塞原因**（I/O、锁、sleep 等）。
- **创建**：`fork()` 复制父进程；常配合 `exec()` 替换映像；`clone()` 可细粒度共享（通向线程）。

**面试常问**：`fork()` 后父子谁先调度？（未规定，实现相关）写时复制（CoW）在 fork 里如何省内存？

### 1.2 `fork()` / `wait()` / `exec()` 的设计思想

- **`fork()`**：创建子进程；返回两次（父得子 PID，子得 0）。配合 CoW：先共享物理页，写时再复制，降低开销。
- **`exec*()`**：同一进程内加载新程序，替换代码/数据/堆栈；**不创建新进程**（PID 不变）。
- **`wait()` / `waitpid()`**：父进程回收子进程退出状态，避免**僵尸进程**；与信号 `SIGCHLD` 相关。

**面试常问**：孤儿进程 vs 僵尸进程？守护进程 double-fork 的目的？

### 1.3 受限的直接执行（Limited Direct Execution, LDE）

- **目标**：用户程序在 CPU 上**尽量快**地跑，但 OS 仍能**夺回控制权**。
- **用户态 / 内核态（Ring）**：敏感指令（如修改页表、停 CPU）只能在**高特权级**执行；违规 → 非法指令 / GP fault。
- **系统调用路径**：用户态陷入（`syscall` / `int 0x80`）→ 内核入口 → 查表分发 → 校验参数 → 执行 → 返回用户态。
- **时钟中断**：抢占式多任务的基础；定时器到期 → 内核调度器决定是否切换进程。

**面试常问**：为什么不能在用户态直接操作页表？从用户态到内核态有哪些进入方式？（syscall、中断、异常）

### 1.4 进程调度：算法与取舍

- **评价维度**：周转时间、响应时间、吞吐量、公平性、调度开销、是否饥饿。
- **典型算法**：FCFS、SJF/STCF、时间片 RR、优先级调度、**多级队列 MLQ**。
- **可行性**：单队列 RR 简单公平但交互性差；纯 SJF 需预知运行时间（实际不可行），引出**结合启发式**的调度。

**面试常问**：交互式系统更关心什么指标？批处理呢？

### 1.5 多级反馈队列（MLFQ）

- **动机**：既不知道任务运行时间，又要兼顾**短任务**与**交互**。
- **常见规则（概念）**：多队列不同时间片；新任务从最高优先级入队；用完时间片降级；若主动让出 CPU（如 I/O 前）可保留/升级优先级；定期**.boost** 防饥饿。
- **问题**：恶意程序故意在片尾 yield 骗取高优先级 → 需老化（aging）或统计规则。

**面试常问**：MLFQ 与 MLQ 区别？如何防止饿死长任务？

### 1.6 多处理器调度

- **亲和性（affinity）**：线程固定在某些核跑，利于 cache 局部性；迁移有冷 cache 成本。
- **负载均衡**：每核 runqueue 之间偷任务（Linux CFS 的 `idle_balance` 等）。
- **可扩展性**：全局锁会成为瓶颈 → per-CPU 队列、减少共享。

**面试常问**：为什么多核下「spin 锁」有时比 sleep 更合适？（持有时间短、避免上下文切换）

### 1.7 内存屏障与乱序执行（与并发也有交叉）

- **编译器乱序**：为优化重排无关内存访问；`volatile` 只保证**可见性语义弱**，不等于内存屏障。
- **CPU 乱序**：流水线、Store Buffer、Invalidate Queue 导致 **W→R 等重排**；需 **fence / barrier**（如 x86 `mfence`、ARM `dmb`）或**用锁**（锁隐含 acquire/release 语义）。
- **与锁实现**：自旋锁、无锁队列依赖正确内存序（C++ `memory_order`）。

**面试常问**：MESI 与内存屏障关系？happens-before 与锁？

## 二、内存虚拟化

### 2.1 虚拟内存的目标

- **隔离**：进程 A 不能访问 B 的物理页；内核与用户隔离。
- **透明**：程序仍用连续虚拟地址编程。
- **高效**：TLB、大页、多级页表、延迟分配、CoW。

### 2.2 时分共享对内存模型的要求

- **多道程序**：内存中驻留多进程 → 必须解决**重定位**与**保护**：程序不能写物理任意位置。

### 2.3 汇编直接操作「物理地址」的问题

- 早期或裸机可写绝对地址；多任务下冲突且无法**重定位**到不同物理区域。

### 2.4 静态 / 动态地址转换与 MMU

- **静态重定位**：装载时整体偏移；**不能**在运行时移动、难共享、碎片问题。
- **动态重定位**：**基址 + 界限**（base & bound）：硬件每次访问检查范围；**分段 / 分页**是其泛化。
- **MMU**：虚拟地址 → 物理地址；配合页表项中的 **R/W/X、U/S** 位做权限检查。

### 2.5 分段：碎片与空闲空间管理

- **逻辑段**：代码/堆/栈独立增长；段表项含基址与界限。
- **外部碎片**：空闲块被切成小块，难以满足大块连续请求；**紧凑**代价高。
- **内部碎片**：段内分配单位大于请求产生浪费。

**面试常问**：为什么现代通用 OS 以分页为主、分段为辅（x86 兼容历史）？

### 2.6 分页：页表、页目录与多级结构

- **页**：固定大小块（如 4KiB），消除外部碎片（以页内浪费换整体）。
- **页表**：VPN→PPN；单级页表空间大（地址空间 × 项大小）→ **多级页表**稀疏节省空间。
- **级数确定**：在「页表总大小 / 单次遍历开销 / TLB miss 成本」间折中；x86-64 典型 4～5 级。

**面试常问**：为什么多级页表省空间？缺页时访问几级页表？

### 2.7 大页（Huge Page）

- **收益**：同样地址范围 **TLB 项更少**，TLB miss 降低，适合大内存工作集（数据库、科学计算、DPDK）。
- **代价**：内部碎片增加、伙伴系统分配压力；需 `hugetlbfs` / transparent huge pages 等策略。

### 2.8 TLB 的出现与作用

- **问题**：每次访存多级页表太慢。
- **TLB**：VPN→PPN 的**地址转换缓存**；利用**时间/空间局部性**。
- **进程切换**：需 flush 或 **ASID**（地址空间标识）避免误命中。
- ** shootdown**：多核下某核修改页表后通知其他核无效化 TLB。

### 2.9 缺页异常（Page Fault）与现象

- **合法但未驻留**：换入页（major fault，读盘）；**CoW**：第一次写触发复制。
- **非法访问**：段错误（SIGSEGV）；**栈增长**：guard page 下扩栈触发 fault 再映射。

**面试常问**：minor vs major fault？mmap 文件映射 fault 时读什么？

### 2.10 内存压力大时

- **页置换**：LRU 近似（Clock）、工作集模型、**抖动（thrashing）**（频繁换入换出 CPU 空转）。
- **交换区（swap）**：冷页写到块设备；**OOM Killer**：最后手段。

**面试常问**：Belady 异常？LRU 实现近似为什么用 Clock？

## 三、并发

### 3.1 原子操作、临界区、竞态

- **临界区**：访问共享资源的代码段；**互斥**保证同一时刻至多一个线程进入。
- **竞态条件**：结果依赖调度时序；**数据竞争**（C++11 后精确定义）常与未同步写有关。

### 3.2 锁的基本思想与硬件前提

- **自旋锁**：忙等，适合**短临界区**、多核；**互斥锁**可能睡眠（futex）。
- **硬件原语**：**Test-And-Set**、**Compare-And-Swap（CAS）** 实现原子「读-改-写」；是构建锁与无锁结构的基石。

**面试常问**：CAS 自旋实现 mutex 的 ABA 问题？（无锁数据结构里更突出）

### 3.3 锁的实现与优化目标

- **目标**：低延迟、高吞吐、公平、可扩展（减少 cache line 乒乓）。
- **优化方向**：** MCS 队列锁**、**读写锁**、**分片锁**、**RCU**（读多写少 Linux 内核典型）。

### 3.4 线程安全的数据结构

- **粗粒度锁**：简单但扩展差；**细粒度 / 分片**：按桶/分片降低冲突。
- **无锁**：CAS 链表/队列；难点在内存回收（Hazard Pointer、Epoch）。

**面试常问**：ConcurrentHashMap 思路（分段锁 → CAS + synchronized 演化，依语言/库版本）？

### 3.5 条件变量（condition variable）

- **用法**：`wait(mutex)` 原子释放锁并睡眠；另一线程 `signal/broadcast` 唤醒；醒来后要**while 再检查条件**防虚假唤醒。
- **Mesa vs Hoare**：多数系统为 Mesa 语义（唤醒不立即获得锁，需与信号量区分口述）。

**面试常问**：`if` 判断条件后 wait 有什么错？应 `while`！

### 3.6 信号量（semaphore）

- **作互斥**：二元信号量等价 mutex。
- **作条件同步**：计数表示资源个数；生产者-消费者经典题。
- **实现**：内部仍依赖 mutex + cond 或 spin + 原子计数。

### 3.7 死锁与非死锁问题

- **死锁四条件**：互斥、占有且等待、不可抢占、循环等待；破坏任一即可预防。
- **处理**：银行家（理论）、超时、检测环（资源分配图）、重启。
- **活锁 / 饥饿**：非死锁类活性问题；**优先级反转**与优先级继承/天花板。

### 3.8 基于事件的并发与网络 I/O 模型

- **`select/poll`**：fd 集合扫描 O(n)；**`epoll`（Linux）**：回调 + 就绪队列，适合大量连接。
- **Reactor**：IO 多路复用 + 非阻塞 + 事件分发；**Proactor**：异步 IO 完成回调（Windows IOCP 等）。

**面试常问**：边缘触发 vs 水平触发（epoll）？

## 四、持久化（面试要点）

### 4.1 目标与层次

- **持久化**：崩溃后数据仍可恢复；与易失性内存对比。
- **层次**：设备驱动 → 块层 → 文件系统（VFS）→ 页缓存 / buffer cache → 用户 `read/write`。

### 4.2 文件系统与一致性

- **inode / dentry**：元数据与目录项缓存；硬链接 vs 软链接。
- **崩溃一致性**：只写了一半元数据怎么办 → **日志（journaling）**、**写时复制（COW，如 ZFS/Btrfs 思路）**、**软更新**。
- **`fsync` / `fdatasync`**：把用户数据/元数据刷到稳定存储；性能与一致性权衡。

### 4.3 RAID 与磁盘（简述）

- **RAID 0/1/5/10**：性能、冗余、写惩罚；** wear leveling**（SSD）延长寿命。

### 4.4 可追问方向

- **VFS 四大对象**：superblock、inode、dentry、file。
- **页缓存回写**：`pdflush`/flusher 线程、`dirty_ratio` 等调优概念。

## 五、自述串联（面试 1～2 分钟版）

可以从本文开头的三个大问题收口：**虚拟化**靠特权级 + MMU + 中断/时钟把 CPU 与内存「切开又拼起来」；**并发**靠原子指令上的锁与条件变量，以及调度与 I/O 模型保证活性与性能；**持久化**靠块设备上的文件系统与日志/COW 在崩溃后仍能恢复一致视图。具体机制按岗位可深入：内核岗追问缺页路径、调度类、epoll 实现；应用岗追问锁、内存序、fsync 语义即可。
