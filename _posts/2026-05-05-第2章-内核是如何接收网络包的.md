## 2.1 相关实际问题

在现在的互联网世界里，所有技术岗位的人员几乎都是天天和网络请求打交道。平时我们在做网络开发的时候，如果需要接收网络数据，只需要简单的几行代码就可以搞定。

如果拿 C语言来举例（Java、Golang、PHP等其他语言也是类似的），一行 `read` 函数调用代码就能接收来自对端的数据：

```c
int main() {
    int sock = socket(AF_INET, SOCK_STREAM, 0);
    connect(sock, ...);
    read(sock, buffer, sizeof(buffer) - 1);
}
```

从开发视角来看，只要客户端有对应的数据发送过来，服务端执行 `read` 后就能收到。那你是否深入思考过，**在Linux下数据是如何从网卡一步步地到达你的进程里的，这中间都需要哪几个内核组件进行协同？**

这个问题看起来简单，但实际上隐藏了非常多的技术点：

## 2.2.1 Linux网络收包总览

在 TCP/IP 网络分层模型里，整个协议栈被分成了物理层、链路层、网络层、传输层和应用层：

```
┌─────────────────────────────────────┐
│           应用层                     │
│    HTTP、FTP 等，用户进程             │
│           socket                     │
├─────────────────────────────────────┤
│           传输层                     │
│           TCP、UDP                   │
├─────────────────────────────────────┤
│           网络层                     │
│        IP、ICMP、IGMP                │
├─────────────────────────────────────┤
│           链路层                     │
│         网络设备驱动                  │
├─────────────────────────────────────┤
│           物理层                     │
│          网卡、网线                   │
└─────────────────────────────────────┘
```

**Linux内核以及网卡驱动主要实现链路层、网络层和传输层这三层上的功能**，内核为更上面的应用层提供 socket 接口来支持用户进程访问。

### 源码位置

- 网络设备驱动：`drivers/net/ethernet/`
  - Intel 系列网卡驱动：`drivers/net/ethernet/intel/`
- 协议栈模块代码：`kernel/` 和 `net/` 目录下

### 中断处理：上半部和下半部

内核和网络设备驱动是通过**中断**的方式来处理的：

1. 当设备上有数据到达时，会给 CPU 的相关引脚触发一个电压变化，以通知 CPU 来处理数据
2. 对于网络模块来说，处理过程比较复杂和耗时
3. 如果在中断函数中完成所有的处理，将会导致中断处理函数（优先级过高）过度占用 CPU

因此 **Linux 中断处理函数分为上半部和下半部**：

| 部分 | 特点 | 工作 |
|------|------|------|
| **上半部（硬中断）** | 只进行最简单的工作，快速处理然后释放 CPU | 记录寄存器，修改 poll_list，发出软中断 |
| **下半部（软中断）** | 可以慢慢、从容处理 | 大部分网络包处理工作 |

> Linux 2.4 以后内核版本采用的下半部实现方式是**软中断**，由 `ksoftirqd` 内核线程全权处理。

### 内核收包路径示意图

```
用户态
    ↑
    │  8. 内核唤醒用户进程
    │
    │  7. 协议层开始处理网络帧，数据被放到 socket 的接收队列中
    │
内核态
    │  6. 帧被从 RingBuffer 上摘下来保存为一个 skb
    │  5. ksoftirqd 线程处理软中断，调用驱动注册的 poll 函数开始收包
    │  4. CPU 响应硬中断，简单处理后发出软中断
    │
    │  3. 硬中断通知 CPU
    │  2. 网卡把帧 DMA 到内存
    │  1. 数据帧从外部网络到达网卡
    ↓
┌───────────────────────────────────────┐
│              网卡                      │
│    RingBuffer (DMA 拷贝到的内存)        │
└───────────────────────────────────────┘
```

当网卡收到数据以后：
1. 以 **DMA** 的方式把网卡收到的帧写到内存里的 RingBuffer 中
2. 向 CPU 发起一个中断，通知 CPU 有数据到达
3. CPU 收到中断请求后，调用网络设备驱动注册的中断处理函数
4. 网卡的中断处理函数并不做过多工作，发出软中断请求，尽快释放 CPU 资源
5. `ksoftirqd` 内核线程检测到有软中断请求到达，调用 `poll` 开始轮询收包
6. 收到后交由各级协议栈处理，对于 TCP 包来说，会被放到用户 socket 的接收队列中

## 2.2.2 Linux启动

Linux驱动、内核协议栈等模块在能够接收网卡数据包之前，要做很多的准备工作才行：

- 提前创建好 `ksoftirqd` 内核线程
- 注册好各个协议对应的处理函数
- 网卡设备子系统要提前初始化好
- 网卡要启动好

### 1. 创建 ksoftirqd 内核线程

Linux 的软中断都是在专门的内核线程（`ksoftirqd`）中进行的。该线程数量不是1个，而是 **N个**，其中 **N 等于你的机器的核数**。

```bash
## 查看系统中的 ksoftirqd 线程
ps -ef | grep ksoftirqd
root   13   0 Jan04 ?   00:00:19 [ksoftirqd/0]
root   18   0 Jan04 ?   00:00:47 [ksoftirqd/1]
root   23   0 Jan04 ?   00:00:10 [ksoftirqd/2]
root   28   0 Jan04 ?   00:00:51 [ksoftirqd/3]
```

**初始化流程**：
```
kernel/smpboot.c
    ↓
smpboot_register_percpu_thread()
    ↓
kernel/softirq.c: spawn_ksoftirqd()
    ↓
创建 ksoftirqd 线程，注册线程函数
```

**软中断类型定义**（`include/linux/interrupt.h`）：
```c
enum {
    HI_SOFTIRQ = 0,
    TIMER_SOFTIRQ,
    NET_TX_SOFTIRQ,     // 网络发送软中断
    NET_RX_SOFTIRQ,     // 网络接收软中断
    BLOCK_SOFTIRQ,
    BLOCK_IOPOLL_SOFTIRQ,
    TASKLET_SOFTIRQ,
    SCHED_SOFTIRQ,
    HRTIMER_SOFTIRQ,
    RCU_SOFTIRQ,
    NR_SOFTIRQS
};
```

### 2. 网络子系统初始化

在网络子系统的初始化过程中：
- 为每个 CPU 初始化 `softnet_data`
- 为 `NET_RX_SOFTIRQ` 和 `NET_TX_SOFTIRQ` 注册处理函数

```c
// file: net/core/dev.c
static int __init net_dev_init(void)
{
    for_each_possible_cpu(i) {
        struct softnet_data *sd = &per_cpu(softnet_data, i);
        memset(sd, 0, sizeof(*sd));
        skb_queue_head_init(&sd->input_pkt_queue);
        skb_queue_head_init(&sd->process_queue);
        sd->completion_queue = NULL;
        INIT_LIST_HEAD(&sd->poll_list);
    }

    open_softirq(NET_TX_SOFTIRQ, net_tx_action);
    open_softirq(NET_RX_SOFTIRQ, net_rx_action);
}
subsys_initcall(net_dev_init);
```

### 3. 协议栈注册

内核实现了网络层的 IP 协议，也实现了传输层的 TCP 协议和 UDP 协议。这些协议对应的实现函数分别是：
- `ip_rcv()`
- `tcp_v4_rcv()`
- `udp_rcv()`

内核通过**注册**的方式来实现：

```c
// file: net/ipv4/af_inet.c
static struct packet_type ip_packet_type __read_mostly = {
    .type = cpu_to_be16(ETH_P_IP),
    .func = ip_rcv,
};

static const struct net_protocol udp_protocol = {
    .handler = udp_rcv,
    .err_handler = udp_err,
};

static const struct net_protocol tcp_protocol = {
    .early_demux = tcp_v4_early_demux,
    .handler = tcp_v4_rcv,
    .err_handler = tcp_v4_err,
};

static int __init inet_init(void)
{
    inet_add_protocol(&icmp_protocol, IPPROTO_ICMP);
    inet_add_protocol(&udp_protocol, IPPROTO_UDP);
    inet_add_protocol(&tcp_protocol, IPPROTO_TCP);
    dev_add_pack(&ip_packet_type);
}
```

**关键数据结构**：
| 数据结构 | 存储内容 |
|---------|---------|
| `inet_protos` 数组 | UDP、TCP 的处理函数地址 |
| `ptype_base` 哈希表 | IP 协议的处理函数 `ip_rcv` 地址 |

### 4. 网卡驱动初始化

每一个驱动程序使用 `module_init` 向内核注册一个初始化函数。

以 igb 网卡驱动为例（`drivers/net/ethernet/intel/igb/igb_main.c`）：

```c
static struct pci_driver igb_driver = {
    .name = igb_driver_name,
    .id_table = igb_pci_tbl,
    .probe = igb_probe,
    .remove = igb_remove,
};

static int __init igb_init_module(void)
{
    ret = pci_register_driver(&igb_driver);
    return ret;
}
```

**igb_probe 主要执行的操作**：
1. 获取 MAC 地址
2. DMA 初始化
3. 注册 ethtool 实现函数
4. 注册 `net_device_ops`、`netdev` 等变量
5. **NAPI 初始化，注册 poll 函数**

```c
// 注册 NAPI poll 函数
static int igb_alloc_q_vector(...)
{
    /* initialize NAPI */
    netif_napi_add(adapter->netdev, &q_vector->napi,
                   igb_poll, 64);
}
```

### 5. 启动网卡

当用户执行 `ifconfig eth0 up` 时，会调用到 `igb_open`：

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
static int igb_open(struct net_device *netdev)
{
    // 分配接收和发送队列
    err = igb_setup_all_tx_resources(adapter);
    err = igb_setup_all_rx_resources(adapter);

    // 注册中断处理函数
    err = igb_request_irq(adapter);

    // 启用 NAPI
    for (i = 0; i < adapter->num_q_vectors; i++)
        napi_enable(&(adapter->q_vector[i]->napi));

    // 启用硬中断
    igb_irq_enable(adapter);
}
```

## 2.2.3 迎接数据的到来

### RingBuffer 结构

RingBuffer 是内存中的一块特殊区域，数据结构包括：
- `igb_rx_buffer` 环形队列数组（描述符数组）
- `e1000_adv_rx_desc` 环形队列数组
- 众多的 `skb`

```
┌─────────────────────────────────────────────────────────┐
│                      RingBuffer                         │
├─────────────────────────────────────────────────────────┤
│  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐     │
│  │ skb │ skb │ skb │ skb │ skb │ skb │ skb │ skb │ ... │
│  └──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┘     │
│     │     │     │     │     │     │     │                │
│  ┌──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┐     │
│  │desc │desc │desc │desc │desc │desc │desc │desc │ ... │
│  └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘     │
│         网卡 DMA 写入 →        ← 内核取走并申请新 skb    │
└─────────────────────────────────────────────────────────┘
```

> **注意**：当 RingBuffer 满的时候，新来的数据包将被丢弃。使用 `ifconfig` 命令可以看到 `overruns` 统计。

### 硬中断处理

当数据帧从网络到达网卡后：
1. 网卡将数据帧 DMA 到内存的 RingBuffer 中
2. 向 CPU 发起硬中断通知

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
static irqreturn_t igb_msix_ring(int irq, void *data)
{
    struct igb_q_vector *q_vector = data;

    /* 记录硬件中断频率 */
    igb_write_itr(q_vector);

    napi_schedule(&q_vector->napi);
    return IRQ_HANDLED;
}
```

**napi_schedule 调用链**：
```c
// file: net/core/dev.c
static inline void __napi_schedule(struct softnet_data *sd,
                                   struct napi_struct *napi)
{
    list_add_tail(&napi->poll_list, &sd->poll_list);
    __raise_softirq_irqoff(NET_RX_SOFTIRQ);
}
```

硬中断处理非常简单：
1. 记录寄存器
2. 修改 CPU 的 `poll_list`
3. 发出软中断 `NET_RX_SOFTIRQ`

### ksoftirqd 内核线程处理软中断

网络包的接收处理过程主要都在 `ksoftirqd` 内核线程中完成：

```c
// file: kernel/softirq.c
static int ksoftirqd_should_run(unsigned int cpu)
{
    return local_softirq_pending();
}

static void run_ksoftirqd(unsigned int cpu)
{
    local_irq_disable();
    if (local_softirq_pending()) {
        __do_softirq();
    }
    local_irq_enable();
}
```

**net_rx_action 函数**：
```c
// file: net/core/dev.c
static void net_rx_action(struct softirq_action *h)
{
    struct softnet_data *sd = &__get_cpu_var(softnet_data);
    unsigned long time_limit = jiffies + 2;
    int budget = netdev_budget;
    void *have;

    local_irq_disable();

    while (!list_empty(&sd->poll_list)) {
        n = list_first_entry(&sd->poll_list, struct napi_struct, poll_list);
        work = 0;

        if (test_bit(NAPI_STATE_SCHED, &n->state)) {
            work = n->poll(n, weight);
            budget -= work;
        }
        ...
    }
}
```

对于 igb 网卡，`poll` 函数就是 `igb_poll`：

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
static int igb_poll(struct napi_struct *napi, int budget)
{
    if (q_vector->tx.ring)
        clean_complete = igb_clean_tx_irq(q_vector);
    if (q_vector->rx.ring)
        clean_complete &= igb_clean_rx_irq(q_vector, budget);
    ...
}
```

### IP 层处理

```c
// file: net/ipv4/ip_input.c
int ip_rcv(struct sk_buff *skb, ...)
{
    return NF_HOOK(NFPROTO_IPV4, NF_INET_PRE_ROUTING, skb, dev, NULL,
                   ip_rcv_finish);
}
```

**NF_HOOK 就是 iptables/netfilter 过滤点**。如果有很多或很复杂的 netfilter 规则，会在这里消耗过多的 CPU 资源。

```c
static int ip_local_deliver_finish(struct sk_buff *skb)
{
    int protocol = ip_hdr(skb)->protocol;
    const struct net_protocol *ipprot;

    ipprot = rcu_dereference(inet_protos[protocol]);
    if (ipprot != NULL) {
        ret = ipprot->handler(skb);  // 调用 tcp_v4_rcv 或 udp_rcv
    }
}
```

## 2.2.4 收包小结

网络模块是 Linux 内核中最复杂的模块了，看起来一个简简单单的收包过程就涉及许多内核组件之间的交互。

### 准备工作

在开始收包之前，Linux 要做许多的准备工作：

1. **创建 ksoftirqd 线程**：为它设置好它自己的线程函数，后面指望着它来处理软中断
2. **协议栈注册**：Linux 要实现许多协议，比如 ARP、ICMP、IP、UDP 和 TCP，每一个协议都会将自己的处理函数注册一下
3. **网卡驱动初始化**：每个驱动都有一个初始化函数，内核会让驱动也初始化一下。在这个初始化过程中，把自己的 DMA 准备好，把 NAPI 的 poll 函数地址告诉内核
4. **启动网卡**：分配 RX、TX 队列，注册中断对应的处理函数

### 数据到来时的处理

当数据到来以后：

1. **网卡**：将数据帧 DMA 到内存的 RingBuffer 中，然后向 CPU 发起中断通知
2. **CPU**：响应中断请求，调用网卡启动时注册的中断处理函数
3. **中断处理函数**：几乎没干什么，只发起了软中断请求
4. **ksoftirqd 线程**：发现有软中断请求到来，先关闭硬中断
5. **poll 函数**：开始调用驱动的 poll 函数收包
6. **协议栈**：poll 函数将收到的包送到协议栈注册的 ip_rcv 函数中
7. **传输层**：ip_rcv 函数将包送到 udp_rcv/tcp_v4_rcv 函数中
8. **用户进程**：数据被放到 socket 的接收队列中，唤醒等待的用户进程

## 2.3 本章总结

本章讲述了网络包是如何一步一步地从网卡、RingBuffer 最后到接收缓存区中的，然后内核又是如何进一步处理把它送到协议栈的。

## 问题回顾

### 问题1：RingBuffer到底是什么，RingBuffer为什么会丢包？

**RingBuffer 结构**：
RingBuffer 是内存中的一块特殊区域，包括：
- `igb_rx_buffer` 环形队列数组
- `e1000_adv_rx_desc` 环形队列数组
- 众多的 `skb`

**关于内存分配的说法**：
- 指针数组是**预先分配好的**
- `skb` 会随着收包过程而**动态申请**

**查看 RingBuffer 设置**：
```bash
## ethtool -g eth0
Ring parameters for eth0:
Pre-set maximums:
RX:             4096
RX Mini:        0
RX Jumbo:       0
TX:             4096
Current hardware settings:
RX:             512
RX Mini:        0
RX Jumbo:       0
TX:             512
```

**查看丢包情况**：
```bash
## ethtool -S eth0
rx_fifo_errors: 0
tx_fifo_errors: 0

## 或者 ifconfig
## overruns 指标如果不为 0，表示有包因为 RingBuffer 装不下而被丢弃了
```

**解决丢包问题**：

1. **加大 RingBuffer 大小**：
```bash
## ethtool -G eth1 rx 4096 tx 4096
```
> 副作用：排队的包过多会增加处理网络包的延时

2. **让内核处理网络包的速度更快**（更好的方案）：
   - 开启多队列网卡
   - 优化软中断处理

### 问题2：网络相关的硬中断、软中断都是什么？

| 类型 | 触发方式 | 工作内容 | CPU消耗显示 |
|------|---------|---------|-------------|
| **硬中断** | CPU 物理引脚电压变化 | 记录寄存器、修改 poll_list、发出软中断 | `hi` |
| **软中断** | 内存变量标记 | 大部分网络包处理工作 | `si` |

**查看中断统计**：
```bash
## 查看软中断统计
cat /proc/softirqs

## 查看硬中断统计
cat /proc/interrupts
```

### 问题3：Linux里的 ksoftirqd 内核线程是干什么的？

```bash
## 查看系统中的 ksoftirqd 线程
ps -ef | grep ksoftirqd
root   13   0 Jan04 ?   00:00:19 [ksoftirqd/0]
root   18   0 Jan04 ?   00:00:47 [ksoftirqd/1]
root   23   0 Jan04 ?   00:00:10 [ksoftirqd/2]
root   28   0 Jan04 ?   00:00:51 [ksoftirqd/3]
```

- 机器上有**几个核**，内核就会创建**几个 ksoftirqd 线程**
- 内核线程 ksoftirqd 包含了所有的软中断处理逻辑
- 对于 `NET_RX_SOFTIRQ` 来说是 `net_rx_action` 函数

### 问题4：为什么网卡开启多队列能提升网络性能？

**原理**：
- 硬中断在哪个 CPU 上被响应，软中断也是在这个 CPU 上处理
- 如果发现 Linux 软中断的 CPU 消耗都集中在一个核上
- 正确的做法应该是**调整硬中断的 CPU 亲和性**，将硬中断打散到不同的 CPU 核上去

**操作方法**：
```bash
## 查看网卡队列的中断号
cat /proc/interrupts | grep eth0

## 设置中断亲和性（将中断分散到不同 CPU）
echo 1 > /proc/irq/irq号/smp_affinity
echo 2 > /proc/irq/irq号/smp_affinity
## ...
```

### 问题5：tcpdump是如何工作的？

tcpdump 通过**虚拟协议**的方式工作：
- 将抓包函数以协议的形式挂到 `ptype_all` 上
- 在 `__netif_receive_skb_core` 中遍历所有"协议"进行抓包

```c
// file: net/packet/af_packet.c
static int packet_create(...)
{
    po->prot_hook.func = packet_rcv;
    register_prot_hook(sk);
}
```

### 问题7：tcpdump能否抓到被iptables封禁的包？

**可以！**

因为 tcpdump 的抓包点在 `__netif_receive_skb_core`，位于 iptables 过滤之前。

数据流：
```
网卡 → RingBuffer → 硬中断 → 软中断 → tcpdump抓包点 → iptables过滤 → 协议栈
```

### 问题9：DPDK是什么？

DPDK 是一种 **Kernel-Bypass** 技术：
- 绕过内核，直接在用户态处理网络包
- 避免了中断、上下文切换等开销
- 适用于极高网络性能需求的场景

当你掌握了 Linux 内核的网络处理过程后，再看 DPDK 这类技术就容易理解了。

## 关键知识点总结

| 组件 | 作用 |
|------|------|
| **网卡** | 接收数据，DMA 到 RingBuffer |
| **RingBuffer** | 数据中转站，满则丢包 |
| **硬中断** | 快速响应，通知软中断 |
| **软中断** | 主要处理工作 |
| **ksoftirqd** | 软中断处理线程 |
| **协议栈** | IP/TCP/UDP 处理 |
| **socket** | 用户进程接口 |

**性能优化要点**：
1. 合理设置 RingBuffer 大小
2. 开启多队列网卡，分散中断到多个 CPU
3. 避免过多的 iptables 规则
4. 必要时考虑 DPDK 等新技术
