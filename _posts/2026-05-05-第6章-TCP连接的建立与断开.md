## 问题7：建立一条TCP连接需要消耗多长时间？

接口耗时是衡量服务接口的重要指标之一。接口很多时候都需要和其他的服务器建立连接然后获取一些必要数据。那么，你知道建立一条TCP连接大概需要多长时间吗？

### TCP连接建立时间分析

**正常情况下的耗时**：

```
客户端                                服务端
  │                                     │
  │ ──── SYN ─────────────────────────► │  RTT/2
  │                                     │
  │ ◄─── SYN+ACK ──────────────────── │  RTT/2
  │                                     │
  │ ──── ACK ─────────────────────────► │  RTT/2
  │                                     │

总耗时：约 1.5 × RTT
```

**不同场景的RTT**：

| 场景 | RTT | 连接耗时 |
|------|-----|---------|
| 本机 | ~0.1ms | ~0.15ms |
| 同机房 | ~0.5ms | ~0.75ms |
| 同城市 | ~2ms | ~3ms |
| 跨省 | ~20ms | ~30ms |
| 跨国（中美） | ~200ms | ~300ms |

## 问题8：把服务器部署在北京，给纽约的用户访问可行吗？

假如中国和美国之间的网络设备非常通畅，我们要建一个网站给美国用户访问。是否可以为了省事直接在北京部署服务器让美国用户来使用？

### 答案分析

**不建议**，原因：

1. **RTT延迟大**：中美之间RTT约200ms，连接建立就需要300ms
2. **用户体验差**：每个请求都有额外的网络延迟
3. **建议方案**：使用CDN或部署在用户附近的服务器

## 问题9：服务器负载很正常，但是CPU被打到底了是怎么回事？

### 问题现象

这是一台4核的虚拟机。按照对负载的正常理解，4核的服务器负载在4以下都算是正常的。这台机器出故障的时候负载不高，只有3左右，但是CPU却被打到了100%。

### 问题原因

这通常是**端口耗尽**导致的。当 `ip_local_port_range` 中的端口快被用光时：
- 内核需要执行很多轮循环才能找到可用端口
- connect系统调用的CPU开销大幅上涨
- 导致CPU使用率飙升

### 解决方法

```bash
## 扩大端口范围
sysctl -w net.ipv4.ip_local_port_range="1024 65535"

## 开启端口复用
sysctl -w net.ipv4.tcp_tw_reuse=1

## 查看当前端口使用情况
netstat -ant | grep TIME_WAIT | wc -l
```

## 6.2 深入理解listen

## 概述

在服务端程序里，在开始接收请求之前都需要先执行listen系统调用。计算机系的学生就像背八股文一样记着服务端socket程序流程：先bind，再listen，然后才能accept。至于为什么需要先listen一下才可以accept，大家平时关注得太少了。

---

## 6.2.1 listen系统调用

可以在net/socket.c下找到listen系统调用的源码：

```c
// file: net/socket.c
SYSCALL_DEFINE2(listen, int, fd, int, backlog)
{
    struct socket *sock;
    int err, fput_needed;
    int somaxconn;

    // 根据fd查找socket内核对象
    sock = sockfd_lookup_light(fd, &err, &fput_needed);

    if (sock) {
        // 获取内核参数net.core.somaxconn
        somaxconn = sock_net(sock->sk)->core.sysctl_somaxconn;

        // backlog不能超过somaxconn
        if ((unsigned int)backlog > somaxconn)
            backlog = somaxconn;

        // 调用协议栈注册的listen函数
        err = sock->ops->listen(sock, backlog);
    }

    return err;
}
```

### 关键点

1. **用户态的socket文件描述符只是一个整数**，内核是没有办法直接用的。所以该函数中第一行代码就是根据用户传入的文件描述符来查找对应的socket内核对象。

2. **获取了系统里的net.core.somaxconn内核参数的值**，和用户传入的backlog比较后取一个最小值传入下一步。

3. **虽然listen允许我们传入backlog**（该值和半连接队列、全连接队列都有关系），但是如果用户传入的值比net.core.somaxconn还大的话是不会起作用的。

---

## 6.2.2 协议栈listen

sock->ops->listen指针指向的是inet_listen函数：

```c
// file: net/ipv4/af_inet.c
int inet_listen(struct socket *sock, int backlog)
{
    struct sock *sk = sock->sk;
    unsigned char old_state = sk->sk_state;
    int err;

    // 还不是listen状态（尚未listen过）
    if (old_state != TCP_LISTEN) {
        // 开始监听
        err = inet_csk_listen_start(sk, backlog);
    }

    // 设置全连接队列长度
    sk->sk_max_ack_backlog = backlog;

    return 0;
}
```

### 关键发现

**`sk->sk_max_ack_backlog`是全连接队列的最大长度。**

服务端的全连接队列长度是执行listen函数时传入的backlog和net.core.somaxconn之间较小的那个值。

> **注意**：如果你在线上遇到了全连接队列溢出的问题，想加大该队列长度，那么可能需要同时考虑执行listen函数时传入的backlog和net.core.somaxconn。

## 6.2.3 接收队列定义

icsk->icsk_accept_queue定义在inet_connection_sock下，是一个request_sock_queue类型的对象，是内核用来接收客户端请求的主要数据结构。

```
┌──────────────────────────────────────────────────────────────┐
│                     接收队列数据结构                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  struct socket                                               │
│      │                                                       │
│      └── struct sock *sk                                     │
│              │                                               │
│              ▼                                               │
│  struct inet_connection_sock                                 │
│      │                                                       │
│      └── struct request_sock_queue icsk_accept_queue         │
│              │                                               │
│              ▼                                               │
│  struct request_sock_queue                                   │
│      │                                                       │
│      ├── 全连接队列（链表）                                   │
│      │   ├── struct request_sock *rskq_accept_head           │
│      │   └── struct request_sock *rskq_accept_tail           │
│      │                                                       │
│      └── 半连接队列（哈希表）                                  │
│          └── struct listen_sock *listen_opt                  │
│                  └── struct request_sock *syn_table[0]       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 源码定义

```c
// file: include/net/inet_connection_sock.h
struct inet_connection_sock {
    struct inet_sock icsk_inet;
    struct request_sock_queue icsk_accept_queue;
    ...
};

// file: include/net/request_sock.h
struct request_sock_queue {
    // 全连接队列
    struct request_sock *rskq_accept_head;
    struct request_sock *rskq_accept_tail;

    // 半连接队列
    struct listen_sock *listen_opt;
};

struct listen_sock {
    u8 max_qlen_log;
    u32 nr_table_entries;
    struct request_sock *syn_table[0];  // 哈希表
};
```

### 两种队列的特点

| 队列类型 | 数据结构 | 原因 |
|---------|---------|------|
| **全连接队列** | 链表 | 只需要先进先出地接受 |
| **半连接队列** | 哈希表 | 第三次握手时需要快速查找 |

## 6.2.4 接收队列申请和初始化

在reqsk_queue_alloc函数中完成了接收队列内核对象的创建和初始化：

```c
// file: net/core/request_sock.c
int reqsk_queue_alloc(struct request_sock_queue *queue,
                      unsigned int nr_table_entries)
{
    size_t lopt_size = sizeof(struct listen_sock);
    struct listen_sock *lopt;

    // 计算半连接队列的长度
    nr_table_entries = min_t(u32, nr_table_entries, sysctl_max_syn_backlog);
    nr_table_entries = max_t(u32, nr_table_entries, 8);
    nr_table_entries = roundup_pow_of_two(nr_table_entries + 1);

    // 为listen_sock对象申请内存（包含半连接队列）
    lopt_size += nr_table_entries * sizeof(struct request_sock *);
    if (lopt_size > PAGE_SIZE)
        lopt = vzalloc(lopt_size);
    else
        lopt = kzalloc(lopt_size, GFP_KERNEL);

    // 全连接队列头初始化
    queue->rskq_accept_head = NULL;

    // 半连接队列设置
    lopt->nr_table_entries = nr_table_entries;
    queue->listen_opt = lopt;

    return 0;
}
```

### 注意细节

半连接队列上每个元素分配的是一个指针大小（sizeof(struct request_sock *)）。这其实是一个**哈希表**。

真正的半连接用的request_sock对象是在握手过程中分配的，计算完哈希值后挂到这个哈希表上。

---

## 6.2.5 半连接队列长度计算

### 计算步骤

传进来的nr_table_entries是内核参数net.core.somaxconn和用户调用listen时传入的backlog二者之间的较小值。

在这个reqsk_queue_alloc函数里，将会完成三次的对比和计算：

```
步骤1: min(nr_table_entries, sysctl_max_syn_backlog)
步骤2: max(结果, 8)  // 保证不能比8小
步骤3: roundup_pow_of_two(结果 + 1)  // 上对齐到2的整数次幂
```

### 实例计算

**案例1**：somaxconn = 128, tcp_max_syn_backlog = 8192, backlog = 5

```
1. min(5, 128) = 5
2. min(5, 8192) = 5
3. max(5, 8) = 8
4. roundup_pow_of_two(8 + 1) = 16

结果：16
```

**案例2**：somaxconn = 128, tcp_max_syn_backlog = 8192, backlog = 512

```
1. min(512, 128) = 128
2. min(128, 8192) = 128
3. max(128, 8) = 128
4. roundup_pow_of_two(128 + 1) = 256

结果：256
```

### 结论

**半连接队列的长度是 `min(backlog, somaxconn, tcp_max_syn_backlog) + 1` 再上取整到2的N次幂，但最小不能小于16。**

> **注意**：如果你在线上遇到了半连接队列溢出的问题，想加大该队列长度，那么就需要同时考虑somaxconn、backlog和tcp_max_syn_backlog这三个内核参数。网上任何告诉你修改某一个参数就能提高半连接队列长度的文章都是错的。

## 6.2.6 listen过程小结

### listen最主要的工作

通过本节对listen源码的简单浏览，我们发现listen最主要的工作就是**申请和初始化接收队列**，包括：

1. **全连接队列**：是一个链表
2. **半连接队列**：由于需要快速查找，使用的是一个哈希表

### 内核如何确定队列长度

**全连接队列长度**：
- listen时传入的backlog和net.core.somaxconn之间较小的那个值
- 需要调整backlog和somaxconn

**半连接队列长度**：
- `min(backlog, somaxconn, tcp_max_syn_backlog) + 1` 再上取整到2的N次幂
- 需要一并考虑backlog、somaxconn和tcp_max_syn_backlog这三个参数

### 为什么需要listen

全/半两个队列是三次握手中很重要的两个数据结构，有了它们服务端才能正常响应来自客户端的三次握手。所以服务端都需要调用listen才行。

## 6.3.1 connect调用链展开

### connect系统调用入口

```c
// file: net/socket.c
SYSCALL_DEFINE3(connect, int, fd, struct sockaddr __user *, uservaddr,
                int, addrlen)
{
    struct socket *sock;
    struct sockaddr_storage address;
    int err, fput_needed;

    // 根据fd查找socket内核对象
    sock = sockfd_lookup_light(fd, &err, &fput_needed);
    if (!sock)
        goto out;

    // 复制用户传入的地址到内核
    err = move_addr_to_kernel(uservaddr, addrlen, (struct sockaddr *)&address);
    if (err < 0)
        goto out_put;

    // 调用协议栈的connect函数
    err = sock->ops->connect(sock, (struct sockaddr *)&address, addrlen,
                              sock->file->f_flags);

out_put:
    fput_light(sock->file, fput_needed);
out:
    return err;
}
```

### inet_stream_connect

```c
// file: net/ipv4/af_inet.c
int inet_stream_connect(struct socket *sock, struct sockaddr *uaddr,
                        int addr_len, int flags)
{
    struct sock *sk = sock->sk;
    int err;

    lock_sock(sk);

    switch (sock->state) {
    default:
        err = -EINVAL;
        goto out;
    case SS_CONNECTED:
        err = -EISCONN;
        goto out;
    case SS_CONNECTING:
        err = -EALREADY;
        goto out;
    case SS_UNCONNECTED:
        err = -EISCONN;
        if (sk->sk_state != TCP_CLOSE)
            goto out;

        // 调用TCP的connect函数
        err = sk->sk_prot->connect(sk, uaddr, addr_len);
        if (err < 0)
            goto out;

        sock->state = SS_CONNECTING;

        // 等待连接完成
        err = inet_wait_for_connect(sk, timeo);
        if (!err)
            sock->state = SS_CONNECTED;
    }

out:
    release_sock(sk);
    return err;
}
```

### tcp_v4_connect

```c
// file: net/ipv4/tcp_ipv4.c
int tcp_v4_connect(struct sock *sk, struct sockaddr *uaddr, int addr_len)
{
    struct sockaddr_in *usin = (struct sockaddr_in *)uaddr;
    struct inet_sock *inet = inet_sk(sk);
    struct tcp_sock *tp = tcp_sk(sk);
    struct rtable *rt;
    __be32 daddr, nexthop;
    int tmp;

    // 设置目的地址
    daddr = usin->sin_addr.s_addr;
    nexthop = daddr;

    // 路由查找
    rt = ip_route_connect(nexthop, inet->inet_saddr,
                          RT_CONN_FLAGS(sk), ip_hdr(skb)->protocol,
                          inet->inet_sport, usin->sin_port, sk);

    // 设置socket状态为TCP_SYN_SENT
    tcp_set_state(sk, TCP_SYN_SENT);

    // 动态选择一个可用端口
    err = inet_hash_connect(&tcp_death_row, sk);
    if (err)
        goto failure;

    // 发送SYN包
    err = tcp_connect(sk);
    if (err)
        goto failure;

    return 0;

failure:
    tcp_set_state(sk, TCP_CLOSE);
    return err;
}
```

### __inet_hash_connect

```c
// file: net/ipv4/inet_hashtables.c
int __inet_hash_connect(struct inet_timewait_sock *death_row,
                        struct sock *sk, u32 port_offset,
                        int (*check_established)(struct inet_timewait_sock *,
                                                 struct sock *, __u16,
                                                 struct inet_timewait_sock **),
                        int (*hash)(struct sock *, struct inet_timewait_sock *))
{
    struct inet_sock *inet = inet_sk(sk);
    struct inet_timewait_sock *tw = NULL;
    struct inet_bind_bucket *tb;
    int port;
    int ret;

    // 从 ip_local_port_range 范围内选择端口
    // port_offset 是随机偏移量
    for (port = inet_sk_get_local_port_range(inet, &low, &high);
         port <= high;
         port++) {
        // 检查端口是否可用
        ret = check_established(death_row, sk, port, &tw);
        if (ret == 0) {
            // 找到可用端口
            inet->inet_num = port;
            return 0;
        }
    }

    return -EADDRNOTAVAIL;  // "Cannot assign requested address"
}
```

## 6.3.3 端口被使用过怎么办

### 检查端口可用性

```c
// file: net/ipv4/inet_hashtables.c
static int __inet_check_established(struct inet_timewait_sock *death_row,
                                    struct sock *sk, __u16 lport,
                                    struct inet_timewait_sock **twp)
{
    struct inet_sock *inet = inet_sk(sk);
    struct inet_timewait_sock *tw;
    struct inet_bind_bucket *tb;
    struct hlist_node *node;

    // 查找该端口是否已被使用
    inet_bind_bucket_for_each(tb, node, &head->chain) {
        if (net_eq(ib_net(tb), net) && tb->port == lport) {
            // 端口已被使用，检查是否可以复用
            if (!check_established(death_row, sk, lport, twp))
                return 0;
        }
    }

    return -EADDRNOTAVAIL;
}
```

### tcp_connect函数

```c
// file: net/ipv4/tcp_output.c
int tcp_connect(struct sock *sk)
{
    struct tcp_sock *tp = tcp_sk(sk);
    struct sk_buff *buff;
    int err;

    // 初始化连接参数
    tcp_connect_init(sk);

    // 申请skb并构造为SYN包
    buff = alloc_skb(MAX_TCP_HEADER + 15, sk->sk_allocation);
    if (!buff)
        return -ENOBUFS;

    // 设置SYN标志
    tcp_init_nondata_skb(buff, tp->write_seq++, TCPHDR_SYN);

    // 添加到发送队列
    tcp_connect_queue_skb(sk, buff);

    // 实际发出SYN
    err = tp->fastopen_req ? tcp_send_syn_data(sk, buff) :
                             tcp_transmit_skb(sk, buff, 1, sk->sk_allocation);
    if (err)
        return err;

    // 启动重传定时器
    inet_csk_reset_xmit_timer(sk, ICSK_TIME_RETRANS,
                               inet_csk(sk)->icsk_rto, TCP_RTO_MAX);

    return 0;
}
```

```c
// file: net/ipv4/tcp_ipv4.c
int tcp_v4_conn_request(struct sock *sk, struct sk_buff *skb)
{
    // 检查半连接队列是否满了
    if (inet_csk_reqsk_queue_is_full(sk) && !isn) {
        want_cookie = tcp_syn_flood_action(sk, skb, "TCP");
        if (!want_cookie)
            goto drop;  // 队列满且未开启syncookies，直接丢弃
    }

    // 检查全连接队列是否满
    if (sk_acceptq_is_full(sk) && inet_csk_reqsk_queue_young(sk) > 1) {
        NET_INC_STATS_BH(sock_net(sk), LINUX_MIB_LISTENOVERFLOWS);
        goto drop;
    }

    // 分配request_sock内核对象
    req = inet_reqsk_alloc(&tcp_request_sock_ops);

    // 构造SYNACK包
    skb_synack = tcp_make_synack(sk, dst, req, ...);

    // 发送SYNACK
    err = ip_build_and_send_pkt(skb_synack, sk, ireq->loc_addr,
                                 ireq->rmt_addr, ireq->opt);

    // 添加到半连接队列，并开启计时器
    inet_csk_reqsk_queue_hash_add(sk, req, TCP_TIMEOUT_INIT);
}
```

## 6.4.5 服务端accept

```c
// file: net/ipv4/inet_connection_sock.c
struct sock *inet_csk_accept(struct sock *sk, int flags, int *err)
{
    struct inet_connection_sock *icsk = inet_csk(sk);
    struct request_sock_queue *queue = &icsk->icsk_accept_queue;
    struct request_sock *req;
    struct sock *newsk;

    // 等待全连接队列非空
    if (reqsk_queue_empty(queue)) {
        long timeo = sock_rcvtimeo(sk, flags & O_NONBLOCK);
        error = inet_csk_wait_for_connect(sk, timeo);
        if (error)
            goto out_err;
    }

    // 从全连接队列取出request_sock
    req = reqsk_queue_remove(queue);
    newsk = req->sk;

    // 释放request_sock
    reqsk_put(req);

    return newsk;
}
```

```
│                                                             │
│  ┌────────────┐                                            │
│  │ 发送SYN    │                                            │
│  │ 启动定时器 │                                            │
│  └─────┬──────┘                                            │
│        │                                                    │
│        │ ──────── SYN ────────► X (丢弃)                   │
│        │                                                    │
│        │ ◄───── 无响应 ───────                              │
│        │                                                    │
│  ┌─────┴──────┐                                            │
│  │ 1秒后重传  │                                             │
│  │ SYN       │                                              │
│  └────────────┘                                            │
│                                                             │
│  重传时间：1s → 3s → 7s → 15s → 31s → 63s                   │
│  最多重传：tcp_syn_retries 次（默认5次）                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 半连接队列满导致丢包

```c
// file: net/ipv4/tcp_ipv4.c
int tcp_v4_conn_request(struct sock *sk, struct sk_buff *skb)
{
    // 检查半连接队列是否满了
    if (inet_csk_reqsk_queue_is_full(sk) && !isn) {
        want_cookie = tcp_syn_flood_action(sk, skb, "TCP");
        if (!want_cookie)
            goto drop;  // 队列满且未开启syncookies，直接丢弃
    }
    ...
drop:
    reqsk_free(req);
    return 0;
}
```

### 重传定时器配置

```bash
## 查看/设置SYN重传次数
sysctl net.ipv4.tcp_syn_retries
sysctl -w net.ipv4.tcp_syn_retries=5  # 默认5次

## 重传时间表（指数退避）
## 第1次：1秒
## 第2次：3秒
## 第3次：7秒
## 第4次：15秒
## 第5次：31秒
## 总计：最长约63秒后放弃
```

### SYN Cookies机制

当开启SYN Cookies时，即使半连接队列满，也不会直接丢弃SYN：

```bash
## 开启SYN Cookies
sysctl -w net.ipv4.tcp_syncookies=1
```

**工作原理**：
1. 服务端不分配半连接队列资源
2. 通过加密算法生成序列号（包含连接信息）
3. 收到ACK时，验证序列号并恢复连接信息

## 6.5.4 连接建立失败的影响

### 对客户端的影响

| 情况 | 影响 | 处理方式 |

| 半连接队列满 | 新连接无法建立 | 开启SYN Cookies |

## 6.7.1 listen的工作原理

### 核心工作

```
┌─────────────────────────────────────────────────────────────┐
│                    listen核心工作                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 状态变更                                                │
│     CLOSED → TCP_LISTEN                                     │
│                                                             │
│  2. 分配接收队列                                            │
│     ┌─────────────────────────────────────────────────┐    │
│     │  struct request_sock_queue                       │    │
│     │  ├── 全连接队列 (链表)                           │    │
│     │  │   rskq_accept_head → req1 → req2 → ...       │    │
│     │  └── 半连接队列 (哈希表)                         │    │
│     │       listen_opt → syn_table[0..n]              │    │
│     └─────────────────────────────────────────────────┘    │
│                                                             │
│  3. 设置队列长度                                            │
│     sk->sk_max_ack_backlog = min(backlog, somaxconn)       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 队列长度计算

| 队列 | 计算方式 |
|------|---------|
| **全连接队列** | `min(backlog, net.core.somaxconn)` |
| **半连接队列** | `roundup_pow_of_two(min(backlog, somaxconn, tcp_max_syn_backlog) + 1)` |

## 6.7.7 本章核心结论

### 关键结论

1. **listen的作用**
   - 创建并初始化半连接队列和全连接队列
   - 设置socket状态为TCP_LISTEN
   - 队列长度受 `somaxconn` 和 `tcp_max_syn_backlog` 限制

2. **connect的工作**
   - 从 `ip_local_port_range` 选择可用端口
   - 发送SYN并启动重传定时器
   - 一个端口可用于多条连接（四元组不同）

3. **三次握手本质**
   - 第一次握手：服务端将连接加入半连接队列
   - 第三次握手：服务端将连接从半连接队列移到全连接队列
   - accept：从全连接队列取出连接

4. **队列溢出影响**
   - 半连接队列满：SYN被丢弃，客户端超时重传
   - 全连接队列满：ACK被丢弃，服务端重发SYN+ACK

5. **内存消耗**
   - 一条ESTABLISHED状态空连接约消耗3.4KB内存
   - 实际瓶颈通常是内存，而非端口数量

### 面试高频问题

| 问题 | 答案要点 |
|------|---------|
| 为什么需要三次握手？ | 防止历史连接、同步序列号 |
| 半连接队列满怎么办？ | 开启syncookies或增大队列 |
| 全连接队列满怎么办？ | 增大队列或加快accept |
| 一个端口能建立多少连接？ | 理论上无限制（四元组不同即可） |
| TIME_WAIT过多怎么办？ | 开启tcp_tw_reuse |

## 6.2 深入理解listen

在服务端程序里，在开始接收请求之前都需要先执行listen系统调用。那么listen到底是干了啥？

### 6.2.1 listen系统调用

```c
// file: net/socket.c
SYSCALL_DEFINE2(listen, int, fd, int, backlog)
{
    struct socket *sock;
    int err, fput_needed;
    int somaxconn;

    // 根据fd查找socket内核对象
    sock = sockfd_lookup_light(fd, &err, &fput_needed);
    if (sock) {
        // 获取内核参数net.core.somaxconn
        somaxconn = sock_net(sock->sk)->core.sysctl_somaxconn;

        if ((unsigned int)backlog > somaxconn)
            backlog = somaxconn;

        // 调用协议栈注册的listen函数
        err = sock->ops->listen(sock, backlog);
    }
}
```

用户态的socket文件描述符只是一个整数而已，内核是没有办法直接用的。所以该函数中第一行代码就是根据用户传入的文件描述符来查找对应的socket内核对象。

再接着获取了系统里的net.core.somaxconn内核参数的值，和用户传入的backlog比较后取一个最小值传入下一步。

**关键点**：虽然listen允许我们传入backlog，但是如果用户传入的值比net.core.somaxconn还大的话是不会起作用的。

### 6.2.2 协议栈listen

sock->ops->listen指针指向的是inet_listen函数：

```c
// file: net/ipv4/af_inet.c
int inet_listen(struct socket *sock, int backlog)
{
    struct sock *sk = sock->sk;
    unsigned char old_state = sk->sk_state;
    int err;

    // 还不是listen状态（尚未listen过）
    if (old_state != TCP_LISTEN) {
        // 开始监听
        err = inet_csk_listen_start(sk, backlog);
    }

    // 设置全连接队列长度
    sk->sk_max_ack_backlog = backlog;

    return 0;
}
```

**关键点**：`sk->sk_max_ack_backlog` 是全连接队列的最大长度。服务端的全连接队列长度是执行listen函数时传入的backlog和net.core.somaxconn之间较小的那个值。

### 6.2.4 接收队列申请和初始化

```c
// file: net/core/request_sock.c
int reqsk_queue_alloc(struct request_sock_queue *queue,
                      unsigned int nr_table_entries)
{
    size_t lopt_size = sizeof(struct listen_sock);
    struct listen_sock *lopt;

    // 计算半连接队列的长度
    nr_table_entries = min_t(u32, nr_table_entries, sysctl_max_syn_backlog);
    nr_table_entries = max_t(u32, nr_table_entries, 8);
    nr_table_entries = roundup_pow_of_two(nr_table_entries + 1);

    // 为listen_sock对象申请内存（包含半连接队列）
    lopt_size += nr_table_entries * sizeof(struct request_sock *);
    if (lopt_size > PAGE_SIZE)
        lopt = vzalloc(lopt_size);
    else
        lopt = kzalloc(lopt_size, GFP_KERNEL);

    // 全连接队列头初始化
    queue->rskq_accept_head = NULL;

    // 半连接队列设置
    lopt->nr_table_entries = nr_table_entries;
    queue->listen_opt = lopt;

    return 0;
}
```

### 6.2.5 半连接队列长度计算

**计算步骤**：

```
1. min(backlog, somaxconn)
2. min(结果, tcp_max_syn_backlog)
3. max(结果, 8)
4. roundup_pow_of_two(结果 + 1)  // 上对齐到2的整数次幂
```

**实例计算**：

假设：somaxconn = 128, tcp_max_syn_backlog = 8192

| backlog | 计算过程 | 结果 |
|---------|---------|------|
| 5 | min(5,128)→min(5,8192)→max(5,8)→roundup(8+1) | 16 |
| 512 | min(512,128)→min(128,8192)→max(128,8)→roundup(128+1) | 256 |

**结论**：半连接队列长度是 `min(backlog, somaxconn, tcp_max_syn_backlog) + 1` 再上取整到2的N次幂，但最小不能小于16。

```c
    // 启动重传定时器
    inet_csk_reset_xmit_timer(sk, ICSK_TIME_RETRANS,
                               inet_csk(sk)->icsk_rto, TCP_RTO_MAX);
}
```

## 6.4 完整TCP连接建立过程

### 三次握手流程

```
客户端                                    服务端
  │                                         │
  │ ──────────── SYN ────────────────────► │ 第一次握手
  │           seq=x                         │
  │                                         │ 状态: SYN_RECEIVED
  │                                         │ 添加到半连接队列
  │ ◄─────────── SYN+ACK ───────────────── │ 第二次握手
  │           seq=y, ack=x+1                │
  │                                         │
  │ ──────────── ACK ────────────────────► │ 第三次握手
  │           ack=y+1                       │
  │                                         │ 从半连接队列移除
  │                                         │ 添加到全连接队列
  │                                         │ 状态: ESTABLISHED
  │                                         │
  │                                         │ accept() 从全连接队列取出
  │                                         │ 创建新socket
```

## 6.5 异常TCP连接建立情况

### 6.5.1 connect系统调用耗时失控

**原因**：端口耗尽时，端口选择算法CPU消耗大增

### 6.5.2 第一次握手丢包

- 半连接队列满 → SYN被丢弃
- 客户端等待超时后重传
- 重传时间：1s → 3s → 7s → 15s → 31s → 63s

### 6.5.3 第三次握手丢包

- 全连接队列满 → ACK被丢弃
- 服务端重发SYNACK
- 客户端以为连接成功，发送的数据被丢弃

## 6.7 本章总结

### 连接队列参数

| 参数 | 作用 |
|------|------|
| `net.core.somaxconn` | 全连接队列最大长度 |
| `net.ipv4.tcp_max_syn_backlog` | 半连接队列最大长度 |
| `backlog` | listen()参数 |

### 队列溢出影响

| 情况 | 结果 |
|------|------|
| 半连接队列满 | SYN被丢弃，客户端超时重传 |
| 全连接队列满 | ACK被丢弃，服务端重发SYNACK |

### 优化建议

```bash
## 增大队列
sysctl -w net.core.somaxconn=4096
sysctl -w net.ipv4.tcp_max_syn_backlog=4096

## 开启SYN cookies防御SYN Flood
sysctl -w net.ipv4.tcp_syncookies=1
```
