## 问题1：发送网络包时CPU消耗在哪里？

### 问题描述

在查看内核发送数据消耗的CPU时，应该看sy还是si？

内核在发送网络包的时候，需要CPU进行很多的处理工作。在top命令展示的结果里，和内核相关的项目有这么几个：sy、hi和si等。

### top命令CPU指标

| 指标 | 含义 | 说明 |
|------|------|------|
| **sy** | system | 内核态CPU时间 |
| **hi** | hardware interrupt | 硬中断CPU时间 |
| **si** | software interrupt | 软中断CPU时间 |

### 发送过程的CPU消耗

发送网络包主要消耗在：
1. **系统调用**：用户态→内核态切换，计入 sy
2. **协议处理**：TCP/IP协议栈处理，计入 sy
3. **内存拷贝**：用户数据到内核skb，计入 sy
4. **软中断**：发送完成后的清理工作，计入 si

**结论**：发送主要消耗在 **sy**（系统态），而接收主要消耗在 **si**（软中断）。

## 问题2：为什么NET_RX比NET_TX大得多？

### 问题描述

在服务器上查看/proc/softirqs，为什么NET_RX要比NET_TX大得多的多？

软中断类型有好几种，只拿网络IO相关的来说：
- NET_RX 是接收（R表示receive）
- NET_TX 是传输（T表示transmit）

对于一个既收取用户请求，又给用户返回数据的服务器来说，这两块的数字应该差不多才对，至少不会有数量级的差异。但事实上，你拿手头的任何一台服务器来看，NET_RX都要比NET_TX多得多。

### 实际示例

```bash
$ cat /proc/softirqs
            CPU0       CPU1       CPU2       CPU3
HI:         384508    285972     244566     258230
TIMER:   1670794507  218940516 3765758957 3937988197
NET_TX:     ...
NET_RX:  1591545176 1212716226 1017620906 1058380340
```

### 原因分析

**关键发现：发送完成触发的软中断是 NET_RX_SOFTIRQ，而不是 NET_TX_SOFTIRQ！**

```
发送完成流程：
1. 网卡发送完毕
2. 触发硬中断
3. 调用 napi_schedule()
4. 触发 NET_RX_SOFTIRQ（不是 NET_TX_SOFTIRQ！）
5. 执行发送完成的清理工作
```

这就是为什么 NET_RX 比 NET_TX 大得多的原因之一。

## 问题3：发送网络数据涉及哪些内存拷贝？

### 问题描述

发送网络数据的时候都涉及哪些内存拷贝操作？

你可能在一些博客里见过一种说法是用"零拷贝"的技术来提高性能。但是在理解"零拷贝"之前，首先应该搞清楚发送网络数据涉及哪些内存拷贝。

### 发送过程的内存拷贝

```
用户进程
    │
    │  1. 用户数据在用户态内存
    ▼
┌─────────────────────────────────────┐
│  系统调用                            │
│  - 拷贝1：用户态 → 内核态skb          │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  协议处理                            │
│  - 可能的拷贝：skb分片/合并           │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  RingBuffer                         │
│  - 映射DMA（不是拷贝）               │
└─────────────────────────────────────┘
    │
    ▼
   网卡发送
```

### 主要拷贝操作

| 拷贝 | 位置 | 说明 |
|------|------|------|
| 用户态→内核态 | tcp_sendmsg() | 将用户数据拷贝到skb |
| skb内部 | 协议处理 | 可能的分片、合并操作 |

## 问题4：零拷贝到底是怎么回事？

### 问题描述

很多性能优化方案里都会提到零拷贝。但是零拷贝到底是怎么回事，是真的没有数据的内存拷贝了吗？究竟避免了哪步到哪步的拷贝操作？

### 传统方式的拷贝

```
传统文件发送（read + write）：

磁盘 ──────→ 内核PageCache ──────→ 用户缓冲区 ──────→ 内核skb ──────→ 网卡
        DMA拷贝         CPU拷贝        CPU拷贝       DMA拷贝

总共4次拷贝，其中2次CPU拷贝
```

### 零拷贝方式

```
零拷贝文件发送（sendfile）：

磁盘 ──────→ 内核PageCache ──────→ 内核skb ──────→ 网卡
        DMA拷贝                    DMA拷贝

总共2次拷贝，0次CPU拷贝
```

### 零拷贝技术

| 技术 | 减少的拷贝 | 使用场景 |
|------|-----------|----------|
| **sendfile** | 避免 用户↔内核 拷贝 | 文件传输 |
| **mmap** | 避免 read 时的拷贝 | 文件映射 |
| **splice** | 零拷贝管道传输 | 管道数据传输 |

## 问题5：为什么Kafka的网络性能很突出？

### 问题描述

大家一定对Kafka出类拔萃的性能有所耳闻。那么它性能优异的秘诀究竟在哪儿？

### Kafka性能优化技术

1. **零拷贝**
   - 使用 sendfile 系统调用
   - 直接从PageCache传输到网卡
   - 避免用户态↔内核态的内存拷贝

2. **顺序写入**
   - 磁盘顺序写入比随机写入快很多
   - 充分利用磁盘带宽

3. **批量处理**
   - 批量发送消息
   - 减少系统调用次数
   - 提高网络吞吐

4. **PageCache**
   - 利用操作系统的PageCache
   - 热数据自动缓存

### 性能对比

```
传统方式（用户态中转）：
磁盘 → PageCache → 用户态 → 内核skb → 网卡
       DMA拷贝    CPU拷贝  CPU拷贝   DMA拷贝

Kafka方式（零拷贝）：
磁盘 → PageCache → 内核skb → 网卡
       DMA拷贝              DMA拷贝
```

### 对性能优化的启发

1. 理解数据在各层之间的拷贝过程
2. 识别可避免的拷贝操作
3. 选择合适的零拷贝技术
4. 批量处理减少系统调用开销

## 发送完成后的处理

### 中断通知流程

虽然这时数据已经发送完毕，但其实还有一件重要的事情没做，那就是释放缓存队列等内存。内核是如何知道什么时候才能释放内存的呢？

网卡在发送完毕的时候，会给CPU发送一个硬中断来通知CPU。

```
硬件中断
    │
    │  // file: drivers/net/ethernet/intel/igb/igb_main.c
    │  static irqreturn_t igb_msix_ring(int irq, void *data)
    │  {
    │      napi_schedule(&q_vector->napi);
    │  }
    │      │
    │      ▼
    │  // 触发 NET_RX_SOFTIRQ（注意：不是 NET_TX_SOFTIRQ！）
    │  static inline void __napi_schedule(...)
    │  {
    │      list_add_tail(&napi->poll_list, &sd->poll_list);
    │      __raise_softirq_irqoff(NET_RX_SOFTIRQ);
    │  }
    │      │
    │      ▼
    │  // file: net/core/dev.c
    │  static void net_rx_action(...)
    │      │
    │      │  while (!list_empty(&sd->poll_list)) {
    │      │      work = n->poll(n, weight);
    │      │  }
    │      ▼
    │  // file: drivers/net/ethernet/intel/igb/igb_main.c
    │  static int igb_poll(...)
    │  {
    │      clean_complete = igb_clean_tx_irq(q_vector);
    │  }
    │      │
    │      ▼
    │  // file: drivers/net/ethernet/intel/igb/igb_main.c
    │  static bool igb_clean_tx_irq(...)
    │  {
    │      // 释放skb
    │      dev_kfree_skb_any(tx_buffer->skb);
    │      // 清理DMA区域
    │      dma_unmap_len_set(tx_buffer, len, 0);
    │  }
    ▼
清理完成
```

## 关键发现

### 发送完成触发的是 NET_RX_SOFTIRQ

**注意，这里的主题虽然是发送数据，但是硬中断最终触发的软中断却是 NET_RX_SOFTIRQ，而并不是 NET_TX_SOFTIRQ！！！**

这解释了开篇问题2的部分原因：
- 传输完成最终会触发 NET_RX，而不是 NET_TX
- 所以自然你观测 /proc/softirqs 也就能看到 NET_RX 更多

## 发送过程各层职责

| 层级 | 职责 | 关键函数 |
|------|------|----------|
| **系统调用层** | 查找socket，构造msghdr | `sendto` → `sock_sendmsg` |
| **传输层** | 申请skb，拷贝数据，封装TCP头 | `tcp_sendmsg` → `tcp_transmit_skb` |
| **网络层** | 路由选择，封装IP头，netfilter | `ip_queue_xmit` → `ip_local_out` |
| **邻居子系统** | ARP解析，封装MAC头 | `neigh_resolve_output` |
| **网络设备层** | Qdisc排队，选择发送队列 | `dev_queue_xmit` → `__dev_xmit_skb` |
| **驱动层** | DMA映射，写入RingBuffer | `igb_xmit_frame` |

## 多队列网卡

现在的服务器上的网卡一般都是支持多队列的。每一个队列都是由一个RingBuffer表示的，开启了多队列以后的网卡就会对应有多个RingBuffer。

```
│    │ 传输队列1 │  │ 传输队列2 │  │ 传输队列3 │  ...   │
│    │ RingBuf  │  │ RingBuf  │  │ RingBuf  │        │
│    ├──────────┤  ├──────────┤  ├──────────┤        │
│    │ 接收队列1 │  │ 接收队列2 │  │ 接收队列3 │  ...   │
```

### 多队列的优势

| 优势 | 说明 |
|------|------|
| **并行处理** | 多个队列可以并行收发 |

| **中断亲和性** | 每个队列的中断可绑定到特定CPU |

## RingBuffer 分配

网卡在启动时最重要的任务之一就是分配和初始化RingBuffer。

### 启动入口函数

在网卡启动的时候，会调用到 `_igb_open` 函数，RingBuffer就是在这里分配的：

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
static int _igb_open(struct net_device *netdev, bool resuming)
{
    struct igb_adapter *adapter = netdev_priv(netdev);

    // 分配传输描述符数组
    err = igb_setup_all_tx_resources(adapter);

    // 分配接收描述符数组
    err = igb_setup_all_rx_resources(adapter);

    // 开启全部队列
    netif_tx_start_all_queues(netdev);
}
```

### 分配所有传输RingBuffer

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
static int igb_setup_all_tx_resources(struct igb_adapter *adapter)
{
    int i, err = 0;

    // 有几个队列就构造几个RingBuffer
    for (i = 0; i < adapter->num_tx_queues; i++) {
        err = igb_setup_tx_resources(adapter->tx_ring[i]);
        if (err) {
            // 错误处理：释放已分配的资源
            while (i--)
                igb_free_tx_resources(adapter->tx_ring[i]);
            break;
        }
    }

    return err;
}
```

---

## 单个RingBuffer的构造

真正的RingBuffer构造过程是在 `igb_setup_tx_resources` 中完成的：

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
int igb_setup_tx_resources(struct igb_ring *tx_ring)
{
    struct device *dev = tx_ring->dev;
    int size;

    // 1. 申请 igb_tx_buffer 数组内存（内核使用的）
    size = sizeof(struct igb_tx_buffer) * tx_ring->count;
    tx_ring->tx_buffer_info = vzalloc(size);
    if (!tx_ring->tx_buffer_info)
        goto err;

    // 2. 申请 e1000_adv_tx_desc DMA数组内存（网卡硬件使用的）
    tx_ring->size = tx_ring->count * sizeof(union e1000_adv_tx_desc);
    tx_ring->size = ALIGN(tx_ring->size, 4096);

    tx_ring->desc = dma_alloc_coherent(dev, tx_ring->size,
                                        &tx_ring->dma, GFP_KERNEL);
    if (!tx_ring->desc)
        goto err;

    // 3. 初始化队列成员
    tx_ring->next_to_use = 0;
    tx_ring->next_to_clean = 0;

    return 0;

err:
    vzfree(tx_ring->tx_buffer_info);
    tx_ring->tx_buffer_info = NULL;
    return -ENOMEM;
}
```

### 两个数组的作用

从上述源码可以看到，一个传输RingBuffer的内部包含两个环形数组：

| 数组 | 用途 | 分配方式 |
|------|------|----------|
| **igb_tx_buffer[]** | 内核使用的指针数组 | vzalloc（普通内存） |
| **e1000_adv_tx_desc[]** | 网卡硬件使用的描述符数组 | dma_alloc_coherent（DMA内存） |

---

## 发送队列细节

### 两个数组的关系

这个时候它们之间还没有什么联系。将来在发送的时候，这两个环形数组中相同位置的指针都将指向同一个skb：

```
┌──────────────────────────────────────────────────────────┐
│                    传输队列 RingBuffer                     │
├──────────────────────────────────────────────────────────┤
│                                                          │
```

## 4.4.1 send系统调用实现

### send系统调用入口

send系统调用的源码位于文件 net/socket.c 中。在这个系统调用里，内部其实真正使用的是sendto系统调用。

```c
// file: net/socket.c
SYSCALL_DEFINE4(send, int, fd, void __user *, buff, size_t, len,
                unsigned int, flags)
{
    return sys_sendto(fd, buff, len, flags, NULL, 0);
}
```

### sendto系统调用

整个调用链条虽然不短，但其实主要只干了两件简单的事情：

1. 根据用户传入的socket句柄号找到真正的socket内核对象
2. 构造一个struct msghdr对象，把用户传入的数据包装进去

```c
// file: net/socket.c
SYSCALL_DEFINE6(sendto, int, fd, void __user *, buff, size_t, len,
                unsigned int, flags, struct sockaddr __user *, addr,
                int, addr_len)
{
    struct socket *sock;
    struct msghdr msg;
    struct iovec iov;
    int err, fput_needed;

    // 1. 根据fd找到socket内核对象
    sock = sockfd_lookup_light(fd, &err, &fput_needed);
    if (!sock)
        goto out;

    // 2. 构造msghdr对象
    iov.iov_base = buff;
    iov.iov_len = len;
    msg.msg_name = NULL;
    msg.msg_iov = &iov;
    msg.msg_iovlen = 1;
    msg.msg_control = NULL;
    msg.msg_controllen = 0;
    msg.msg_namelen = 0;
    msg.msg_flags = flags;

    // 3. 发送数据
    err = sock_sendmsg(sock, &msg, len);

    fput_light(sock->file, fput_needed);
out:
    return err;
}
```

### msghdr结构体


### tcp_sendmsg函数

```c
// file: net/ipv4/tcp.c
int tcp_sendmsg(struct kiocb *iocb, struct sock *sk,
                struct msghdr *msg, size_t size)
{
    struct iovec *iov;
    struct tcp_sock *tp = tcp_sk(sk);
    int flags, err, copied = 0;
    long timeo;

    // 获取用户传递过来的数据和标志
    iov = msg->msg_iov;        // 用户数据地址
    iovlen = msg->msg_iovlen;  // 数据块数为1
    flags = msg->msg_flags;    // 各种标志

    // 获取超时时间
    timeo = sock_sndtimeo(sk, flags & MSG_DONTWAIT);

    // 锁定socket
    lock_sock(sk);

    // 遍历用户层的数据块
    while (--iovlen >= 0) {
        unsigned char __user *from = iov->iov_base;
        int seglen = iov->iov_len;

        while (seglen > 0) {
            int copy;

            // 获取发送队列中最后一个skb
            skb = tcp_write_queue_tail(sk);

            // 判断是否需要申请新的skb
            if (copy <= 0) {
new_segment:
                // 申请skb，并添加到发送队列的尾部
                skb = sk_stream_alloc_skb(sk, select_size(sk, sg),
                                          sk->sk_allocation);
                if (!skb)
                    goto wait_for_sndbuf;

                // 把skb挂到socket的发送队列上
                skb_entail(sk, skb);
            }

            // skb中有足够的空间
            if (skb_availroom(skb) > 0) {
                // 将用户空间的数据拷贝到内核空间
                copy = min_t(int, copy, skb_availroom(skb));
                err = skb_add_data_nocache(sk, skb, from, copy);
                if (err)
                    goto do_error;

                from += copy;
                copied += copy;
                seglen -= copy;
            }
        }
    }

    // 发送数据
    tcp_push(sk, flags);

out:
    release_sock(sk);
    return copied;
}
```

### igb_xmit_frame函数

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
static int igb_xmit_frame(struct sk_buff *skb, struct net_device *netdev)
{
    struct igb_adapter *adapter = netdev_priv(netdev);
    struct igb_ring *tx_ring;
    struct igb_tx_buffer *first;
    union e1000_adv_tx_desc *tx_desc;

    // 选择发送队列

    // 获取TX queue中下一个可用缓冲区
    first = &tx_ring->tx_buffer_info[tx_ring->next_to_use];
    tx_desc = E1000_TX_DESC(*tx_ring, tx_ring->next_to_use);

    // 设置DMA地址

    tx_desc->read.buffer_addr = cpu_to_le64(dma);

    // 设置描述符标志
    tx_desc->read.cmd_type_len = cpu_to_le32(cmd_type);

    // 通知网卡发送
    writel(tx_ring->next_to_use, tx_ring->tail);

    return NETDEV_TX_OK;
```

### 网卡发送完成

### napi_schedule

```c
// file: net/core/dev.c
static inline void __napi_schedule(struct napi_struct *n)
{
    unsigned long flags;

    local_irq_save(flags);
    ____napi_schedule(&__get_cpu_var(softnet_data), n);
    local_irq_restore(flags);
}

static inline void ____napi_schedule(struct softnet_data *sd,
                                      struct napi_struct *napi)
{
    // 将napi添加到poll_list
    list_add_tail(&napi->poll_list, &sd->poll_list);

    // 触发 NET_RX_SOFTIRQ（注意：不是 NET_TX_SOFTIRQ！）
    __raise_softirq_irqoff(NET_RX_SOFTIRQ);
}
```

### 软中断处理

```c
// file: net/core/dev.c
static void net_rx_action(struct softirq_action *h)
{
    struct softnet_data *sd = &__get_cpu_var(softnet_data);
    unsigned long time_limit = jiffies + 2;
    int budget = netdev_budget;
    void *have;

    local_irq_disable();

    while (!list_empty(&sd->poll_list)) {
        struct napi_struct *n;
        int work, weight;

        n = list_first_entry(&sd->poll_list, struct napi_struct, poll_list);

        weight = n->weight;
        work = 0;

        if (test_bit(NAPI_STATE_SCHED, &n->state)) {
            work = n->poll(n, weight);
        }

        budget -= work;
    }
}
```

## 驱动poll函数

### igb_poll

对于igb网卡，poll函数是 `igb_poll`：

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
static int igb_poll(struct napi_struct *napi, int budget)
{
    struct igb_q_vector *q_vector = container_of(napi,
                                                  struct igb_q_vector, napi);
    bool clean_complete = true;

    // 清理发送完成的数据
    if (q_vector->tx.ring)
        clean_complete = igb_clean_tx_irq(q_vector);

    // 清理接收数据
    if (q_vector->rx.ring)
        clean_complete &= igb_clean_rx_irq(q_vector, budget);

    if (!clean_complete)
        return budget;

    // 所有工作完成，退出NAPI
    napi_complete(napi);
    igb_ring_irq_enable(q_vector);

    return 0;
}
```

## 发送完成清理

### igb_clean_tx_irq

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
static bool igb_clean_tx_irq(struct igb_q_vector *q_vector)
{
    struct igb_adapter *adapter = q_vector->adapter;
    struct igb_ring *tx_ring = q_vector->tx.ring;
    struct igb_tx_buffer *tx_buffer;
    union e1000_adv_tx_desc *tx_desc;
    unsigned int total_bytes = 0, total_packets = 0;
    unsigned int i = tx_ring->next_to_clean;

    tx_desc = E1000_TX_DESC(*tx_ring, i);
    tx_buffer = &tx_ring->tx_buffer_info[i];

    while ((tx_desc->wb.status & cpu_to_le32(E1000_TXD_STAT_DD))) {
        // 获取已发送的skb
        struct sk_buff *skb = tx_buffer->skb;

        // 统计
        total_bytes += tx_buffer->bytecount;
        total_packets += tx_buffer->gso_segs;

        // 释放skb
        dev_kfree_skb_any(tx_buffer->skb);
        tx_buffer->skb = NULL;

        // 清理DMA映射
        dma_unmap_len_set(tx_buffer, len, 0);

        // 移动到下一个
        i++;
        if (i == tx_ring->count)
            i = 0;

        tx_desc = E1000_TX_DESC(*tx_ring, i);
        tx_buffer = &tx_ring->tx_buffer_info[i];
    }

    // 更新next_to_clean
    tx_ring->next_to_clean = i;

    // 更新统计信息
    u64_stats_update_begin(&tx_ring->tx_syncp);
    tx_ring->tx_bytes += total_bytes;
    tx_ring->tx_packets += total_packets;
    u64_stats_update_end(&tx_ring->tx_syncp);

    // 检查是否有待发送的数据
    if (unlikely(total_packets && netif_carrier_ok(tx_ring->netdev) &&
                 igb_desc_unused(tx_ring) >= TX_WAKE_THRESHOLD)) {
        // 唤醒发送队列
        netif_wake_subqueue(tx_ring->netdev, tx_ring->queue_index);
    }

    return !!total_packets;
}
```

## 为什么是NET_RX_SOFTIRQ

### 原因分析

发送完成触发的软中断是 NET_RX_SOFTIRQ，而不是 NET_TX_SOFTIRQ：

1. **历史原因**：Linux网络子系统设计时，将所有网络相关的软中断处理统一放在一起
2. **代码复用**：NAPI机制同时用于接收和发送完成的处理
3. **性能考虑**：减少软中断类型，统一处理可以提高效率

### 这解释了什么

这解释了为什么在 `/proc/softirqs` 中，NET_RX 总是比 NET_TX 大得多：

```bash
$ cat /proc/softirqs
            CPU0       CPU1       CPU2       CPU3
NET_TX:     12345      23456      34567      45678
NET_RX:  1591545176 1212716226 1017620906 1058380340
```

## 小结

### 发送完成处理流程

1. **网卡发送完成**：触发硬中断
2. **硬中断处理**：调用 `napi_schedule()`
3. **软中断处理**：`NET_RX_SOFTIRQ` → `net_rx_action()`
4. **poll函数**：`igb_poll()` → `igb_clean_tx_irq()`
5. **清理工作**：
   - 释放skb
   - 解除DMA映射
   - 更新统计信息
   - 唤醒发送队列

### 关键点

| 要点 | 说明 |
|------|------|
| 软中断类型 | NET_RX_SOFTIRQ（不是 NET_TX_SOFTIRQ） |
| 清理内容 | skb、DMA映射、描述符 |
| 唤醒时机 | RingBuffer有足够空闲空间时 |

### 问题1：发送消耗的CPU在哪？

**答案**：主要在 **sy（系统态）**，因为大部分工作在系统调用和内核中进行。

**答案**：
- sendfile：避免用户态↔内核态的拷贝
- 直接从PageCache到网卡
- 减少CPU拷贝次数

**答案**：
1. 零拷贝（sendfile）

4. 利用PageCache

```c
// file: drivers/net/ethernet/intel/igb/igb_main.c
int igb_setup_tx_resources(struct igb_ring *tx_ring)
{
    // 1. 申请 igb_tx_buffer 数组内存（内核使用）
    size = sizeof(struct igb_tx_buffer) * tx_ring->count;
    tx_ring->tx_buffer_info = vzalloc(size);

    // 2. 申请 e1000_adv_tx_desc DMA数组内存（网卡硬件使用）
    tx_ring->size = tx_ring->count * sizeof(union e1000_adv_tx_desc);
    tx_ring->size = ALIGN(tx_ring->size, 4096);

    tx_ring->desc = dma_alloc_coherent(dev, tx_ring->size,
                                        &tx_ring->dma, GFP_KERNEL);

    // 3. 初始化队列成员
    tx_ring->next_to_use = 0;
    tx_ring->next_to_clean = 0;
}
```

## 4.4 数据从用户进程到网卡的详细过程

### 4.4.1 send系统调用实现

send系统调用位于 `net/socket.c`：

```c
// file: net/socket.c
SYSCALL_DEFINE4(send, int, fd, void __user *, buff, size_t, len,
                unsigned int, flags)
{
    return sys_sendto(fd, buff, len, flags, NULL, 0);
}
```

**send系统调用主要做两件事**：
1. 根据fd找到socket内核对象
2. 构造struct msghdr对象，把用户数据包装进去

```c
// file: net/socket.c
SYSCALL_DEFINE6(sendto, int, fd, void __user *, buff, size_t, len,
                unsigned int, flags, struct sockaddr __user *, addr,
                int, addr_len)
{
    // 1. 根据fd找到socket
    sock = sockfd_lookup_light(fd, &err, &fput_needed);

    // 2. 构造msghdr
    struct msghdr msg;
    struct iovec iov;

    iov.iov_base = buff;
    iov.iov_len = len;
    msg.msg_iovlen = 1;
    msg.msg_iov = &iov;
    msg.msg_flags = flags;

    // 3. 发送数据
    sock_sendmsg(sock, &msg, len);
}
```

### 调用链

```
send() → sendto() → sock_sendmsg() → sock_sendmsg_nosec()
                                         ↓
                                   sock->ops->sendmsg
                                         ↓
                                   inet_sendmsg()
```

### 4.4.2 传输层处理

进入协议栈后，会调用具体协议的发送函数：

```c
// file: net/ipv4/af_inet.c
int inet_sendmsg(struct kiocb *iocb, struct socket *sock,
                 struct msghdr *msg, size_t size)
{
```

对于TCP，调用的是 `tcp_sendmsg`：

```c
// file: net/ipv4/tcp.c
int tcp_sendmsg(struct kiocb *iocb, struct sock *sk,
                struct msghdr *msg, size_t size)
{
    // 获取发送队列
    skb = tcp_write_queue_tail(sk);

    // 遍历用户层的数据块
    while (--iovlen >= 0) {
        unsigned char __user *from = iov->iov_base;

        while (seglen > 0) {
            // 需要申请新的skb
            if (copy <= 0) {
                // 申请skb，并添加到发送队列的尾部
                skb = sk_stream_alloc_skb(sk, select_size(sk, sg),
                                          sk->sk_allocation);

                // 把skb挂到socket的发送队列上
                skb_entail(sk, skb);
            }

            // 将用户空间的数据拷贝到内核空间
            skb_add_data_nocache(sk, skb, from, copy);
        }
    }
}
```

### TCP头封装

```c
// file: net/ipv4/tcp_output.c
static int tcp_transmit_skb(struct sock *sk, struct sk_buff *skb, ...)
{
    // 封装TCP头
    th = tcp_hdr(skb);
    th->source = inet->inet_sport;
```

### 4.4.4 邻居子系统

邻居子系统负责MAC地址解析：

```c
// file: include/net/neighbour.h
int neigh_resolve_output(struct neighbour *neigh, struct sk_buff *skb)
{
    // 1. 检查邻居状态
    if (neigh->nud_state & NUD_VALID) {
        // 邻居可达，直接封装MAC头
        dev_hard_header(skb, dev, ntohs(skb->protocol),
                        neigh->ha, NULL, skb->len);

        // 邻居不可达，发送ARP请求
        neigh_probe(neigh);
    }
}
```

### 发送 vs 接收对比

| 方面 | 发送 | 接收 |
|------|------|------|
| 软中断触发 | 配额用尽时 | 每个包都会触发 |
| CPU开销 | 主要在sy | 主要在si |
| 内存拷贝 | 用户态→内核态 | 内核态→用户态 |
| 完成中断 | NET_RX_SOFTIRQ | NET_RX_SOFTIRQ |
