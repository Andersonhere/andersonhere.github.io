> **核心问题**：如何按比例分配 CPU 时间？

## 本章大纲

### 9.1 基本概念：彩票数表示份额

**彩票调度 (Lottery Scheduling)**：
- 每个进程拥有一定数量的"彩票"
- 定期抽取彩票，中奖者运行
- 彩票数量与 CPU 份额成正比

```
进程 A: 75 张彩票 (75% CPU)
进程 B: 25 张彩票 (25% CPU)

随机抽取 0-99:
  0-74   → 进程 A 运行
  75-99  → 进程 B 运行
```

### 9.2 彩票机制

| 机制 | 说明 |
|------|------|
| 彩票货币 | 用户可以用自己的货币分配彩票，系统自动转换 |
| 彩票转让 | 进程可以临时转让彩票给其他进程 |
| 彩票通胀 | 进程可以增加自己的彩票数（需要信任） |

### 9.3 实现

```c
// 彩票调度伪代码
int total = 0;
for (p in processes) {
    total += p->tickets;
}
int winner = random() % total;
int current = 0;
for (p in processes) {
    current += p->tickets;
    if (current > winner) {
        return p;  // 中奖进程
    }
}
```

**时间复杂度**：O(n)，可通过红黑树优化到 O(log n)

### 9.4 一个例子

```
进程 A: 100 张彩票
进程 B: 50 张彩票
进程 C: 250 张彩票

总彩票数: 400
份额: A=25%, B=12.5%, C=62.5%
```

### 9.5 如何分配彩票

**策略**：
- 用户指定
- 根据任务重要性
- 根据付费金额（云计算场景）

### 9.6 为什么不是确定的？

**随机调度的优势**：
- 简单实现
- 无需全局状态同步
- 统计上公平

**确定性替代方案**：
- 步长调度 (Stride Scheduling)
- 每个进程有固定步长
- 跟踪每个进程的"通过值"

```
进程 A: 步长 100, 通过值 0
进程 B: 步长 200, 通过值 0
进程 C: 步长 40,  通过值 0

运行顺序: C, C, A, C, C, A, B, C...
```

### 9.7 小结

| 方案 | 优点 | 缺点 |
|------|------|------|
| 彩票调度 | 简单，无需全局状态 | 不够精确 |
| 步长调度 | 精确控制比例 | 需要全局状态 |

## 思考题

1. 彩票调度如何保证长期公平？
2. 为什么步长调度需要全局状态？
