## 9.1 网络请求优化

## 概述

具备了对内核网络模块的深刻理解之后，在性能方面有哪些优化手段可用呢？本章给出一些开发或者运维中的性能优化建议。

---

## 建议1：尽量减少不必要的网络IO

### 网络IO开销分析

网络很好，能降低模块的开发难度，也能用它搭建出更强大的系统。但是这不是滥用它的理由！

```
┌─────────────────────────────────────────────────────────────┐
│                    网络IO开销分析                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  本机网络IO开销：                                           │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 1. 用户态 → 内核态（系统调用开销）                    │   │
│  │ 2. 经过协议栈（花费CPU周期）                         │   │
│  │ 3. 进入回环设备的"驱动程序"                          │   │
│  │ 4. 触发软中断                                        │   │
│  │ 5. 接收协议栈处理                                    │   │
│  │ 6. 唤醒用户进程                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  跨机网络IO额外开销：                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ - 双方网卡的DMA拷贝                                  │   │
│  │ - 网络RTT耗时延迟                                    │   │
│  │ - 网络带宽竞争                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 优化案例

**问题场景**：服务进程需要查询Redis获取配置信息

**错误做法**：每次请求都通过网络调用Redis
```php
<?php
// 每次请求都调用Redis
$config = $redis->get("config_key");
```

**正确做法**：直接在服务进程内实现功能
```php
<?php
// 启动时加载配置，进程内缓存
$config = loadConfig();

// 或者直接在代码中定义
const CONFIG_VALUE = "...";
```

### 减少网络调用的好处

| 方面 | 好处 |
|------|------|
| 延迟 | 消除网络RTT |
| CPU | 减少协议栈处理 |
| 稳定性 | 减少网络故障影响 |
| 部署 | 降低部署复杂度 |

## 建议2：尽量合并网络请求

### 问题示例

```php
<?php
// 错误：循环中发起多次网络请求
foreach ($package_list as $package_name) {
    $info = $redis->get($package_name);  // 每次都发起网络请求
    process($info);
}

// 假设用户安装了60个APP
// 总耗时 = 60 × RTT = 60 × 1ms = 60ms（本地Redis）
// 如果是跨机房，RTT=30ms，总耗时 = 1800ms！
```

### 优化方法

```php
<?php
// 正确：使用批量获取命令
$infos = $redis->mget($package_list);  // 一次网络请求
foreach ($infos as $info) {
    process($info);
}

// 使用Pipeline进一步优化
$pipe = $redis->pipeline();
foreach ($package_list as $package_name) {
    $pipe->get($package_name);
}
$results = $pipe->execute();  // 一次网络往返
```

### 请求合并效果

```
┌─────────────────────────────────────────────────────────────┐
│                    请求合并效果对比                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  优化前（60次请求）：                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ RTT: 1ms × 60 = 60ms                                │   │
│  │ 跨机房 RTT: 30ms × 60 = 1800ms                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  优化后（1次批量请求）：                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ RTT: 1ms × 1 = 1ms                                  │   │
│  │ 跨机房 RTT: 30ms × 1 = 30ms                         │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  提升：60倍 / 60倍                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 建议3：调用者与被调用机器尽可能部署得近一些

### RTT延迟对比

| 场景 | RTT延迟 |
|------|---------|
| 机房内部 | ~0.几毫秒 |
| 同地区不同机房 | ~1毫秒 |
| 北京到广州 | ~30-40毫秒 |
| 中国到美国 | ~200毫秒 |

## 建议4：内网调用不要用外网域名

### 为什么不要用外网域名

```
┌─────────────────────────────────────────────────────────────┐
│                    外网域名的问题                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题1：外网接口慢                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 内网可能过个交换机就到达                              │   │
│  │ 走外网会绕一圈，增加延迟                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  问题2：带宽成本高                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 出入带宽都要计费                                      │   │
│  │ 内网带宽通常不计费                                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  问题3：NAT单点瓶颈                                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ NAT服务器容易成为瓶颈                                │   │
│  │ 影响整体性能和稳定性                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 正确做法

```bash
## 错误：使用外网域名
api.example.com → 公网IP → NAT → 服务

## 正确：使用内网IP或内网域名
api.internal.example.com → 内网IP → 服务
```

### 配置示例

```nginx
## /etc/hosts 或 DNS 配置
## 内网服务使用内网IP
10.0.0.100 api.internal.example.com
```

## 9.1.5 小结

### 网络请求优化清单

| 优化项 | 方法 | 效果 |
|--------|------|------|
| 减少网络IO | 本地处理代替远程调用 | 消除RTT |
| 合并请求 | 批量操作、Pipeline | 减少网络往返次数 |
| 就近部署 | 机房内闭环 | 减少RTT延迟 |
| 内网调用 | 使用内网IP/域名 | 减少延迟和成本 |

### 优化原则

1. **能不走网络就不走网络**
2. **必须走网络就尽量少走**
3. **走网络就尽量走近路**
4. **走内网不走外网**

## 建议1：调整网卡RingBuffer大小

### RingBuffer作用

```
┌─────────────────────────────────────────────────────────────┐
│                    RingBuffer作用                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  网卡                                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │   RingBuffer                                        │   │
│  │   ┌───┬───┬───┬───┬───┬───┬───┬───┐               │   │
│  │   │ 1 │ 2 │ 3 │ 4 │ 5 │...│ n │   │  ← 接收队列   │   │
│  │   └───┴───┴───┴───┴───┴───┴───┴───┘               │   │
│  │                                                     │   │
│  │   网卡收到数据包 → DMA写入RingBuffer               │   │

│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │

└─────────────────────────────────────────────────────────────┘
```

## 建议2：多队列网卡RSS调优

### RSS原理

RSS（Receive Side Scaling）是网卡的硬件特性，可以将接收到的数据包分发到多个接收队列，由不同的CPU核心处理。

```
┌─────────────────────────────────────────────────────────────┐
│                    RSS工作原理                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  网卡                                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                      RSS                            │   │
│  │   数据包 ──► Hash计算 ──► 分发到不同队列            │   │
│  │                                                     │   │
│  │   ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐                 │   │
│  │   │Queue│ │Queue│ │Queue│ │Queue│                 │   │
│  │   │  0  │ │  1  │ │  2  │ │  3  │                 │   │
│  │   └──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘                 │   │
│  └──────┼───────┼───────┼───────┼─────────────────────┘   │
│         │       │       │       │                          │
│         ▼       ▼       ▼       ▼                          │
│       CPU0    CPU1    CPU2    CPU3                         │
│                                                             │
│  多队列并行处理，提高吞吐量                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 查看网卡队列配置

```bash
## 查看队列数
ethtool -l eth0

## 输出示例：
Channel parameters for eth0:
Pre-set maximums:
RX:              0
TX:              0
Other:           0
Combined:        63    # 最大合并队列数
Current hardware settings:
RX:              0
TX:              0
Other:           0
Combined:        8     # 当前合并队列数
```

### 查看中断分布

```bash
## 查看中断情况
cat /proc/interrupts | grep eth0

## 输出示例：
27:  1109986815          0          0          0  PCI-MSI-edge  virtio1-input.0
28:          0 1109986815          0          0  PCI-MSI-edge  virtio1-input.1

## 查看中断亲和性
cat /proc/irq/27/smp_affinity
## 输出：8
## 8是二进制1000，第4位为1，表示由CPU3处理
```

### 设置队列数

```bash
## 设置队列数（需要网卡支持）
ethtool -L eth0 combined 16
```

### 设置中断亲和性

```bash
## 将不同队列的中断绑定到不同CPU
echo 1 > /proc/irq/27/smp_affinity   # CPU0
echo 2 > /proc/irq/28/smp_affinity   # CPU1
echo 4 > /proc/irq/29/smp_affinity   # CPU2
echo 8 > /proc/irq/30/smp_affinity   # CPU3

## 或者使用 irqbalance 服务自动分配
systemctl start irqbalance
```

## 建议3：RPS/RFS优化

### RPS原理

RPS（Receive Packet Steering）是软件层面的负载均衡，将数据包分散到多个CPU处理。

```
┌─────────────────────────────────────────────────────────────┐
│                    RPS工作原理                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  单队列网卡：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │   网卡只有一个队列                                  │   │
│  │   所有数据包都由一个CPU处理                         │   │
│  │   该CPU成为瓶颈                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  开启RPS后：                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │   软件计算Hash，分发到不同CPU                        │   │
│  │                                                     │   │
│  │   ┌──────┐                                          │   │
│  │   │Queue │ ──► Hash ──┬──► CPU0                    │   │
│  │   └──────┘            ├──► CPU1                    │   │
│  │                       ├──► CPU2                    │   │
│  │                       └──► CPU3                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### RFS原理

RFS（Receive Flow Steering）基于流的负载均衡，同一连接的数据包分配到同一CPU。

```
好处：同一连接的数据包在同一CPU处理，提高缓存命中率
```

### 配置方法

```bash
## 查看RPS配置
cat /sys/class/net/eth0/queues/rx-0/rps_cpus

## 设置RPS（将数据包分散到CPU0-3）
echo f > /sys/class/net/eth0/queues/rx-0/rps_cpus

## 设置RFS
cat /sys/class/net/eth0/queues/rx-0/rps_flow_cnt
echo 4096 > /sys/class/net/eth0/queues/rx-0/rps_flow_cnt

## 启用RFS
sysctl -w net.core.rps_sock_flow_entries=4096
```

## 9.3 发送过程优化

## 概述

## 建议1：避免分片

### MTU限制

以太网默认MTU：1500字节

超过MTU的数据包需要分片：
- 增加CPU开销
- 增加网络开销
- 任何一个分片丢失，整个数据包需要重传

```
┌─────────────────────────────────────────────────────────────┐
│                    IP分片问题                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  发送3000字节的数据：                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 原始数据：[=========== 3000字节 ===========]         │   │
│  │                                                     │   │
│  │ 分片后：                                            │   │
│  │ [==== 1500字节 ====] 分片1                          │   │
│  │ [==== 1500字节 ====] 分片2                          │   │
│  │                                                     │   │
│  │ 问题：                                              │   │
│  │ - 额外的IP头开销                                    │   │
│  │ - 分片2丢失会导致整个包重传                         │   │
│  │ - 增加路由器处理负担                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 避免分片的方法

```bash
## 1. 控制数据包大小 < MTU
## 应用层控制每次发送的数据大小

## 2. 使用PMTU发现机制
sysctl -w net.ipv4.tcp_mtu_probing=1

## 3. 调整MTU（如果网络设备支持）
ip link set eth0 mtu 9000  # Jumbo Frame
```

## 建议2：零拷贝技术

### 传统数据发送过程

```

│                                                             │

│  ┌─────────┐                                                │
│  │  磁盘   │                                                │
│  └────┬────┘                                                │
│       │ DMA拷贝                                             │
│       ▼                                                     │
│  ┌─────────┐                                                │
│  │PageCache│                                                │
│  └────┬────┘                                                │
│       │ CPU拷贝（read系统调用）                              │
│       ▼                                                     │
│  ┌─────────┐                                                │
│  │用户缓冲区│                                                │
│  └────┬────┘                                                │
│       │ CPU拷贝（write系统调用）                             │
│       ▼                                                     │
│  ┌─────────┐                                                │
│  │内核skb  │                                                │

│  总计：4次拷贝（2次DMA + 2次CPU）                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 零拷贝（sendfile）

```
┌─────────────────────────────────────────────────────────────┐
│                    零拷贝发送                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  使用sendfile：                                             │
│                                                             │

│  └────┬────┘                                                │
│       │ DMA拷贝                                             │
│       ▼                                                     │
│  ┌─────────┐                                                │
│  │PageCache│                                                │
│  └────┬────┘                                                │

│  └─────────┘                                                │

└─────────────────────────────────────────────────────────────┘
```

### 代码示例

```c
// 传统方式
char buf[4096];
int in_fd = open("file.txt", O_RDONLY);
int out_fd = socket(AF_INET, SOCK_STREAM, 0);

read(in_fd, buf, sizeof(buf));    // CPU拷贝
write(out_fd, buf, sizeof(buf));  // CPU拷贝

// 零拷贝方式
sendfile(out_fd, in_fd, NULL, file_size);  // 无CPU拷贝

// splice方式（管道传输）
int pipefd[2];
pipe(pipefd);
splice(in_fd, NULL, pipefd[1], NULL, len, 0);
splice(pipefd[0], NULL, out_fd, NULL, len, 0);
```

### 零拷贝优势

| 方式 | CPU拷贝次数 | 上下文切换 |
|------|------------|-----------|
| 传统read/write | 2次 | 4次 |
| sendfile | 0次 | 2次 |
| splice | 0次 | 2次 |

```
│  │                                                     │   │
```

```
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### epoll的优势

```
┌─────────────────────────────────────────────────────────────┐
│                    epoll优势                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  epoll模型：                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │  单进程可监控多个socket                              │   │
│  │                                                     │   │
│  │  epoll_create() ──► 创建epoll实例                   │   │
│  │  epoll_ctl()    ──► 添加/删除监控的socket           │   │
│  │  epoll_wait()   ──► 等待事件就绪                    │   │
│  │                                                     │   │
│  │  ┌─────────────────────────────────────────────┐   │   │
│  │  │  Socket1 ─┐                                 │   │   │
│  │  │  Socket2 ─┼──► epoll ──► 就绪事件 ──► 处理  │   │   │
│  │  │  Socket3 ─┤                                 │   │   │
│  │  │  ...     ─┘                                 │   │   │
│  │  └─────────────────────────────────────────────┘   │   │
│  │                                                     │   │
│  │  一个进程可以处理大量连接                           │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 对比

| 特性 | 阻塞IO | select/poll | epoll |
|------|--------|-------------|-------|
| 单进程处理连接数 | 1个 | 受FD_SETSIZE限制 | 几乎无限制 |
| 时间复杂度 | O(1) | O(n) | O(1) |
| 等待方式 | 进程阻塞 | 轮询 | 事件通知 |
| 上下文切换 | 每连接一次 | 少 | 少 |
| 适用场景 | 简单程序 | 少量连接 | 高并发 |

### epoll代码示例

```c
#include <sys/epoll.h>
#include <fcntl.h>

#define MAX_EVENTS 10000

int main() {
    int epoll_fd = epoll_create1(0);
    
    // 添加监听socket到epoll
    struct epoll_event ev, events[MAX_EVENTS];
    ev.events = EPOLLIN;
    ev.data.fd = listen_fd;
    epoll_ctl(epoll_fd, EPOLL_CTL_ADD, listen_fd, &ev);
    
    while(1) {
        // 等待事件就绪
        int nfds = epoll_wait(epoll_fd, events, MAX_EVENTS, -1);
        
        for(int i = 0; i < nfds; i++) {
            if(events[i].data.fd == listen_fd) {
                // 新连接
                int client_fd = accept(listen_fd, NULL, NULL);
                ev.events = EPOLLIN | EPOLLET;  // 边缘触发
                ev.data.fd = client_fd;
                epoll_ctl(epoll_fd, EPOLL_CTL_ADD, client_fd, &ev);
            } else {
                // 处理客户端数据
                handle_client(events[i].data.fd);
            }
        }
    }
    
    return 0;
}
```

## 全连接队列优化

### 问题背景

全连接队列满时，第三次握手ACK会被丢弃，影响连接建立。

```
┌─────────────────────────────────────────────────────────────┐
│                    全连接队列问题                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  正常情况：                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ SYN → SYN+ACK → ACK → 连接进入全连接队列 → accept  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  队列满时：                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ SYN → SYN+ACK → ACK → 队列满，ACK被丢弃            │   │
│  │                     ↓                               │   │
│  │              服务端重发SYN+ACK                      │   │
│  │                     ↓                               │   │
│  │              客户端以为已连接，发送数据被丢弃        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 优化方法

```bash
## 增大全连接队列
sysctl -w net.core.somaxconn=4096

## 或者在listen()时传入足够大的backlog
listen(fd, 4096);

## 注意：实际队列长度 = min(backlog, somaxconn)
```

### 查看溢出情况

```bash
## 查看全连接队列溢出统计
netstat -s | grep "listen queue"
## 或者
netstat -s | grep overflow

## 查看当前队列情况
ss -lnt
## Recv-Q: 当前队列长度
## Send-Q: 队列最大长度
```

## 半连接队列优化

### 问题背景

半连接队列满时，SYN包会被丢弃。

```
┌─────────────────────────────────────────────────────────────┐
│                    半连接队列问题                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  正常情况：                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ SYN → 创建request_sock → 加入半连接队列 → SYN+ACK  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  队列满时（SYN Flood攻击）：                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 大量SYN → 队列快速填满 → 新SYN被丢弃                │   │
│  │     ↓                                               │   │
│  │ 正常用户无法建立连接                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 优化方法

```bash
## 增大半连接队列
sysctl -w net.ipv4.tcp_max_syn_backlog=4096

## 开启SYN Cookies防御
sysctl -w net.ipv4.tcp_syncookies=1
```

### SYN Cookies原理

```
┌─────────────────────────────────────────────────────────────┐
│                    SYN Cookies原理                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  正常三次握手：                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 服务端需要保存半连接状态，消耗内存                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  SYN Cookies方式：                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 1. 收到SYN，不保存状态，而是计算特殊序列号           │   │
│  │ 2. 序列号 = hash(连接信息 + 时间戳 + 密钥)          │   │
│  │ 3. 发送SYN+ACK，序列号为计算值                       │   │
│  │ 4. 收到ACK时，验证序列号，恢复连接信息               │   │
│  │                                                     │   │
│  │ 优势：不消耗内存，防御SYN Flood攻击                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## TIME_WAIT优化

### TIME_WAIT的作用

```
┌─────────────────────────────────────────────────────────────┐
│                    TIME_WAIT作用                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 确保最后的ACK到达                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 主动关闭方发送FIN                                     │   │
│  │ 被动方回复ACK + FIN                                   │   │
│  │ 主动方回复ACK ──► 如果丢失，被动方会重发FIN          │   │
│  │ TIME_WAIT确保能重发ACK                               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  2. 等待旧连接的数据消散                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 防止旧连接的延迟数据包影响新连接                      │   │
│  │ 等待2MSL时间（约60秒）                               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### TIME_WAIT过多的问题

```bash
## 查看TIME_WAIT数量
netstat -ant | grep TIME_WAIT | wc -l

## 问题：
## 1. 占用内存（每条约2KB）
## 2. 占用端口（连接同一目标时）
## 3. 可能导致新连接失败
```

### 优化方法

```bash
## 1. 复用TIME_WAIT端口（推荐）
sysctl -w net.ipv4.tcp_tw_reuse=1

## 2. 减少TIME_WAIT时间
sysctl -w net.ipv4.tcp_fin_timeout=30

## 3. 增加可用端口范围
sysctl -w net.ipv4.ip_local_port_range="1024 65535"

## 注意：不建议开启 tcp_tw_recycle（已废弃）
```

### 参数说明

| 参数 | 说明 |
|------|------|
| `tcp_tw_reuse` | 允许复用TIME_WAIT状态的端口 |
| `tcp_fin_timeout` | TIME_WAIT持续时间（秒） |
| `ip_local_port_range` | 客户端端口范围 |

## 第9章 网络性能优化建议

## 概述

具备了对内核网络模块的深刻理解之后，在性能方面有哪些优化手段可用呢？本章给出一些开发或者运维中的性能优化建议。注意，是建议，而不是原则。每一种性能优化方法都有它适用或者不适用的场景。

---

## 9.1 网络请求优化

### 建议1：尽量减少不必要的网络IO

网络很好，能降低模块的开发难度，也能用它搭建出更强大的系统。但是这不是你滥用它的理由！

**本机网络IO开销分析**：

```
发送一个网络包：
1. 用户态 → 内核态（系统调用开销）
2. 经过协议栈（花费CPU周期）
3. 进入回环设备的"驱动程序"
4. 触发软中断
5. 接收协议栈处理
6. 唤醒用户进程

如果是跨机，还有：
- 双方网卡的DMA拷贝
- 网络RTT耗时延迟
```

**优化案例**：
- 直接在服务进程内实现功能，而不是通过本机网络调用SDK
- 减少进程上下文切换开销
- 降低部署复杂度

### 建议2：尽量合并网络请求

**问题代码示例**：

```php
<?php
for (安装列表 as 包名) {
    redis->get(包名)
}
```

- 平均用户安装60个APP
- 每次请求需要60次网络请求
- 总耗时至少60个RTT

**优化方法**：

使用批量获取命令：

```php
<?php
// 使用 hmget 或 pipeline
redis->hmget(包1, 包2, 包3, ...);
```

### 建议3：调用者与被调用机器尽可能部署得近一些

**RTT延迟对比**：

| 场景 | RTT延迟 |
|------|---------|
| 机房内部 | ~0.几毫秒 |
| 同地区不同机房 | ~1毫秒 |
| 北京到广东 | ~30-40毫秒 |

**优化建议**：
- 每个机房内部的数据请求在本地机房解决
- 减少跨地区网络传输

### 建议4：内网调用不要用外网域名

**为什么不要用外网域名**：

1. **外网接口慢**：内网可能过个交换机就到达，走外网会绕一圈
2. **带宽成本高**：出入带宽都要计费
3. **NAT单点瓶颈**：NAT服务器容易成为瓶颈

### 建议3：RPS/RFS优化

**RPS（Receive Packet Steering）**：
- 软中断层面的负载均衡
- 将数据包分散到多个CPU处理

**RFS（Receive Flow Steering）**：
- 基于流的负载均衡
- 同一连接的数据包分配到同一CPU

**配置方法**：

```bash
## 查看RPS配置

## 设置RPS
echo f > /sys/class/net/eth0/queues/rx-0/rps_cpus
```

## 9.3 发送过程优化

### 建议1：避免分片

**MTU限制**：
- 以太网默认MTU：1500字节
- 超过MTU需要分片

**优化方法**：
- 控制数据包大小 < MTU
- 使用PMTU发现机制

### 建议2：零拷贝技术

**传统数据发送**：

```
磁盘 → PageCache → 用户缓冲区 → 内核skb → 网卡
       DMA拷贝     CPU拷贝      CPU拷贝    DMA拷贝
```

**零拷贝（sendfile）**：

```
磁盘 → PageCache → 内核skb → 网卡
       DMA拷贝              DMA拷贝
```

**代码示例**：

```c
// 零拷贝发送文件
sendfile(out_fd, in_fd, offset, count);

// splice 管道传输
splice(in_fd, out_fd, len);
```

### 建议3：批量发送

**GSO/TSO**：

| 技术 | 说明 |
|------|------|
| **GSO** | Generic Segmentation Offload |
| **TSO** | TCP Segmentation Offload |

让网卡完成TCP分片，减少CPU开销。

**查看是否开启**：

```bash
## ethtool -k eth0 | grep segmentation
tcp-segmentation-offload: on
generic-segmentation-offload: on
```
