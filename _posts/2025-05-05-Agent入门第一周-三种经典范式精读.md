## 一、Agent 是什么

**定义**：能够感知环境、做出决策、执行行动的智能体。

**四大核心组件**：

| 组件 | 角色 | 类比 |
|---|---|---|
| **LLM** | 大脑 | 理解和生成语言，进行推理决策 |
| **Memory** | 记忆 | 维护对话历史和上下文 |
| **Tools** | 手脚 | 调用外部 API / 计算 / 查数据库 |
| **Planning** | 思维方式 | 决定"先做什么再做什么" |

**Agent ≠ ChatBot**。ChatBot 只有 LLM；Agent 多了 **Memory + Tools + Planning**，能主动产生行动并影响外部世界。

## 二、三种经典范式速览

### ReAct（Reasoning + Acting）

```
Thought → Action → Observation → 循环 / 结束
```

- **特点**：动态反应式，**边想边做**
- **适用**：需要外部工具交互的任务（查天气、查数据库、做计算）
- **优势**：可解释性高（每步都有 Thought），出错能动态纠错
- **劣势**：可能陷入循环；非常依赖模型的"格式遵循"能力

### Plan-and-Solve

```
Plan(规划) → Execute(执行)
```

- **特点**：预先规划式，**先谋后动**
- **适用**：结构性强、可拆解的任务（复杂数学题、多步报告生成）
- **优势**：结构清晰，每步目的明确
- **劣势**：计划是静态的，执行中遇到意外没法动态调整

### Reflection

```
Execute → Reflect → Refine → 循环 / 结束
```

- **特点**：迭代优化式，**边做边改**
- **适用**：对结果质量要求高的任务（代码生成、文章写作）
- **优势**：显著提升输出质量
- **劣势**：调用成本高（每轮都要再过一遍 LLM），实时性差

## 三、核心对比表

| 维度 | ReAct | Plan-and-Solve | Reflection |
|---|---|---|---|
| **决策模式** | 动态反应式 | 预先规划式 | 迭代优化式 |
| **执行顺序** | 边想边做 | 先规划后执行 | 边做边改 |
| **历史结构** | `list[str]` | `str` | `Memory` 类 |
| **格式化方式** | `"\n".join(history)` | 直接累加 | `get_trajectory()` |
| **典型场景** | 工具调用 | 复杂分解 | 质量优化 |

## 四、最重要的发现：历史处理本质相同

跑通三种范式后，我对比了它们的源码，得到一个反直觉的结论：

> **三种范式的历史处理本质是同一件事 —— 把过往步骤拼成字符串塞回 prompt。**

只是"拼接方式"和"封装层级"不同：

- ReAct 用 `list` + `join`，因为每步是结构化的 Thought/Action/Observation
- Plan-and-Solve 直接 `str` 累加，因为 Plan 阶段一次性产出，Execute 是线性追加
- Reflection 包成了 `Memory` 类，因为它要支持"按轨迹回溯 + 选择性重写"

**所以**：表面上是"三种不同范式"，底层都是同一个**"历史 → prompt"循环**。差异在**循环的形状**和**触发条件**。

理解这一点后，再去读 LangGraph、AutoGen 之类的框架源码会快很多 —— 它们都是这个 mental model 的不同包装。

## 五、踩坑实录

### 坑 1：Agent 不肯用 `Finish` 收尾

**现象**：LLM 一直在 Thought / Action 之间循环，不输出 `Finish[...]`。

**解决**：
1. 在 system prompt 里**明确举例** Finish 的格式
2. 解析逻辑改成**先匹配 Finish 再匹配 Action**（因为 LLM 偶尔会输出不带 `Action:` 前缀的 `Finish[...]`）

### 坑 2：原项目代码强依赖外部 API（Tavily 搜索）

**现象**：教材跑示例代码要 Tavily API key，没 key 就跑不动。

**解决**：把 `web_search` 工具替换成本地 mock 或自己写的 `get_time` 等无依赖工具。**核心心法**：工具是 Agent 的"插件"，可以随便换，不影响范式本身。

### 坑 3：模型对"输出格式"的遵循度差异巨大

不同 LLM（Qwen / GPT / Claude）对 ReAct 这种结构化 prompt 的服从度差别很大。学习阶段建议先用对格式服从最好的模型（Claude / GPT-4 系列），调通流程后再换便宜的。

## 六、学到的最重要的事

1. **范式选择 = 任务特征匹配**：没有"最好的范式"，只有"对当前任务最合适的范式"
2. **Prompt 工程是 Agent 的基础工程**：精确的格式说明 = 稳定的 Agent 表现
3. **本地工具优先**：能用 Python 实现的别去调外部 API，省钱省事
4. **三种范式可组合**：复杂任务可以"Plan + ReAct"（规划阶段用 Plan，执行步骤用 ReAct），这就是后面 Week 2 要学的多 Agent 协作的雏形
