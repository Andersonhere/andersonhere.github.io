> **核心问题**：如何优化写入性能？

## 本章大纲

### 43.1 按顺序写入磁盘

**核心思想**：所有写入追加到日志末尾

```
传统文件系统:
随机写入，需要寻道

LFS:
顺序写入，无寻道
┌─────────────────────────────────────────────┐
│ 写入1 → 写入2 → 写入3 → 写入4 → ...        │
└─────────────────────────────────────────────┘
```

### 43.2 顺序而高效地写入

**写入缓冲**：

```
收集多个写入，一次性写入:

内存缓冲:
┌─────┬─────┬─────┬─────┐
│文件A│文件B│元数据│ ... │
└─────┴─────┴─────┴─────┘
         ↓
    一次性写入磁盘
```

### 43.3 要缓冲多少

**权衡**：
- 缓冲越多，效率越高
- 缓冲越多，数据丢失风险越大

**典型设置**：
- 定期刷新（如每秒）
- 超过阈值刷新（如缓冲满）

### 43.4 问题：查找 inode

**传统方式**：固定位置的 inode 表

**LFS 方式**：inode 分散在各处

### 43.5 通过间接解决方案：inode 映射

**inode 映射表**：

```
imap: inode号 → 磁盘地址

读取文件:
1. 查找 imap，获取 inode 地址
2. 读取 inode
3. 根据 inode 读取数据块
```

### 43.6 检查点区域

**问题**：imap 在哪里？

**解决方案**：检查点区域 (CR)

```
磁盘开头:
┌────────────────────────────────────────┐
│ CR: 指向 imap, 超级块等                │
└────────────────────────────────────────┘

启动时读取 CR，获取整个文件系统状态
```

### 43.7 从磁盘读取文件：回顾

```
读取 /foo/bar:

1. 读取 CR
2. 从 CR 找到 imap
3. 查找根目录 inode
4. 读取根目录，找 foo
5. 从 imap 找 foo 的 inode
6. 读取 foo 目录，找 bar
7. 从 imap 找 bar 的 inode
8. 读取 bar 的数据
```

### 43.8 目录如何

**目录结构与传统相同**：

```
目录项:
┌──────────────────────────┐
│ inode号 │ 文件名         │
└──────────────────────────┘

但 inode 位置通过 imap 查找
```

### 43.9 一个新问题：垃圾收集

**问题**：旧版本数据占用空间

```
文件 A 的历史:
版本1: 块100
版本2: 块200  (版本1 变为垃圾)
版本3: 块300  (版本2 变为垃圾)
```

### 43.10 确定块的死活

**引用计数**：

```
每个块记录:
- 是否被引用
- 引用计数

垃圾: 引用计数 = 0
```

### 43.11 策略问题：要清理哪些块，何时清理

**清理策略**：

| 策略 | 说明 |
|------|------|
| 贪婪 | 选择垃圾最多的段 |
| 代价感知 | 考虑清理开销 |

### 43.12 崩溃恢复和日志

**LFS 天然支持崩溃恢复**：

- 写入是原子的（整个段）
- 检查点确保一致性

### 43.13 小结

**LFS 优缺点**：

| 优点 | 缺点 |
|------|------|
| 写入性能高 | 读取需要间接查找 |
| 崩溃恢复简单 | 垃圾回收开销 |
| 支持快照 | 实现复杂 |
