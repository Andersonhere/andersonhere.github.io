> **核心问题**：崩溃后如何保证文件系统一致性？

## 本章大纲

### 42.1 一个详细的例子

**场景**：追加数据到文件

```
操作涉及:
1. 更新数据块
2. 更新 inode (大小)
3. 更新数据位图

崩溃可能在任何步骤发生！
```

**可能的崩溃状态**：

| 情况 | 已完成 | 结果 |
|------|--------|------|
| 情况1 | 数据块 | 数据存在但无法访问 |
| 情况2 | 数据块+位图 | 空间泄露 |
| 情况3 | 全部 | 正确 |
| 情况4 | 仅inode | 数据丢失，垃圾数据 |
| ... | ... | ... |

### 42.2 解决方案 1：文件系统检查程序 (fsck)

**fsck 检查项**：

| 检查 | 说明 |
|------|------|
| 超级块 | 检查文件系统信息 |
| 空闲块 | 检查位图与实际使用一致性 |
| inode状态 | 检查 inode 是否有效 |
| inode链接 | 检查链接计数 |
| 重复块 | 检查块是否被多次引用 |
| 坏块 | 检查无效块指针 |
| 目录结构 | 检查目录完整性 |

**问题**：
- 恢复时间可能很长
- 可能无法恢复所有数据

### 42.3 解决方案 2：日志（或预写日志）

**核心思想**：先记录要做什么，再做

```
日志协议:
1. 写入事务开始 (TxB)
2. 写入操作内容
3. 写入事务结束
4. 执行实际操作
5. 释放日志 (TxE)
```

**日志结构**：

```
┌────────────────────────────────────────┐
│ TxB │ 数据块 │ inode │ 位图 │ TxE │...│
└────────────────────────────────────────┘
```

**恢复流程**：

```
崩溃恢复:
1. 扫描日志
2. 找到完整事务 (有 TxB 和 TxE)
3. 重做事务中的操作
4. 丢弃不完整事务
```

#### 数据日志

```c
// 写入日志
log_write(TxB);
log_write(data_block);
log_write(inode);
log_write(bitmap);
log_write(TxE);

// 执行实际操作
write_data();
update_inode();
update_bitmap();

// 释放日志
log_free();
```

#### 元数据日志

```c
// 只记录元数据变更，减少日志量
log_write(TxB);
log_write(inode);    // 只记录元数据
log_write(bitmap);
log_write(TxE);
```

### 42.4 解决方案 3：其他方法

#### 写时复制 (Copy-on-Write)

```
LFS / ZFS / Btrfs 使用:

1. 分配新块写入新数据
2. 更新指针指向新块
3. 原子切换指针
```

#### 日志结构文件系统 (LFS)

```
所有写入都是顺序追加:
- 无覆盖写入
- 天然支持崩溃恢复
```

### 42.5 小结

| 方案 | 优点 | 缺点 |
|------|------|------|
| fsck | 简单 | 恢复慢 |
| 日志 | 快速恢复 | 写放大 |
| 写时复制 | 原子性 | 空间开销 |
