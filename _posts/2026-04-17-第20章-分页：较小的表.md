> **核心问题**：如何减少页表占用？

## 本章大纲

### 20.1 简单的解决方案：更大的页

**思路**：增大页大小，减少页数

| 页大小 | 页表大小 (32位,4GB) |
|--------|---------------------|
| 4KB | 4MB |
| 4MB | 4KB |

**问题**：
- 内部碎片增加
- 不适合稀疏地址空间

### 20.2 混合方法：分页和分段

**思路**：每个段一个页表

```
地址空间:
┌─────────┐
│  栈段   │ → 栈页表
├─────────┤
│  堆段   │ → 堆页表
├─────────┤
│ 代码段  │ → 代码页表
└─────────┘

虚拟地址: [段号 | VPN | 偏移]
```

**优点**：稀疏地址空间内存占用少
**缺点**：分段的问题又回来了

### 20.3 多级页表

**核心思想**：页表也分页

**二级页表示例**：

```
虚拟地址 (32位, 4KB页):
┌────────────┬────────────┬────────────┐
│ 页目录索引 │ 页表索引   │ 页内偏移   │
│   (10位)   │  (10位)    │  (12位)   │
└────────────┴────────────┴────────────┘

转换过程:
1. 页目录索引 → 页目录项 (PDE)
2. 页表索引 → 页表项 (PTE)
3. 偏移 → 最终物理地址
```

**多级页表结构**：

```
页目录 (1页)
    │
    ├── 页表 0 (1页或不存在)
    │      └── 页框映射
    ├── 页表 1 (不存在则节省)
    └── ...
```

**优点**：
- 稀疏地址空间节省内存
- 页表按需分配

**缺点**：
- 多级查找增加延迟
- TLB 更重要

### 20.4 反向页表

**思路**：每个物理页框一个表项

```
传统页表: 虚拟页 → 物理页
反向页表: 物理页 → 虚拟页

查找: 使用哈希表加速
```

**优点**：页表大小与物理内存相关，与虚拟地址空间无关
**缺点**：查找复杂，需要哈希

### 20.5 将页表交换到磁盘

**思路**：页表太大时换出到磁盘

**适用场景**：
- 32位系统大地址空间
- 64位系统

### 20.6 小结

| 方案 | 特点 |
|------|------|
| 更大的页 | 简单，但内部碎片 |
| 分段+分页 | 复杂，有分段问题 |
| 多级页表 | 常用，需 TLB |
| 反向页表 | 节省空间，查找复杂 |
