> **核心问题**：如何改进 NFS 的设计？

## 本章大纲

### 47.1 AFS 版本 1

**设计目标**：
- 支持大规模分布式环境
- 提供一致的文件访问

**基本架构**：

```
客户端              Venus            服务器
   │                  │                │
   │── 文件请求 ─────→│                │
   │                  │── 检查缓存 ───→│
   │                  │←── 获取文件 ───│
   │←── 返回文件 ─────│                │
```

### 47.2 版本 1 的问题

| 问题 | 说明 |
|------|------|
| 性能差 | 每次操作都访问服务器 |
| 延迟高 | 网络往返开销 |
| 扩展性差 | 服务器负载高 |

### 47.3 改进协议

**关键改进**：
1. 整个文件缓存
2. 回调机制

### 47.4 AFS 版本 2

**回调机制**：

```
文件缓存流程:
1. 客户端请求文件
2. 服务器发送文件 + 建立回调
3. 客户端缓存整个文件
4. 其他客户端修改文件时，服务器发送回调
5. 原客户端使缓存失效
```

```
客户端A              服务器              客户端B
   │                   │                   │
   │── 获取文件 ──────→│                   │
   │←── 文件+回调 ─────│                   │
   │                   │←── 修改文件 ──────│
   │←── 回调通知 ──────│                   │
   │                   │                   │
   缓存失效            │                   │
```

### 47.5 缓存一致性

**AFS 保证**：

- 回调承诺：文件改变时通知
- 回调后缓存失效
- 下次访问重新获取

**一致性保证**：

```
强一致性保证:
- 写入立即可见
- 回调确保缓存失效
- 无陈旧数据
```

### 47.6 崩溃恢复

**客户端崩溃**：

```
重新连接时:
1. 丢弃所有缓存
2. 重新建立回调
```

**服务器崩溃**：

```
客户端检测:
1. 心跳失败
2. 丢弃缓存
3. 重连服务器
```

### 47.7 AFSv2 的扩展性和性能

**优势**：

| 方面 | 改进 |
|------|------|
| 服务器负载 | 减少（缓存命中率高） |
| 网络流量 | 减少（整文件缓存） |
| 一致性 | 更强（回调机制） |

### 47.8 AFS：其他改进

| 特性 | 说明 |
|------|------|
| 卷位置透明 | 文件可迁移 |
| 安全性 | Kerberos 认证 |
| 可扩展 | 支持数千客户端 |

### 47.9 小结

**AFS vs NFS 对比**：

| 方面 | NFS | AFS |
|------|-----|-----|
| 缓存粒度 | 块 | 整文件 |
| 一致性 | 弱 | 强 |
| 状态 | 无状态 | 有回调状态 |
| 复杂性 | 低 | 高 |
