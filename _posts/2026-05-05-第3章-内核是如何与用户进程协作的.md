## 3.3 内核和用户进程协作之阻塞方式

## 概述

本章开头说过同步阻塞的网络IO（在Java中习惯叫BIO）的优点是使用起来非常方便，但缺点就是性能非常差。

在同步阻塞IO模型中，虽然用户进程里在最简单的情况下只有两三行代码，但实际上用户进程和内核配合做了非常多的工作：
1. 用户进程发起创建socket的指令
2. 切换到内核态完成了内核对象的初始化
3. Linux在数据包的接收上，是硬中断和ksoftirqd线程进行处理
4. 当ksoftirqd线程处理完以后，再通知相关的用户进程

### 同步阻塞IO总体流程

```
┌──────────────────────────────────────────────────────────────┐
│  1.创建socket    2.等待接收        用户进程                    │
│       ↓             ↓                                         │
│       └─────────────────────────────────┐                   │
└──────────────────────────────────────────│───────────────────┘
                                           │
┌──────────────────────────────────────────│───────────────────┐
│                              等待队列    │                    │
│                                 ↓        │                    │
│  ┌──────────┐    socket接收队列  ↓        │                    │
│  │  skb     │ ←───────────────────┘        │                    │
│  └──────────┘                              │                    │
│       ↑                                    │                    │
│       │  8.放到socket的接收队列中           │                    │
│       │                                    │                    │
│  7.从RingBuffer上摘下数据包                │                    │
│       ↑                                    │                    │
│       │                            内核对象                    │
│       │                                    │                    │
│  6.处理完发出软中断       9.唤醒等待队列上的进程              │
│       ↑                                    │                    │
│  ksoftirqd ───────────────────────────────┘                   │
│  内核线程                                                       │
│       ↑                                                         │
│       │  5.硬中断通知                                            │
│       │                                                         │
│  硬中断                                                          │
│       ↑                                                         │
│       │  4.网卡把帧DMA到内存                                     │
│       │                                                         │
│  3.数据包抵达网卡                                               │
└───────────────────────────────────────────────────────────────┘
```

## 3.3.1 等待接收消息

### 系统调用入口

用户进程调用 `recv` 函数，通过 strace 跟踪可以看到会执行 `recvfrom` 系统调用：

```c
// file: net/socket.c
SYSCALL_DEFINE6(recvfrom, int, fd, void __user *, ubuf, size_t, size,
                unsigned int, flags, struct sockaddr __user *, addr,
                int __user *, addr_len)
{
    struct socket *sock;
    // 根据用户传入的fd找到socket对象
    sock = sockfd_lookup_light(fd, &err, &fput_needed);

    err = sock_recvmsg(sock, &msg, size, flags);
}
```

### 调用链

```
recvfrom()
    ↓
sock_recvmsg()
    ↓
__sock_recvmsg_nosec()
    ↓
sock->ops->recvmsg()  →  inet_recvmsg()
    ↓
sk->sk_prot->recvmsg()  →  tcp_recvmsg()
```

### tcp_recvmsg 核心实现

```c
// file: net/ipv4/tcp.c
int tcp_recvmsg(struct kiocb *iocb, struct sock *sk, struct msghdr *msg,
                size_t len, int nonblock, int flags, int *addr_len)
{
    int copied = 0;

    // 遍历接收队列接收数据
    skb_queue_walk(&sk->sk_receive_queue, skb) {
        ...
    }

    do {
        if (copied >= target) {
            release_sock(sk);
            lock_sock(sk);
        } else {
            // 没有收到足够数据，启用 sk_wait_data 阻塞当前进程
            sk_wait_data(sk, &timeo);
        }
    }
}
```

### sk_wait_data 阻塞进程

```c
// file: net/core/sock.c
int sk_wait_data(struct sock *sk, long *timeo)
{
    // 当前进程(current)关联到所定义的等待队列项上
    DEFINE_WAIT(wait);

    // 调用 sk_sleep 获取 sock 对象下的 wait
    // 并准备挂起，将进程状态设置为可打断 (INTERRUPTIBLE)
    prepare_to_wait(sk_sleep(sk), &wait, TASK_INTERRUPTIBLE);
    set_bit(SOCK_ASYNC_WAITDATA, &sk->sk_socket->flags);

    // 通过调用 schedule_timeout 让出 CPU，然后进行睡眠
    rc = sk_wait_event(sk, timeo,
                       !skb_queue_empty(&sk->sk_receive_queue));
}
```

### 关键操作总结

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | `DEFINE_WAIT(wait)` | 定义等待队列项，关联当前进程 |
| 2 | `prepare_to_wait()` | 将进程状态设为 `TASK_INTERRUPTIBLE` |
| 3 | `sk_sleep(sk)` | 获取 socket 的等待队列头 |
| 4 | `schedule_timeout()` | 让出 CPU，进程睡眠 |

### 进程阻塞示意图

```
进程A (用户态)
    ↓ recvfrom 系统调用
进程A (内核态)
    ↓ tcp_recvmsg()
    ↓ 发现接收队列为空
    ↓ prepare_to_wait()
    ↓ 设置进程状态为 TASK_INTERRUPTIBLE
    ↓ schedule()
进程A 睡眠，让出 CPU
    ↓
Linux 调度器选择进程B执行
```

## 3.3.2 软中断模块

当网络数据包到达网卡后，经过硬中断和软中断处理，最终会唤醒等待的进程。

### 数据到达时的处理流程

```
1. 网卡收到数据包
    ↓
2. DMA 到 RingBuffer
    ↓
3. 硬中断通知 CPU
    ↓
4. ksoftirqd 线程处理软中断
    ↓
5. 协议栈处理 (ip_rcv → tcp_v4_rcv)
    ↓
6. 数据放到 socket 接收队列
    ↓
7. 调用 sk_data_ready 唤醒等待进程
```

### tcp_v4_rcv 处理

```c
// file: net/ipv4/tcp_ipv4.c
int tcp_v4_rcv(struct sk_buff *skb)
{
    // 查找对应的 sock
    sk = __inet_lookup_skb(&tcp_hashinfo, skb);

    // 将数据放入接收队列
    if (!sk_filter(sk, skb)) {
        tcp_queue_rcv(sk, skb);
    }

    // 唤醒等待的进程
    sk->sk_data_ready(sk);  // 实际调用 sock_def_readable
}
```

### sock_def_readable 唤醒进程

```c
// file: net/core/sock.c
static void sock_def_readable(struct sock *sk, int len)
{
    struct socket_wq *wq;

    rcu_read_lock();
    wq = rcu_dereference(sk->sk_wq);

    // 唤醒等待队列上的进程
    wake_up_interruptible_sync_poll(&wq->wait, POLLIN | POLLPRI |
                                    POLLRDNORM | POLLRDBAND);
    rcu_read_unlock();
}
```

### 进程唤醒流程

```
ksoftirqd 线程
    ↓ tcp_v4_rcv()
    ↓ tcp_queue_rcv() - 数据放入接收队列
    ↓ sk_data_ready() → sock_def_readable()
    ↓ wake_up_interruptible_sync_poll()
    ↓
唤醒等待队列上的进程A
    ↓
进程A 从 TASK_INTERRUPTIBLE → TASK_RUNNING
    ↓
等待调度器调度执行
```

## 3.3.3 同步阻塞总结

### 性能问题分析

**同步阻塞IO的主要开销：**

1. **上下文切换开销**
   - 用户态 → 内核态（系统调用）
   - 进程睡眠（schedule）
   - 进程唤醒（wake_up）
   - 内核态 → 用户态（返回用户空间）

2. **CPU 调度开销**
   - 每个阻塞的进程都需要被调度
   - 高并发时大量进程阻塞/唤醒

3. **内存开销**
   - 每个连接需要一个进程/线程
   - 进程/线程的栈空间

### 为什么性能差？

| 问题 | 说明 |
|------|------|
| **进程/线程数爆炸** | 一个连接需要一个进程/线程，高并发时资源消耗巨大 |
| **上下文切换频繁** | 每次数据到达都需要唤醒进程，切换开销大 |
| **CPU 缓存失效** | 频繁切换导致 CPU 缓存命中率低 |

### 适用场景

- 客户端应用
- 连接数较少的服务
- 简单的网络编程场景

### 不适用场景

- 高并发服务端
- 需要处理大量连接
- 对性能要求高的场景

## 3.4 内核和用户进程协作之epoll

## 概述

多路复用 epoll 是 Linux 上最高效的 IO 模型，广泛应用于高并发服务端开发。

### epoll vs 同步阻塞

| 特性 | 同步阻塞 (BIO) | epoll |
|------|---------------|-------|
| 一个进程处理的连接数 | 1个 | 多个 |
| 等待方式 | 进程阻塞 | 进程阻塞，但可监控多个socket |
| 上下文切换 | 每个连接一次 | 多个连接共用一次 |
| 内存开销 | 每连接一个进程/线程 | 单进程处理多连接 |

## 3.4.2 为epoll添加socket

### epoll_ctl 系统调用

```c
// 用户态调用
epoll_ctl(efd, EPOLL_CTL_ADD, cfd, &ev);
```

```c
// file: fs/eventpoll.c
SYSCALL_DEFINE4(epoll_ctl, int, epfd, int, op, int, fd,
                struct epoll_event __user *, event)
{
    struct eventpoll *ep;
    struct file *file, *tfile;

    // 获取 eventpoll 对象
    ep = file->private_data;

    // 获取要添加的 socket 文件
    tfile = fcheck(fd);

    // 执行添加操作
    switch (op) {
    case EPOLL_CTL_ADD:
        error = ep_insert(ep, &epds, tfile, fd);
        break;
    }
}
```

### ep_insert 核心实现

```c
// file: fs/eventpoll.c
static int ep_insert(struct eventpoll *ep, struct epoll_event *event,
                     struct file *tfile, int fd)
{
    struct epitem *epi;

    // 分配 epitem 对象（代表一个被监控的socket）
    epi = kmem_cache_alloc(epi_cache, GFP_KERNEL);

    // 初始化 epitem
    INIT_LIST_HEAD(&epi->rdllink);
    ep_rb_initnode(&epi->rbn);

    // 将 epitem 加入红黑树
    ep_rbtree_insert(ep, epi);

    // 注册回调函数
    ep_ptable_queue_proc(epi);
}
```

### epitem 结构体

```c
// file: fs/eventpoll.c
struct epitem {
    // 红黑树节点
    struct rb_node rbn;

    // 就绪队列节点
    struct list_head rdllink;

    // 关联的 socket
    struct epoll_filefd ffd;

    // 所属的 eventpoll
    struct eventpoll *ep;

    // 等待队列
    struct epoll_event event;

    // 等待队列项
    wait_queue_t wait;

    // 回调函数
    wait_queue_func_t wait_func;
};
```

### 回调函数注册

```c
// file: fs/eventpoll.c
static void ep_ptable_queue_proc(struct file *file,
                                 wait_queue_head_t *whead,
                                 poll_table *pt)
{
    struct epitem *epi = ep_item_from_epqueue(pt);

    // 初始化等待队列项，设置回调函数
    init_waitqueue_func_entry(&epi->wait, ep_poll_callback);

    // 将等待队列项添加到 socket 的等待队列
    add_wait_queue(whead, &epi->wait);
}
```

### 关键点：回调函数

```c
// file: fs/eventpoll.c
static int ep_poll_callback(wait_queue_t *wait, unsigned mode,
                            int sync, void *key)
{
    struct epitem *epi = ep_item_from_wait(wait);
    struct eventpoll *ep = epi->ep;

    // 将 epitem 添加到就绪队列
    list_add_tail(&epi->rdllink, &ep->rdllist);

    // 唤醒在 epoll_wait 上等待的进程
    wake_up_locked(&ep->wq);
}
```

## 3.4.3 epoll_wait之等待接收

### epoll_wait 系统调用

```c
// 用户态调用

```

```c
// file: fs/eventpoll.c
SYSCALL_DEFINE4(epoll_wait, int, epfd, struct epoll_event __user *, events,
                int, maxevents, int, timeout)
{
    struct eventpoll *ep;

    ep = file->private_data;

    // 核心等待函数
    error = ep_poll(ep, events, maxevents, timeout);
}
```

### ep_poll 核心实现

```c
// file: fs/eventpoll.c
static int ep_poll(struct eventpoll *ep, struct epoll_event __user *events,
                   int maxevents, long timeout)
{
    // 检查就绪队列是否有数据
    if (!list_empty(&ep->rdllist)) {
        // 有数据，直接返回
        return ep_send_events(ep, events, maxevents);
    }

    // 没有数据，阻塞等待
    init_waitqueue_entry(&wait, current);
    add_wait_queue(&ep->wq, &wait);

    for (;;) {
        set_current_state(TASK_INTERRUPTIBLE);

        if (!list_empty(&ep->rdllist) || timeout == 0)
            break;

        // 让出 CPU，进入睡眠
        schedule();
    }

    // 从就绪队列复制事件到用户空间
    return ep_send_events(ep, events, maxevents);
}
```

## 3.4.4 数据来了

### 数据到达时的处理流程

```
1. 网卡收到数据包
    ↓
2. DMA 到 RingBuffer
    ↓
3. 硬中断
    ↓
4. ksoftirqd 处理软中断
    ↓
5. tcp_v4_rcv → tcp_rcv_established
    ↓
6. 数据放入 socket 接收队列
    ↓
7. 调用 sk_data_ready
    ↓
8. 触发 ep_poll_callback
    ↓
9. socket 加入就绪队列
    ↓
10. 唤醒 epoll_wait 中的进程
```

### 与同步阻塞的对比

| 阶段 | 同步阻塞 | epoll |
|------|---------|-------|
| 等待时 | 每个socket一个进程阻塞 | 单进程监控多个socket |
| 数据到达 | 唤醒对应进程 | 触发回调，加入就绪队列 |
| 唤醒 | 每次唤醒一个进程 | 批量处理就绪事件 |
| 上下文切换 | 每连接一次 | 多连接共用 |

## 3.4.5 小结

### epoll 高性能的根本原因

**不是红黑树，而是：**

1. **单进程监控多个连接**
   - 一个进程可以处理成千上万个连接
   - 避免了进程/线程数爆炸

2. **回调机制**
   - 数据到达时主动通知
   - 不需要轮询检查

3. **就绪队列**
   - 只返回有数据的socket
   - 避免无效遍历

4. **批量处理**
   - 一次 epoll_wait 可以获取多个就绪事件
   - 减少系统调用次数

### epoll 阻塞的本质

epoll_wait 也会阻塞进程，但这**不影响高性能**：

- 阻塞的是监控者（单进程）
- 而不是被监控者（多个socket）
- 一个进程阻塞，管理千万个连接

### 为什么 Redis 性能高？

Redis 使用单线程 + epoll：

```c
// Redis 事件循环伪代码
while (true) {
    int n = epoll_wait(efd, events, maxevents, timeout);
    for (int i = 0; i < n; i++) {
        // 处理就绪事件
        processEvent(events[i]);
    }
}
```

优势：
- 单线程避免了锁竞争
- epoll 高效处理多连接
- 内存操作极快

## 3.5 本章总结

### 阻塞的本质

**阻塞 = 进程让出 CPU，进入睡眠状态**

- 修改进程状态为 `TASK_INTERRUPTIBLE`
- 调用 `schedule()` 让出 CPU
- 等待被唤醒

### 同步阻塞的开销

| 开销项 | 说明 |
|--------|------|
| 进程/线程资源 | 每连接一个进程，内存开销大 |
| 上下文切换 | 每次阻塞/唤醒都需要切换 |
| CPU 调度 | 大量进程竞争 CPU |

### epoll 的优势

| 优势 | 说明 |
|------|------|
| 资源占用少 | 单进程处理多连接 |
| 上下文切换少 | 批量处理就绪事件 |
| 无效操作少 | 只处理有数据的连接 |

### 适用场景

- **同步阻塞**: 客户端、连接数少的场景
- **epoll**: 服务端、高并发场景

## 第3章 内核是如何与用户进程协作的

## 3.1 相关实际问题

在上一章中讲述了网络包是如何被从网卡送到协议栈的，接下来内核还有一项重要的工作，就是在协议栈接收处理完输入包以后，要能通知到用户进程，让用户进程能够收到并处理这些数据。

进程和内核配合有很多种方案，本章只深入分析两种典型的：

### 方案一：同步阻塞（BIO）

一般在客户端使用。优点是使用起来非常方便，非常符合人的思维方式，但缺点就是性能较差。

```c
int main() {
    int sk = socket(AF_INET, SOCK_STREAM, 0);
    connect(sk, ...);
    recv(sk, ...);
}
```

### 方案二：多路IO复用（epoll）

这种方案在服务端用得比较多。Linux上多路复用方案有 select、poll、epoll，其中 epoll 的性能表现最优秀。

```c
int main() {
    listen(lfd, ...);

    cfd1 = accept(...);
    cfd2 = accept(...);
    efd = epoll_create(...);

    epoll_ctl(efd, EPOLL_CTL_ADD, cfd1, ...);
    epoll_ctl(efd, EPOLL_CTL_ADD, cfd2, ...);

    epoll_wait(efd, ...);
}
```

### 学习目标

无论是这两种方案中的哪一种，内核都能在接收到数据的时候和用户进程协作，通知用户进程进行下一步处理。但是在高并发情况下，同步阻塞的IO方案的性能比较差，epoll 的表现较好。具体原因是什么？

学习完本章以后，你将深刻理解以下几个工作实践中的问题：

1. **阻塞到底是怎么一回事？**
   在网络开发模型中，经常会遇到阻塞和非阻塞的概念。更要命的是还有人经常把这两个概念和同步、异步放到一起。通过本章对源码的分析，你将深刻理解阻塞到底是怎么一回事。

2. **同步阻塞IO都需要哪些开销？**
   都说同步阻塞IO性能差，我们应该更深入地了解这种方式下都需要哪些 CPU 开销，而不是只简单地说差。

3. **多路复用epoll为什么就能提高网络性能？**
   多路复用的概念在网络编程里非常重要，但可惜很多人对它理解得不够彻底。比如为什么 epoll 就比同步阻塞的IO模型性能好？可能有的人能隐隐约约知道内部的一个红黑树，但其实这仍然不是 epoll 性能优越的根本原因。

4. **epoll也是阻塞的？**
   很多人以为只要一提到阻塞，就是性能差。当听说 epoll 也是可能会阻塞进程的以后，感觉诧异。这都是对 epoll 理解不深造成的，本章将会深度拆解 epoll 的工作原理。epoll 的阻塞并不影响它高性能。

5. **为什么Redis的网络性能很突出？**
   Redis 的性能表现非常抢眼。那么它性能优异的秘诀究竟在哪里？我们在开发自己的接口的时候是否有可以学习它的点？

## 3.2 socket的直接创建

在开始介绍网络IO模型之前，需要先介绍一个前序知识，那就是 socket 是如何在内核中表示的。

### socket 系统调用

从开发者的角度来看，调用 `socket` 函数可以创建一个 socket：

```c
int main() {
    int sk = socket(AF_INET, SOCK_STREAM, 0);
}
```

等这个 socket 函数调用执行完以后，用户层面看到返回的是一个整数型的句柄，但其实内核在内部创建了**一系列的 socket 相关的内核对象**。

### socket 内核结构

```
┌─────────────────────────────────────────────────────────┐
│                    struct socket                         │
│  ┌─────────────────────────────────────────────────────┐│
│  │ struct file *file;                                  ││
│  │ struct proto_ops *ops;                              ││
│  │ struct sock *sk;  ─────────────────┐                ││
│  └────────────────────────────────────│────────────────┘│
└───────────────────────────────────────│─────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────┐
│                      struct sock                         │
│  ┌─────────────────────────────────────────────────────┐│
│  │ struct proto *sk_prot;    // 协议处理               ││
│  │ sk_receive_queue;         // 接收队列               ││
│  │ sk_wq;                    // 等待队列               ││
│  │ void (*sk_data_ready)(...); // 数据就绪回调         ││
│  └─────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────┘
```

### 关键数据结构

| 结构体 | 作用 |
|--------|------|
| `struct socket` | BSD socket 层，用户态接口 |
| `struct sock` | INET socket 层，协议无关层 |
| `sk_receive_queue` | 接收数据队列 |
| `sk_wq` | 等待队列，阻塞进程在这里等待 |
| `sk_data_ready` | 数据就绪时的回调函数 |

### 源码跟踪

```c
// file: net/socket.c
SYSCALL_DEFINE3(socket, int, family, int, type, int, protocol)
{
    retval = sock_create(family, type, protocol, &sock);
    ...
}
```

这些内核对象互相之间的关系比较复杂，后续在分析阻塞或者 epoll 的时候，我们需要不定时回顾 socket 的内核结构。
