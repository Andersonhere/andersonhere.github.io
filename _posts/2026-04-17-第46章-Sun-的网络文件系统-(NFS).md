> **核心问题**：如何实现分布式文件系统？

## 本章大纲

### 46.1 基本分布式文件系统

**目标**：让远程文件访问像本地文件一样

```
客户端                    NFS服务器
   │                          │
   │── mount /remote ────────→│
   │                          │
   │── read /remote/file ────→│
   │←── 文件内容 ─────────────│
```

### 46.2 交出 NFS

**关键设计决策**：

1. **无状态服务器**：服务器不维护客户端状态
2. **幂等操作**：操作可安全重试

### 46.3 关注点：简单快速的服务器崩溃恢复

**无状态的优势**：

```
服务器崩溃恢复:
- 无需恢复状态
- 无需重建连接
- 客户端重试即可
```

### 46.4 快速崩溃恢复的关键：无状态

**NFS 协议**：

| 操作 | 说明 |
|------|------|
| GETATTR | 获取文件属性 |
| SETATTR | 设置文件属性 |
| LOOKUP | 查找文件 |
| READ | 读取文件 |
| WRITE | 写入文件 |
| CREATE | 创建文件 |
| REMOVE | 删除文件 |

### 46.5 NFSv2 协议

**文件句柄**：

```
文件句柄包含:
- 文件系统标识
- inode 号
- 代号 (generation number)

用于唯一标识文件，即使 inode 重用
```

### 46.6 从协议到分布式文件系统

**客户端缓存**：

```
客户端缓存文件数据:
1. 首次读取从服务器获取
2. 后续读取使用缓存
3. 定期验证缓存有效性
```

### 46.7 利用幂等操作处理服务器故障

**幂等操作**：执行多次与执行一次效果相同

```
幂等操作:
- READ
- GETATTR
- WRITE (使用偏移量)

非幂等操作:
- MKDIR (重复创建会失败)
- 解决: 使用唯一标识符
```

### 46.8 提高性能：客户端缓存

**缓存一致性**：

```
问题: 多客户端缓存同一文件

解决方案:
- 定期验证 (GETATTR)
- 写入时使其他缓存失效
```

### 46.9 缓存一致性问题

**NFS 方式**：

```
close-to-open 一致性:
- 打开文件时验证缓存
- 关闭文件时刷新写入

宽松的一致性模型，但性能好
```

### 46.10 小结

**NFS 设计要点**：

| 设计 | 目的 |
|------|------|
| 无状态 | 简化崩溃恢复 |
| 幂等操作 | 支持重试 |
| 客户端缓存 | 提高性能 |
| 缓存一致性 | 保证正确性 |
