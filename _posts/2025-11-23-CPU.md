## 一、从发展历史看 CPU 与操作系统

通过操作系统技术的发展史，能直观看到 **人力瓶颈 → 吞吐瓶颈 → 交互需求 → 软硬协同** 这条主线；下面仅从 **CPU 利用与控制权** 的角度做极简回顾（与内存、I/O 的协同在文末「硬件协同」一句收口）。

- **纸带打孔时代**：程序与数据以穿孔纸带表示，需人工上带、取带；人在机器时间尺度上极慢，**人力**成为瓶颈。
- **批处理**：由机器自动处理成批纸带，提高吞吐；此时新瓶颈是 **I/O 时 CPU 空转**（CPU 等待外设）。
- **多道程序**：内存中同时驻留多道程序，在某道程序等待 I/O 时 **切换** 去跑另一道，提高 CPU 利用率。注意：早期「能装入内存且整段装入」与今天 **按需分页 + 交换** 的多进程模型不同，不要混为一谈。
- **分时系统**：以 **时间片** 轮转，让多进程都有机会运行，并结合调度策略决定「下一刻谁占用 CPU」，从而支持 **交互式** 使用（例如边编译边看代码）。

> **硬件协同（与 CPU 并列出现的能力）**  
> 多道与分时需要 **保存/恢复现场**（更多通用寄存器、程序状态字等）、**内存布局与保护**（后续引出 MMU、TLB 等）。这些是 **软硬件协同演化** 的结果，而非仅靠软件「想出来」即可。

**1）早期操作系统 ≈ 一组常用库**  
不必每个程序员都写底层 I/O，而由「OS」提供 API，减轻开发负担。大型机上通常 **一次只跑一个程序**，由 **操作员** 决定作业顺序——许多今天由 OS 自动完成的调度决策，当时由人完成；聪明的开发者会「讨好」操作员以把作业挪到队前。**批处理**：先把作业准备好，再成批运行。当时 **交互式使用成本极高**——用户占着机器却大量闲置，在经济上不划算（书中引用 [BH00] 说明设施浪费）。

**2）超越「库」：保护与系统调用**  
随着 OS 管理机器的角色加重，人们意识到 **内核代码是特殊的**：它控制设备，不能与一般应用一视同仁。若任意程序都能从磁盘任意位置读取，**隐私**无从谈起，文件系统也不能再仅是「库」。于是出现 **系统调用**（system call）思想：Atlas 等系统较早探索——不是用普通 **过程调用** 进 OS 例程，而是增加 **硬件指令与状态**，使进入 OS 成为 **正式、受控** 的过程。

**3）系统调用 vs 过程调用；trap；特权级**  
关键差异：系统调用把控制转移到 OS，并 **提高硬件特权级**。用户程序在 **用户模式（user mode）** 下运行，硬件限制其能力（通常不能直接发起磁盘 I/O、不能随意访问任意物理页、不能随便发网络包等）。发起系统调用时，常通过 **trap** 类指令，硬件跳到预先登记的 **trap handler**，同时进入 **内核模式（kernel mode）**，OS 可完整使用硬件完成 I/O、扩内存等；完成后以 **return-from-trap** 返回用户模式，把控制交回应用 **离开前的现场**。

**4）多道程序时代与小型机**  
主机之后是 **小型机（minicomputer）** 时代（如 DEC PDP），计算机更便宜，更多开发者接触机器，系统更有趣。为提高资源利用率，**多道程序（multiprogramming）** 普及：内存中驻留多道作业并在其间快速切换，尤其避免 **I/O 慢时仍占着 CPU**。这迫使 OS 在概念上推进：**内存保护（memory protection）**、**并发（concurrency）**、**中断存在时仍保证内核正确** 等议题变得重要。书中指出后续章节会展开这些主题。**UNIX**（贝尔实验室 **Ken Thompson**、**Dennis Ritchie** 等）吸收 **Multics [O72]**、**TENEX [B+72]**、**Berkeley 分时 [S+68]** 等思想并更易用，通过磁带分发源码，吸引全球参与者共同演进（书中脚注①说明「补充材料」与主线的分工；此处不展开玩笑段落，以免喧宾夺主）。

**5）PC 时代：一度「倒退」与再追赶**  
除小型机外，**个人计算机（PC）**（Apple II、IBM PC 等）因低价普及到每张桌子。书中认为 **早期 PC 对 OS 而言曾是一大倒退**：例如 **DOS** 不重视内存保护，缺陷或恶意程序可破坏整机内存；早期 **macOS（System 7 及更早）** 采用 **合作式** 调度，线程死循环可拖垮整机需重启——由「OS 功能缺失」带来的痛苦列表很长。之后 **小型机/工作站的成熟经验** 回流桌面：**macOS X** 以 **UNIX** 为核心；**Windows** 自 **Windows NT** 起吸收大量成熟思想；今天手机上的 **Linux** 也更接近「70 年代小型机范式」而非「80 年代弱保护 PC」。书中强调：**好想法会进入现实世界并持续演化**。

## 二、虚拟化：三个大问题与直观示例

> **三个主要问题（虚拟化总纲）**
>
> - 操作系统通过哪些 **机制与策略** 实现虚拟化？
> - 如何 **高效** 地实现虚拟化？
> - 需要哪些 **硬件** 支持？

**虚拟化**可理解为：OS 把物理资源变成 **通用、易用、可共享** 的虚拟形式，例如 **CPU 的并发假象**、**虚拟内存** 等。

下面用经典 **「多进程同时跑」** 的小程序具象化 **CPU 虚拟化**（与 OSTEP 的 `cpu` 示例同型）。

```cpp
#include "common.h"
#include <assert.h>
#include <stdio.h>
#include <stdlib.h>
#include <sys/time.h>
int main(int argc, char *argv[])
{
    if (argc != 2) {
        fprintf(stderr, "usage: cpu <string>\n");
        exit(1);
    }
    char *str = argv[1];
    while (1) {
        Spin(1);
        printf("%s\n", str);
    }
    return 0;
}
```

```bash
# 编译
gcc -o cpu cpu.c -Wall
# 同时跑多个进程（后台并行）
./cpu A & ./cpu B & ./cpu C & ./cpu D &
```

初学编程时常见：**多个相同程序一起跑，却像「同时在跑」**——这是 **CPU 虚拟化** 的现象：在硬件与内核支持下，**单（或少）个物理 CPU** 被表现为 **许多虚拟 CPU**，多程序看似并行。**时分共享**（轮流占用 CPU）是典型手段之一；代价是相对独占 CPU 时，**单进程会变慢**。

在这些 **机制**（时分共享、以及与之对照常一并提及的 **空分/空间共享** 等）之上，还有 **策略**（例如 **进程调度**）：利用历史行为、负载类型、性能指标等综合决策 **下一个运行谁**。

## 三、进程

在操作系统中，**进程**是运行中的程序。机器上进程数常远大于 **CPU 核心数**，却仍能「同时」工作，核心挑战是：

> **如何营造「有许多 CPU」的假象？**

### 3.1 进程创建

OS 运行程序时先把 **代码与静态数据** 装入进程的 **地址空间**。

- **早期 / 批处理时代**：常见 **一次性完整装入**。
- **现代 OS**：多在执行期 **按需装入**，涉及 **分页、交换（swap）** 等。

装入后创建 **栈、堆**，初始化 **I/O** 等（与 **`task_struct` 如何被填充**、页表、写时复制等可对照内核源码），最后进入 `main()`，**CPU 控制权交给该进程**。

### 3.2 进程状态与调度触发（早期三态模型）

- **运行态**：正占用 CPU 执行。
- **就绪态**：可运行，等待被调度器选中。
- **阻塞态**：因某操作（典型如 **等待 I/O**）不能继续，需等事件完成。

![进程状态与调度示意](/images/posts/cpu-process-states-scheduling.jpg)

**调度如何发生？** 新进程就绪后，若各核忙，OS 仍要在适当时机把它放上去运行——常依赖 **时钟中断**：内核初始化时注册 **中断处理例程**（含调度逻辑），定时器到期则 **内核重新获得 CPU**，可做一次调度决策。另一典型路径是进程执行 `read()` 等阻塞 I/O，**主动让出** CPU 进入阻塞；I/O 完成中断到达后，内核再次获得控制权，可决定是否唤醒/调度。

归纳：**进程调度依赖内核周期性或事件性地夺回 CPU**；常见入口包括：

- **时钟中断**（抢占、时间片耗尽等）
- **系统调用路径**（阻塞、自愿让出等）

进入内核态还有异常、信号等其它路径；此处只强调与 **调度** 强相关的两类。（进一步可对照 **Linux 中断/软中断**、**嵌套中断防护**、**就绪/阻塞队列数据结构** 等。）

### 3.3 进程数据结构

OS 为 **就绪进程** 维护列表/结构以追踪信息；**阻塞进程** 另有组织方式，以便 I/O 完成时能 **唤醒正确进程**（涉及 **I/O 完成与 `task_struct`/等待队列** 的关联、调度器数据结构等）。

## 四、进程 API

### 4.1 `fork()`

一次调用、**两次返回**：以当前进程为模板创建子进程，子进程继承父进程诸多属性（涉及 **`task_struct` 复制**、哪些字段共享/独立、**页表与写时复制 COW** 等）。语义上：**从 `fork()` 返回点之后**，父子才各自继续执行「同一份后续代码」。

### 4.2 `wait()`

创建子进程后常用 `wait()`/`waitpid()` **回收**子进程退出状态，完成完整生命周期；否则子进程先退出而父进程不 `wait`，可能形成 **僵尸进程**（仍占 PID 等资源元数据）。父进程先退出而子进程仍在，则子进程可能被 **init** 接管成为 **孤儿进程**（与僵尸不同，概念需区分）。

### 4.3 `exec()`

`fork()` 后子进程与父进程同镜像；若要运行 **新程序**，需 **`exec*()`**：在 **同一 PID** 上替换地址空间中的 **代码段、数据段** 等，构建新进程映像，因此 **`exec` 成功则其后用户态代码不执行**（被新程序入口取代）。典型保留：**文件描述符**、**工作目录**、**PID**；**信号处理** 常被重置。替换地址空间可避免无意义的 **COW 之后整段复制**，提高效率。

## 五、机制：受限直接执行（LDE）

虚拟化 CPU 需多进程 **共享物理 CPU**（时分共享），涉及 **切换**；矛盾是：**既要快（尽量少额外开销），又要能随时收回控制权（安全）**。

> **关键问题（书中表述）**  
> OS 必须以 **高性能** 虚拟化 CPU，同时 **保持对系统的控制**；这通常需要 **硬件配合**，OS **明智地利用硬件** 完成工作。

### 5.1 基本技巧：直接执行

为尽量快，让进程代码 **直接在 CPU 上跑**；但必须防止其 **越权** 访问敏感资源。

### 5.2 防止越权：模式与陷阱表

靠 **硬件执行模式** 区分 **用户模式** 与 **特权（内核）模式**；用户态下部分敏感指令无法直接执行，会 **陷入（trap）** 内核。（原文用「特权位 + 电路门禁」的直觉类比帮助理解：**特权位未置位时，部分敏感指令无法通过硬件检查**；不必拘泥于具体元器件表述。）内核初始化时把 **陷阱表（trap vector）** 等登记好，以便进入统一的 **内核路径** 处理。

### 5.3 如何在进程间切换：协作式 vs 非协作式

**协作式**：依赖进程在 **系统调用** 或 **非法操作** 时把控制权交给 OS；但若遇 **恶意死循环** 不调用 OS，则 **内核无法插入**，系统失控。

**非协作式：时钟中断**

> **关键问题**  
> 即使进程不协作，OS 如何 **夺回 CPU**？如何避免 **流氓进程占死机器**？

**答案**：**可编程定时器 / 时钟中断**——每隔若干毫秒产生中断，打断当前进程，进入内核预设的中断处理路径，从而 **OS 重新获得控制权**。硬件需在 **中断入口** 保存足够现场，以便 **中断返回** 后用户进程可继续正确执行。

### 5.4 并发与关中断的权衡

多源中断并发时，内核可能在处理某中断时 **暂时禁止再次中断**（或屏蔽部分级别）；若 **关中断时间过长**，可能 **丢失中断** —— 这是真实内核实现里要仔细权衡的点。

## 六、进程调度

底层「如何切换」清楚后，上层问题是：**获得 CPU 后，下一刻运行谁？**

### 6.1 先进先出（FIFO）

实现简单，但对 **周转时间**（到达→完成）不友好：短任务可能被长任务 **堵在后面**。

### 6.2 短作业优先（SJF）

优先最短作业；**非抢占** 时，若短作业 **晚到** 而长作业已运行，仍可能长时间等待。

### 6.3 最短完成时间优先（STCF / 可抢占 SJF）

允许 **抢占**：新任务到达时比较 **剩余运行时间**，优先运行更短者，可显著改善周转时间。

### 6.4 新指标：响应时间

分时与交互式场景引入 **响应时间**（到达→**首次运行**）：

> **响应时间** = **首次运行时刻** − **到达时刻**

纯 STCF 下，若前面总有更短任务，交互任务可能 **长时间得不到首次运行**，体验灾难。

### 6.5 轮转（RR）

**时间片轮转**：队列内任务轮流运行固定片长，直到完成。片长过短 → **上下文切换** 过频，开销大；切换不仅保存寄存器，还可能 **抖动缓存、TLB、分支预测器** 等 **热状态**，需 **合理选择时间片**。

### 6.6 存在 I/O 时

某进程阻塞于 I/O 时，应把 CPU **让给** 其他就绪进程，避免空转。

### 6.7 小结

上述理想化策略常假设 **预知作业长度**；现实中往往 **未知**，需把 **理想目标** 与 **可实现机制** 结合，设计 **可用且可调** 的策略。

## 七、多级反馈队列（MLFQ）

> **提示：从历史中学习**  
> MLFQ 是用 **历史行为预测未来** 的典型：OS 里多处类似思路（硬件分支预测、缓存替换等）。若负载有明显阶段性，预测会有效；但若误用，可能比「一无所知」更糟。

### 7.1 基本规则

多个队列、**不同优先级**；任务同一时刻只处于一个队列。

- **规则 1**：若 A 优先级 **高于** B，则运行 A（不运行 B）。
- **规则 2**：若优先级 **相同**，则 **轮转**。

### 7.2 动态调整优先级（初版三条）

- 新任务先进入 **最高优先级队列**。
- **用完一个时间片** → 降一级。
- 若在一个时间片内 **主动放弃 CPU**（如阻塞 I/O）→ **优先级不变**。

**问题**：若某进程频繁短 I/O、总在片内让出，则 **长期霸占高优先级队列**，其它曾「用完一片」而降级的进程可能 **饿死**。

### 7.3 周期性提升优先级

- 每隔一段时间 **S**，把所有工作 **重新提升到最高优先级队列**。

作用：**防饿死**；且当进程从 **CPU 密集** 转为 **交互型** 时，有机会被重新「当作交互任务」对待。代价：**S 难以调参**。

### 7.4 更好的计时：防「戏弄调度器」

防止进程 **故意在片尾 yield** 以保住高优先级：改为记录在某层的 **累计 CPU 消耗配额**，**一旦在该层配额用尽**（无论中间主动放弃多少次），**强制降级**。

### 7.5 队列调参与其它工程问题

各层 **时间片长度**、**提升周期 S** 等往往无唯一最优解，依赖 **负载经验** 与线上反馈迭代。许多 MLFQ 变体让 **高层片短、低层片长**（低层偏计算密集）。

### 7.6 规则汇总（小结）

1. 若 A 的优先级 **>** B 的优先级，运行 A（不运行 B）。
2. 若 A 的优先级 **=** B 的优先级，轮转运行 A 与 B。
3. 工作进入系统时，放在 **最高优先级**（最上层队列）。
4. 一旦工作在某层 **用尽时间配额**（无论主动放弃 CPU 多少次），**降低**其优先级（移入低一级队列）。
5. 每隔时间 **S**，将系统中 **所有工作** 重新加入 **最高优先级队列**。

尽管 MLFQ 很经典，**现代 Linux 默认调度已转向 CFS**（完全公平调度器），见下一节。

## 八、多处理器调度

### 8.1 多处理器与缓存

与单 CPU 的关键差异在于 **缓存层次** 与 **多核间共享/可见的数据路径**。（注意：**内存屏障** 等问题主要来自 **并发与缓存一致性模型**；不要把「多处理器」本身误当成屏障的唯一来源——屏障话题在并发章节单独展开。）

![多核与缓存示意](/images/posts/cpu-multicore-cache.png)

单核上 **多级缓存** 利用 **时间局部性、空间局部性** 降低访存延迟。多核下若 **CPU1** 在缓存中把 **A 改成 A1** 尚未写回内存，而线程被调度到 **CPU2** 再读，可能仍看到旧值——属 **缓存一致性** 问题。硬件常用 **总线嗅探（snooping）** 等协议维护 **各核视图与内存的一致性**（值得单独深入学习，对上层系统设计也有启发）。

### 8.2 缓存亲和度（affinity）

进程在某一核上积累了 **缓存热数据**；若频繁迁移到别核，会 **冷缓存、反复重载**。调度上倾向 **让线程 sticky 在某核**（在迁移收益与负载间折中）。

### 8.3 单队列多核调度

**全局单一就绪队列**：多核争用同一把（或少数）锁，**锁竞争** 开销大；且任务在核间漂移，**亲和度差**。

### 8.4 每核多队列与负载均衡

**每 CPU 一个就绪队列**：降低锁竞争、改善亲和；但可能 **负载不均**——某核队列空而别核仍忙。需要 **窃取（work stealing）/迁移** 策略，并面临 **观察频率、一次迁移多少** 等工程参数难题。

### 8.5 小结

**完美调度器**不存在；要在 **扩展性、负载均衡、缓存亲和、公平与延迟** 之间持续折中。

## 九、Linux CFS 调度算法

> **扩展阅读**：关于 CFS 的更细实现与案例，见独立笔记 [Linux 进程调度详解]({% post_url 2025-04-16-Linux-进程调度详解 %})。

### 9.1 CFS 概述

**CFS（Completely Fair Scheduler）** 自 **Linux 2.6.23** 起成为默认调度器，取代 **O(1)**。核心理念：在理想多任务 CPU 上，各进程 **按权重比例** 获得 CPU 时间。

主要特点：

1. **红黑树**：可运行进程按 **vruntime** 组织；
2. **公平性**：常选 **vruntime 最小** 者运行；
3. **权重 / nice**：把优先级映射为 **时间比例**；
4. **动态时间**：弱化固定「时间片」直觉，以 **虚拟时间追赶** 建模公平。

### 9.2 虚拟运行时间（vruntime）

```
vruntime += 实际运行时间 × (标准权重 / 进程权重)
```

- 权重高（nice 更「高优先级」）→ vruntime 增长更慢 → 更容易被选中、占更多 CPU。
- CFS 常选红黑树 **最左**（vruntime 最小）进程运行。

### 9.3 CFS 与 MLFQ 对比

| 特性 | CFS | MLFQ |
|------|-----|------|
| 数据结构 | 红黑树 | 多级队列 |
| 公平性 | 基于 vruntime 的严格公平 | 基于启发式规则 |
| 时间复杂度 | O(log n) | 均摊常数级常见讨论 |
| 配置复杂度 | 相对较低 | 规则与参数较多 |

### 9.4 实时调度策略

Linux 还支持：

- **SCHED_FIFO**：同优先级 FIFO，无普通时间片概念；
- **SCHED_RR**：同优先级轮转带时间片；
- **SCHED_DEADLINE**：截止时间调度（Linux 3.14+）。

实时优先级 **0–99** 高于普通 **nice 映射区间（100–139）**，用于 **时效敏感** 任务。
