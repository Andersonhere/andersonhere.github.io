> **核心问题**：如何在不知道任务运行时间的情况下优化调度？

## 本章大纲

### 8.1 MLFQ：基本规则

**设计目标**：
1. 优化周转时间
2. 降低响应时间
3. 不需要预知任务运行时间

**基本结构**：
- 多个优先级队列
- 不同队列时间片不同（高优先级时间片短）
- 新任务从最高优先级开始

```
优先级
  高  │  Queue 0  │  时间片短
      │  Queue 1  │
      │  Queue 2  │
  低  │  Queue 3  │  时间片长
```

### 8.2 尝试 1：如何改变优先级

**规则**：
1. 优先级高的队列先运行
2. 同优先级使用 Round Robin
3. 用完时间片，优先级降低
4. 主动放弃 CPU（I/O），优先级不变

```
新任务 → Queue 0 → 运行时间片 → Queue 1 → 运行时间片 → Queue 2 → ...
         ↑
         I/O 操作保持优先级
```

**问题**：
- 长任务最终会沉到最低优先级
- 交互式任务（频繁 I/O）可能独占高优先级

### 8.3 尝试 2：提升优先级

**新规则**：每隔时间 S，将所有任务提升到最高优先级

**解决的问题**：
1. 防止长任务饥饿
2. 让改变行为的任务有机会提升优先级

**新问题**：
- S 如何选择？
- 可被聪明用户利用

### 8.4 尝试 3：更好的计时方式

**问题**：任务可能通过频繁放弃 CPU 来保持高优先级

**解决方案**：记录在每个优先级的累计运行时间

```
规则修改：
- 任务在当前优先级累计运行达到时间片配额后降低优先级
- 无论中间放弃 CPU 多少次
```

### 8.5 MLFQ 调优及其他问题

**可调参数**：
- 队列数量
- 每个队列的时间片大小
- 提升优先级的间隔 S

**常见配置**：
- 高优先级队列：短时间片（适合交互式）
- 低优先级队列：长时间片（适合计算密集型）

### 8.6 MLFQ：小结

**完整规则**：

| 规则 | 描述 |
|------|------|
| 规则 1 | 优先级 A > B，运行 A |
| 规则 2 | 优先级 A = B，轮转运行 |
| 规则 3 | 新任务进入最高优先级 |
| 规则 4 | 任务用完当前级别配额，降低优先级 |
| 规则 5 | 每隔时间 S，所有任务提升到最高优先级 |

**优点**：
- 自动区分交互式和计算密集型任务
- 不需要预知任务运行时间
- 公平性好

**缺点**：
- 参数调优复杂
- 可能被恶意利用
- 已被 Linux CFS 取代

## MLFQ 行为示例

```
时间线：

任务 A (长任务)     ────────────────────────────────────────→
                    Q0  Q1  Q2  Q3  Q3  Q3  Q3  ...

任务 B (交互式)     ─I/O─┐  ┌─I/O─┐  ┌─I/O─┐
                    Q0    └──┘    └──┘    └──  (保持 Q0)

任务 C (混合型)     ─────I/O────┐ ┌────────────────
                    Q0  Q1  Q1  └─┘ Q2  Q3  Q3  ...

优先级提升（每 S 时间）
                    ↑              ↑              ↑
                   所有任务回到 Q0
```
